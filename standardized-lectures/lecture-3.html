本节核心
数据持久化是生产环境部署的关键。没有数据卷，容器重启或删除后所有数据都会丢失，这对于数据库等应用是致命的。
Part 1

容器数据丢失问题

为什么需要数据持久化？

Part 1.1

问题演示：容器数据丢失

创建文件后删除容器 
# 1. 启动容器并创建文件
docker run -it --name test centos:7 /bin/bash
echo "hello" > /tmp/test.txt
exit

# 2. 重新启动容器查看文件
docker start -i test
cat /tmp/test.txt  # 文件存在

# 3. 删除容器后重新创建
docker rm -f test
docker run -it --name test2 centos:7 /bin/bash
cat /tmp/test.txt  # 文件不存在！
核心问题

容器删除后，容器内的所有数据都会丢失！

这对于数据库等需要持久化数据的应用是致命的。

Part 1.2

解决方案：数据卷

数据卷 Volume
数据卷是一个可供容器使用的特殊目录，它绕过容器的文件系统，直接存在于宿主机上。即使容器被删除，数据卷中的数据也不会丢失。
数据卷的优势

数据持久化：容器删除后数据不丢失

数据共享：多个容器可以共享同一数据卷

数据备份：可以直接在宿主机上备份数据

核心概念

容器的文件系统是临时的，数据卷是持久的。

数据卷将数据从容器中分离出来，存储在宿主机上。

Part 2

数据卷三种挂载方式

掌握不同的数据挂载场景

Part 2.1

三种挂载方式概览

01

宿主机目录挂载

-v /host:/container

02

匿名卷

-v /container

03

命名卷

-v name:/container

挂载的本质
挂载就是将宿主机的目录与容器内的目录建立映射关系。容器内对该目录的读写操作，实际上是对宿主机目录的操作。
Part 2.2

方式一：宿主机目录挂载

挂载宿主机目录 
# 将宿主机/opt/data挂载到容器/app/data
docker run -it -v /opt/data:/app/data centos:7 /bin/bash

# 在容器内创建文件
echo "test" > /app/data/test.txt

# 在宿主机查看
cat /opt/data/test.txt  # 文件存在！
语法说明

-v 宿主机路径:容器路径：将宿主机的指定目录挂载到容器内的指定路径。两边目录会实时同步。

使用场景

开发环境：代码目录挂载，实时更新

配置文件：挂载配置文件，便于修改

日志收集：挂载日志目录，统一管理

Part 2.3

方式二：匿名卷

创建匿名卷 
# 创建匿名卷
docker run -it -v /app/data centos:7 /bin/bash

# 查看卷信息
docker inspect 容器ID | grep -A 10 Mounts
匿名卷特点

Docker自动在/var/lib/docker/volumes下创建目录

没有明确的名称，只有一串随机ID

适合临时数据存储

匿名卷的局限
匿名卷由于没有名称，难以管理和复用。当容器被删除时，匿名卷可能成为"孤儿卷"，占用磁盘空间。生产环境建议使用命名卷。
Part 2.4

方式三：命名卷

创建和使用命名卷 
# 创建命名卷
docker volume create mydata

# 使用命名卷
docker run -it -v mydata:/app/data centos:7 /bin/bash

# 查看所有卷
docker volume ls

# 查看卷详情
docker volume inspect mydata

# 删除命名卷
docker volume rm mydata
命名卷优势

有明确的名称，便于管理

可以在多个容器间共享

易于备份和迁移

生产环境推荐
命名卷是生产环境的首选，它提供了最佳的可管理性和可维护性。数据库数据、应用日志等都应使用命名卷。
Part 2.5

三种挂载方式对比

方式 语法 特点 适用场景
宿主机目录 -v /host:/container 灵活可控 开发环境、配置文件
匿名卷 -v /container 自动管理 临时数据
命名卷 -v name:/container 易于管理 生产环境、数据共享
选择建议

开发环境：使用宿主机目录挂载，方便编辑代码

生产环境：使用命名卷，便于管理和备份

临时数据：使用匿名卷，让Docker自动管理

Part 3

读写权限控制

rw/ro模式区别

Part 3.1

读写权限控制

权限设置 
# 默认读写模式(rw)
docker run -v /opt/data:/app centos:7

# 只读模式(ro)
docker run -v /opt/data:/app:ro centos:7

# 容器内无法写入只读挂载的目录
echo "test" > /app/test.txt  # 报错！
rw 读写模式（默认），容器可以读写挂载目录
ro 只读模式，容器只能读取，不能写入
ro模式使用场景

配置文件：防止容器意外修改配置

共享数据：多个容器读取同一数据，防止误写

安全隔离：限制容器对敏感数据的访问

Part 4.1

实操：MySQL数据持久化

Step 1

启动MySQL并挂载数据卷

docker run -d \
  --name mysql \
  -p 3306:3306 \
  -e MYSQL_ROOT_PASSWORD=123456 \
  -v /opt/mysql/data:/var/lib/mysql \
  -v /opt/mysql/conf:/etc/mysql/conf.d \
  mysql:5.7

挂载数据目录和配置目录

Step 2

创建测试数据

# 进入MySQL容器
docker exec -it mysql mysql -uroot -p123456

# 创建数据库和表
CREATE DATABASE testdb;
USE testdb;
CREATE TABLE users (id INT, name VARCHAR(50));
INSERT INTO users VALUES (1, 'docker');
SELECT * FROM users;
Part 4.2

验证数据持久化

删除容器后验证数据 
# 删除容器
docker rm -f mysql

# 重新启动MySQL
docker run -d \
  --name mysql2 \
  -p 3306:3306 \
  -e MYSQL_ROOT_PASSWORD=123456 \
  -v /opt/mysql/data:/var/lib/mysql \
  mysql:5.7

# 验证数据
docker exec -it mysql2 mysql -uroot -p123456
USE testdb;
SELECT * FROM users;  # 数据还在！
验证成功

即使删除容器，数据仍然保存在宿主机的挂载目录中。

重新启动容器后数据自动恢复。

Part 4.3

实操：数据卷共享

两个容器共享同一数据卷 
# 创建共享数据卷
docker volume create shared-data

# 启动容器1
docker run -it --name c1 -v shared-data:/data centos:7

# 在容器1中创建文件
echo "from c1" > /data/share.txt

# 启动容器2
docker run -it --name c2 -v shared-data:/data centos:7

# 在容器2中查看文件
cat /data/share.txt  # 输出：from c1
使用场景

多个容器需要访问同一份数据：

Web服务器集群共享静态文件

日志收集：多个应用写入同一日志目录
