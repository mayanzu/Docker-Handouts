Part 1

Docker镜像概述

理解镜像的本质

Part 1.1

什么是Docker镜像？

镜像定义
Docker镜像是一个只读模板，包含了运行应用程序所需的所有内容：代码、运行时环境、系统工具、系统库和配置。镜像是容器的基础，容器是镜像的运行实例。

可以把镜像理解为类（Class），容器理解为对象（Object）。一个镜像可以创建多个容器实例，每个容器都是独立的运行环境。

镜像的组成

1. 基础镜像层：如ubuntu、alpine等操作系统基础

2. 依赖层：应用程序运行所需的库和工具

3. 应用层：应用程序代码和配置文件

4. 元数据：镜像的配置信息（环境变量、端口、启动命令等）

Part 1.2

镜像分层机制

Docker镜像采用联合文件系统（UnionFS）技术，由多个只读层（Layer）叠加而成。每一层代表Dockerfile中的一条指令。

分层优势

- 共享层：多个镜像可以共享相同的基础层，节省存储空间

- 快速构建：只重建变化的层，未变化层直接使用缓存

- 版本控制：每层有唯一ID，便于追踪和管理

Part 1.3

写时复制（CoW）

写时复制机制
写时复制（Copy-on-Write）是Docker镜像的核心技术。当容器启动时，Docker会在镜像的最上层添加一个可写层（Container Layer）。所有对文件的修改都发生在这个可写层，而不会改变底层的只读镜像层。

1读取文件

直接从镜像的只读层读取，性能高效

2修改文件

将文件从只读层复制到可写层，然后修改

3删除文件

在可写层创建删除标记，原文件仍保留在镜像层

4创建文件

直接在可写层创建新文件

理解要点

容器删除后，可写层数据会丢失。如需持久化数据，应使用数据卷（Volume）或绑定挂载（Bind Mount）。

Part 3

Dockerfile基础

构建自定义镜像的脚本

Part 3.1

什么是Dockerfile？

Dockerfile定义
Dockerfile是一个文本文件，包含一系列指令，用于自动化构建Docker镜像。每条指令在镜像中创建一个新层，描述了如何从基础镜像构建出目标镜像。
简单的Dockerfile示例 Dockerfile 
# 基础镜像
FROM ubuntu:22.04

# 维护者信息
LABEL maintainer="your-email@example.com"

# 设置环境变量
ENV DEBIAN_FRONTEND=noninteractive

# 更新包列表并安装nginx
RUN apt-get update && \
    apt-get install -y nginx && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*

# 复制本地文件到镜像
COPY index.html /var/www/html/

# 暴露端口
EXPOSE 80

# 启动命令
CMD ["nginx", "-g", "daemon off;"]
Dockerfile特点

1. 自动化：一次编写，多次构建，确保环境一致性

2. 可维护：文本文件，版本控制友好

3. 可复用：基于基础镜像，层层叠加

4. 透明：构建过程清晰可见

Part 4

Dockerfile指令详解

掌握每个指令的用法和参数

Part 4.1

FROM指令

FROM指令
FROM指令指定构建镜像的基础镜像，必须是Dockerfile的第一条指令（除ARG外）。基础镜像可以是官方镜像、私有镜像或scratch（空镜像）。
FROM语法与示例 Dockerfile 
# 语法格式：
# FROM [--platform=] [:] [AS ]
# FROM [--platform=] [@] [AS ]

# 参数说明：
# --platform    指定平台（如linux/amd64, linux/arm64）
# image         基础镜像名称
# tag           镜像标签（默认latest）
# digest        镜像摘要（SHA256）
# AS name       多阶段构建时指定阶段名称

# 示例1：使用官方镜像的最新版
FROM ubuntu

# 示例2：使用指定版本的镜像
FROM ubuntu:22.04

# 示例3：使用指定平台的镜像
FROM --platform=linux/amd64 nginx:alpine

# 示例4：使用镜像摘要（确保使用特定版本）
FROM ubuntu@sha256:9a0bdde4188b896a372804be2384015e90e3f8493751292d5de9e3e5f8a6f3c8

# 示例5：多阶段构建，指定阶段名称
FROM golang:1.19 AS builder

# 示例6：从零开始构建（空镜像）
FROM scratch
常用基础镜像推荐

- ubuntu:22.04：完整的Ubuntu系统，适合通用应用

- alpine:latest：轻量级Linux，仅5MB，适合微服务

- debian:bullseye-slim：精简版Debian，平衡大小和功能

- scratch：空镜像，用于静态编译的二进制程序

Part 4.2

RUN指令

RUN指令
RUN指令在镜像构建过程中执行命令，并提交结果作为新层。用于安装软件包、下载文件、编译代码等构建时操作。
RUN语法与示例 Dockerfile 
# 语法格式：
# RUN                     # Shell格式（默认使用/bin/sh -c）
# RUN ["executable", "param1", "param2"]  # Exec格式（JSON数组）

# 参数说明：
# --mount=type=...    挂载缓存或秘密（BuildKit特性）
# --network=...       指定网络模式

# 示例1：Shell格式（推荐用于简单的shell命令）
RUN apt-get update && apt-get install -y nginx

# 示例2：Exec格式（推荐用于避免shell字符串处理问题）
RUN ["apt-get", "update"]
RUN ["apt-get", "install", "-y", "nginx"]

# 示例3：多行命令（使用&&连接，减少镜像层数）
RUN apt-get update && \
    apt-get install -y \
        curl \
        vim \
        git \
        wget && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*

# 示例4：使用管道
RUN wget -O - https://example.com/script.sh | bash

# 示例5：切换shell（使用其他shell执行）
SHELL ["/bin/bash", "-c"]
RUN echo "Using bash instead of sh"

# 示例6：使用BuildKit缓存挂载（加速重复构建）
# syntax=docker/dockerfile:1
RUN --mount=type=cache,target=/var/cache/apt \
    apt-get update && apt-get install -y nginx

# 示例7：使用秘密挂载（安全传递敏感信息）
RUN --mount=type=secret,id=mysecret cat /run/secrets/mysecret

# 示例8：指定网络模式
RUN --network=none echo "No network access"
RUN --network=host git clone https://github.com/example/repo.git
最佳实践

- 将多个相关命令用 && 连接，减少镜像层数

- 安装完成后清理缓存（apt-get clean、rm -rf /var/cache）

- 将不常变动的指令放在Dockerfile前面，充分利用缓存

Part 4.3

COPY与ADD指令

COPY语法与示例 Dockerfile 
# 语法格式：
# COPY [--chown=:] [--chmod=] ... 
# COPY [--chown=:] [--chmod=] ["",... ""]

# 参数说明：
# --chown       设置文件所有者和组
# --chmod       设置文件权限（BuildKit特性）
# --from        从其他构建阶段或镜像复制
# src           源文件/目录（相对构建上下文）
# dest          目标路径（镜像内路径）

# 示例1：复制单个文件
COPY app.py /app/

# 示例2：复制多个文件到目录
COPY app.py config.py requirements.txt /app/

# 示例3：复制目录
COPY src/ /app/src/

# 示例4：使用通配符
COPY *.txt /app/
COPY app.* /app/

# 示例5：设置文件所有者
COPY --chown=www-data:www-data app.py /app/

# 示例6：设置文件权限
COPY --chmod=755 entrypoint.sh /app/

# 示例7：从其他构建阶段复制（多阶段构建）
COPY --from=builder /app/build /app/

# 示例8：从其他镜像复制
COPY --from=nginx:latest /etc/nginx/nginx.conf /app/nginx.conf

# 示例9：保留空格的文件名（使用JSON数组格式）
COPY ["my file.txt", "/app/"]
COPY ["config file.json", "settings.json", "/app/config/"]
ADD语法与示例 Dockerfile 
# 语法格式：
# ADD [--chown=:] [--chmod=] ... 
# ADD [--chown=:] [--chmod=] ["",... ""]

# ADD与COPY的区别：
# 1. ADD可以自动解压tar文件（gzip, bzip2, xz）
# 2. ADD可以从URL下载文件

# 示例1：复制本地tar文件并自动解压
ADD archive.tar.gz /app/

# 示例2：从URL下载文件（不推荐，使用curl或wget替代）
ADD https://example.com/file.txt /app/

# 示例3：从URL下载tar并自动解压
ADD https://example.com/archive.tar.gz /app/

# 示例4：使用通配符
ADD *.tar.gz /app/

# 示例5：设置文件所有者
ADD --chown=root:root config.tar.gz /etc/

# 示例6：保留权限的复制
ADD --chmod=644 config.ini /app/
COPY vs ADD选择建议

优先使用COPY，行为明确可预测。仅在需要自动解压tar文件时使用ADD。从URL下载文件建议使用RUN配合curl/wget，更透明可控。

Part 4.4

ENV与ARG指令

ENV语法与示例 Dockerfile 
# 语法格式：
# ENV             # 单个变量
# ENV = ...       # 多个变量

# 参数说明：
# key           环境变量名
# value         环境变量值

# 示例1：设置单个环境变量
ENV NODE_VERSION 18.17.0

# 示例2：设置多个环境变量
ENV NODE_VERSION=18.17.0 \
    NPM_VERSION=9.6.7 \
    YARN_VERSION=1.22.19

# 示例3：环境变量在后续指令中使用
ENV APP_HOME=/app
WORKDIR $APP_HOME
COPY . $APP_HOME

# 示例4：设置PATH环境变量
ENV PATH=/usr/local/node/bin:$PATH

# 示例5：设置默认环境变量（可被覆盖）
ENV DATABASE_URL=postgres://localhost:5432/mydb
ENV DEBUG=false
ENV LOG_LEVEL=info

# 示例6：使用环境变量构建其他变量
ENV BASE_URL=https://api.example.com
ENV API_URL=$BASE_URL/v1

# 运行时的效果：
# 容器启动后，这些环境变量会保留在容器中
# 可以通过 docker run -e 覆盖
# $ docker run -e DATABASE_URL=postgres://prod:5432/mydb myapp
ARG语法与示例 Dockerfile 
# 语法格式：
# ARG [=]

# 参数说明：
# name          参数名
# default value 默认值（可选）

# 示例1：定义构建参数
ARG VERSION=latest

# 示例2：使用构建参数
FROM node:$VERSION

# 示例3：多个构建参数
ARG NODE_VERSION=18
ARG APP_ENV=production
ARG BUILD_DATE

# 示例4：在构建时使用参数
# docker build --build-arg VERSION=16 -t myapp:16 .

# 示例5：ARG在FROM之前（可用于动态选择基础镜像）
ARG BASE_IMAGE=ubuntu:22.04
FROM $BASE_IMAGE

# 示例6：ARG作用域（在FROM之后需要重新定义）
ARG VERSION=latest
FROM node:$VERSION
ARG VERSION              # 需要重新定义才能在FROM后使用
RUN echo "Building version: $VERSION"

# 示例7：预定义ARG（无需声明即可使用）
# TARGETPLATFORM - 目标平台（如linux/amd64）
# TARGETOS - 目标操作系统
# TARGETARCH - 目标架构
# TARGETVARIANT - 目标变体
# BUILDPLATFORM - 构建平台
# BUILDOS - 构建操作系统
# BUILDARCH - 构建架构
# BUILDVARIANT - 构建变体

FROM alpine
ARG TARGETPLATFORM
RUN echo "Building for $TARGETPLATFORM"

# 构建命令示例：
# $ docker build --build-arg NODE_VERSION=16 --build-arg APP_ENV=development -t myapp .
ENV vs ARG区别

- ARG：构建时变量，不会保留在最终镜像中

- ENV：运行时变量，会保留在容器中，可被覆盖

- 敏感信息（密码、密钥）应使用ARG而非ENV

Part 4.5

WORKDIR与USER指令

WORKDIR语法与示例 Dockerfile 
# 语法格式：
# WORKDIR /path/to/workdir

# 参数说明：
# path          工作目录路径（绝对或相对路径）

# 示例1：设置工作目录
WORKDIR /app

# 示例2：相对路径（基于当前WORKDIR）
WORKDIR /app
WORKDIR src        # 最终路径：/app/src

# 示例3：自动创建目录
WORKDIR /app/src/components    # 如果目录不存在会自动创建

# 示例4：使用环境变量
ENV APP_HOME=/app
WORKDIR $APP_HOME

# 示例5：完整示例
FROM node:18
WORKDIR /app                    # 设置工作目录
COPY package*.json ./           # 复制到/app/
RUN npm install                 # 在/app/中执行
COPY . .                        # 复制所有文件到/app/
EXPOSE 3000
CMD ["npm", "start"]            # 在/app/中启动

# 效果说明：
# - 后续的RUN、CMD、ENTRYPOINT、COPY、ADD都在此目录执行
# - 如果目录不存在会自动创建
# - 可以多次使用，支持相对路径
USER语法与示例 Dockerfile 
# 语法格式：
# USER [:]
# USER [:]

# 参数说明：
# user          用户名或UID
# group         组名或GID（可选）

# 示例1：切换到指定用户
USER nginx

# 示例2：指定用户和组
USER nginx:nginx

# 示例3：使用UID和GID
USER 1000:1000

# 示例4：创建用户后切换
RUN groupadd -r appgroup && useradd -r -g appgroup appuser
USER appuser

# 示例5：完整的安全实践示例
FROM node:18

# 创建非root用户
RUN groupadd -r nodejs && useradd -r -g nodejs -s /bin/false nodejs

WORKDIR /app

# 先以root安装依赖
COPY package*.json ./
RUN npm ci --only=production

# 更改文件所有者
RUN chown -R nodejs:nodejs /app

# 切换到非root用户运行
USER nodejs

COPY --chown=nodejs:nodejs . .

EXPOSE 3000
CMD ["node", "server.js"]

# 验证当前用户
# $ docker run --rm myapp id
# uid=999(nodejs) gid=999(nodejs) groups=999(nodejs)
安全建议

生产环境的容器应使用非root用户运行，减少安全风险。使用USER指令切换到普通用户，或确保应用本身不以root启动。

Part 4.6

EXPOSE指令

EXPOSE语法与示例 Dockerfile 
# 语法格式：
# EXPOSE  [/...]

# 参数说明：
# port          端口号
# protocol      协议类型（tcp或udp，默认tcp）

# 示例1：暴露单个端口
EXPOSE 80

# 示例2：暴露多个端口
EXPOSE 80 443

# 示例3：指定协议
EXPOSE 80/tcp
EXPOSE 53/udp

# 示例4：暴露端口范围
EXPOSE 8000-8010

# 示例5：完整示例
FROM node:18
WORKDIR /app
COPY . .
RUN npm install

# 暴露应用端口
EXPOSE 3000

# 暴露调试端口
EXPOSE 9229/tcp

CMD ["npm", "start"]

# 使用说明：
# EXPOSE只是声明，不会自动映射到主机端口
# 实际映射需要在运行时指定：
# $ docker run -p 8080:3000 myapp
# 
# 查看暴露的端口：
# $ docker inspect myapp | grep -A 5 ExposedPorts

# 预期输出（docker inspect）：
# "ExposedPorts": {
#     "3000/tcp": {},
#     "9229/tcp": {}
# }
EXPOSE vs -p 参数

- EXPOSE：声明容器内部监听的端口，仅作为文档说明

- -p 主机:容器：运行时实际映射端口，如 -p 8080:80

- -P：随机映射所有EXPOSE的端口到主机高位端口

Part 4.7

CMD与ENTRYPOINT指令

CMD语法与示例 Dockerfile 
# 语法格式：
# CMD ["executable","param1","param2"]    # Exec格式（推荐）
# CMD command param1 param2               # Shell格式
# CMD ["param1","param2"]                 # 作为ENTRYPOINT的默认参数

# 示例1：Exec格式（推荐）
CMD ["nginx", "-g", "daemon off;"]

# 示例2：Shell格式
CMD nginx -g 'daemon off;'

# 示例3：作为ENTRYPOINT的参数
ENTRYPOINT ["nginx"]
CMD ["-g", "daemon off;"]

# 示例4：提供默认命令可被覆盖
FROM ubuntu:22.04
CMD ["echo", "Hello from default CMD"]

# 运行示例：
# $ docker run myimage
# Hello from default CMD
#
# $ docker run myimage echo "Custom command"
# Custom command

# 示例5：完整的Web应用示例
FROM node:18
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]

# 注意：
# - Dockerfile中只能有一个CMD，多个时只有最后一个生效
# - CMD可被docker run命令行参数覆盖
ENTRYPOINT语法与示例 Dockerfile 
# 语法格式：
# ENTRYPOINT ["executable", "param1", "param2"]    # Exec格式（推荐）
# ENTRYPOINT command param1 param2               # Shell格式

# 示例1：Exec格式（推荐）
ENTRYPOINT ["nginx", "-g", "daemon off;"]

# 示例2：Shell格式
ENTRYPOINT nginx -g 'daemon off;'

# 示例3：ENTRYPOINT + CMD 组合使用
ENTRYPOINT ["nginx"]
CMD ["-g", "daemon off;"]

# 运行时的行为：
# $ docker run mynginx
# 实际执行：nginx -g daemon off;
#
# $ docker run mynginx -t
# 实际执行：nginx -t（CMD被覆盖）

# 示例4：使用ENTRYPOINT作为固定命令
FROM ubuntu:22.04
ENTRYPOINT ["echo", "Hello"]
CMD ["World"]

# 运行示例：
# $ docker run myimage
# Hello World
#
# $ docker run myimage Docker
# Hello Docker

# 示例5：使用脚本作为ENTRYPOINT
COPY docker-entrypoint.sh /usr/local/bin/
RUN chmod +x /usr/local/bin/docker-entrypoint.sh
ENTRYPOINT ["docker-entrypoint.sh"]
CMD ["default-arg"]

# docker-entrypoint.sh示例：
#!/bin/bash
set -e

# 执行初始化操作
if [ "$1" = 'default-arg' ]; then
    echo "Running with default configuration"
    exec myapp
else
    exec "$@"
fi
CMD vs ENTRYPOINT选择

- CMD：提供默认命令，可被覆盖，适合灵活的容器

- ENTRYPOINT：固定命令，追加参数，适合工具类容器

- 组合使用：ENTRYPOINT定义主命令，CMD提供默认参数

Part 4.8

VOLUME指令

VOLUME语法与示例 Dockerfile 
# 语法格式：
# VOLUME ["/data"]                              # JSON数组格式
# VOLUME /data                                  # 直接指定路径
# VOLUME /var/www /var/log                      # 多个路径
# VOLUME ["/var/www", "/var/log"]               # 多个路径JSON格式

# 参数说明：
# path          挂载点路径（必须是绝对路径）

# 示例1：创建单个卷
VOLUME /data

# 示例2：创建多个卷
VOLUME /data /config

# 示例3：JSON数组格式
VOLUME ["/data", "/config", "/logs"]

# 示例4：数据库应用示例
FROM mysql:8.0
VOLUME /var/lib/mysql
EXPOSE 3306

# 示例5：Web应用示例
FROM nginx:alpine
VOLUME ["/usr/share/nginx/html", "/etc/nginx/conf.d", "/var/log/nginx"]
EXPOSE 80

# 示例6：应用数据持久化
FROM node:18
WORKDIR /app
COPY . .
RUN npm install

# 创建数据卷
VOLUME ["/app/data", "/app/uploads"]

EXPOSE 3000
CMD ["node", "server.js"]

# 使用说明：
# 运行时挂载卷：
# $ docker run -v mydata:/app/data -v uploads:/app/uploads myapp
#
# 或挂载到主机目录：
# $ docker run -v /host/data:/app/data myapp
#
# 查看卷的挂载情况：
# $ docker inspect container_id | grep -A 10 Mounts
VOLUME注意事项

- VOLUME创建的是匿名卷，容器删除后数据仍保留

- 无法在Dockerfile中指定主机挂载点（运行时指定）

- VOLUME路径中的文件会被复制到卷中

- 后续RUN指令对VOLUME的修改不会生效

LABEL version="1.0.0"

# 示例4：使用标准标签（OCI规范推荐）
LABEL org.opencontainers.image.title="My Application"
LABEL org.opencontainers.image.description="A web application built with Node.js"
LABEL org.opencontainers.image.version="1.0.0"
LABEL org.opencontainers.image.authors="your-email@example.com"
LABEL org.opencontainers.image.source="https://github.com/username/repo"

HEALTHCHECK语法与示例 Dockerfile 
# 语法格式：
# HEALTHCHECK [OPTIONS] CMD command
# HEALTHCHECK NONE                              # 禁用健康检查

# 参数说明：
# --interval=DURATION      检查间隔（默认30s）
# --timeout=DURATION       超时时间（默认30s）
# --start-period=DURATION  启动延迟（默认0s）
# --retries=N              重试次数（默认3次）
# CMD command              健康检查命令

# 示例1：基本的HTTP健康检查
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1

# 示例2：使用wget的健康检查
HEALTHCHECK --interval=30s --timeout=3s \
  CMD wget --quiet --tries=1 --spider http://localhost:80/ || exit 1

# 示例3：自定义脚本健康检查
HEALTHCHECK --interval=30s --timeout=10s --start-period=30s --retries=3 \
  CMD /app/healthcheck.sh

# 示例4：数据库连接检查
HEALTHCHECK --interval=10s --timeout=5s --retries=5 \

Part 5.2

多阶段构建

多阶段构建示例（Go应用） Dockerfile 
# 第一阶段：构建阶段
FROM golang:1.21-alpine AS builder

# 安装构建依赖
RUN apk add --no-cache git

# 设置工作目录
WORKDIR /app

# 复制依赖文件
COPY go.mod go.sum ./
RUN go mod download

# 复制源代码
COPY . .

# 构建应用（静态链接）
RUN CGO_ENABLED=0 GOOS=linux go build -a -installsuffix cgo -o main .

# 第二阶段：运行阶段（使用最小镜像）
FROM alpine:latest

# 安装CA证书（用于HTTPS请求）
RUN apk --no-cache add ca-certificates

# 创建非root用户
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

# 设置工作目录
WORKDIR /root/

# 从构建阶段复制二进制文件
COPY --from=builder /app/main .

# 切换到非root用户
USER appuser

# 暴露端口
EXPOSE 8080

# 启动应用
CMD ["./main"]

多阶段构建优势

- 减小镜像体积：只保留运行时必需的文件

- 提高安全性：不包含编译工具等敏感组件

- 优化构建缓存：每个阶段独立缓存

- 支持多种语言：Go、Java、Node.js、Python等都适用

Part 5.3

镜像优化技巧

1. 选择合适的基础镜像

- alpine：最小化，约5MB，适合静态编译语言

- debian:slim：平衡大小和功能，约30MB

- distroless：Google出品，无shell，超安全

- 避免使用latest标签，指定明确版本

2. 减少镜像层数

- 将多个RUN命令用 && 连接

- 清理缓存和临时文件在同一RUN中完成

- 避免不必要的COPY操作

优化前后对比 Dockerfile 
# 优化前（多层，体积大）
FROM ubuntu:22.04
RUN apt-get update
RUN apt-get install -y curl
RUN apt-get install -y vim
RUN apt-get install -y git
RUN rm -rf /var/lib/apt/lists/*

# 优化后（单层，体积小）
FROM ubuntu:22.04
RUN apt-get update && \
    apt-get install -y curl vim git && \
    rm -rf /var/lib/apt/lists/*

# 优化前镜像大小：约250MB
# 优化后镜像大小：约180MB

# 进一步优化（使用slim版本）
FROM debian:bullseye-slim
RUN apt-get update && \
    apt-get install -y --no-install-recommends curl vim git && \
    rm -rf /var/lib/apt/lists/*

# 最终镜像大小：约90MB
3. 利用构建缓存

- 将不常变动的指令放在Dockerfile前面

- COPY依赖文件（package.json）先于源代码

- 使用.dockerignore排除不需要的文件
