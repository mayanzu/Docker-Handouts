Part 1

Docker镜像基础

理解镜像的本质与分层机制

Part 1.1

什么是Docker镜像？

镜像定义
Docker镜像是一个只读模板，包含了运行应用程序所需的所有内容：代码、运行时环境、系统工具、系统库和配置文件。镜像是容器的基础，容器是镜像的运行实例。

可以把镜像理解为类（Class），容器理解为对象（Object）。一个镜像可以创建多个容器实例，每个容器都是独立的运行环境。

镜像的组成

1. 基础镜像层：如ubuntu、alpine等操作系统基础

2. 依赖层：应用程序运行所需的库和工具

3. 应用层：应用程序代码和配置文件

4. 元数据：镜像的配置信息（环境变量、端口、启动命令等）

Part 1.2

镜像分层机制

Docker镜像采用联合文件系统（UnionFS）技术，由多个只读层（Layer）叠加而成。每一层代表Dockerfile中的一条指令。

镜像分层结构 概念说明 
┌────────────────────────────────────────┐
│         应用层 (Application Layer)       │
│         COPY app.py /app/              │
├────────────────────────────────────────┤
│         依赖层 (Dependencies Layer)      │
│         RUN pip install flask          │
├────────────────────────────────────────┤
│         运行时层 (Runtime Layer)        │
│         FROM python:3.11               │
├────────────────────────────────────────┤
│         基础层 (Base Layer)             │
│         Ubuntu / Alpine / Debian       │
└────────────────────────────────────────┘
分层优势

- 共享层：多个镜像可以共享相同的基础层，节省存储空间

- 快速构建：只重建变化的层，未变化层直接使用缓存

- 版本控制：每层有唯一ID，便于追踪和管理

docker history - 查看镜像分层 Bash 
# 查看Ubuntu镜像的分层结构
$ docker history ubuntu:22.04

IMAGE          CREATED        CREATED BY                                      SIZE
605c77e624dd   2 weeks ago    /bin/sh -c #(nop)  CMD ["bash"]                0B
      2 weeks ago    /bin/sh -c #(nop)  ADD file:... in /            80.4MB

# 查看nginx镜像的分层结构
$ docker history nginx:latest

IMAGE          CREATED        CREATED BY                                      SIZE
605c77e624dd   2 weeks ago    /bin/sh -c #(nop)  CMD ["nginx"]              0B
a6bd71dd48d8   2 weeks ago    /bin/sh -c #(nop)  EXPOSE 80 443              0B
...（省略部分输出）
Part 1.3

镜像命名与标签

Docker镜像采用命名:标签的格式来标识不同的镜像版本。理解命名规范对于正确拉取和使用镜像至关重要。

镜像命名格式 格式说明 
[registry/][namespace/]name[:tag][@digest]

# 完整示例
docker.io/library/ubuntu:22.04@sha256:abc123...

# 各部分说明：
# - registry: 仓库地址（默认docker.io）
# - namespace: 命名空间（官方镜像无此部分）
# - name: 镜像名称
# - tag: 标签版本（默认latest）
# - digest: 镜像摘要（唯一标识）

1官方镜像

ubuntu、alpine、debian
docker pull ubuntu:22.04

2用户镜像

username/image-name
docker pull nginx:alpine

3私有仓库

registry.example.com/image
docker pull my-registry.com/myapp:v1

4多架构镜像

自动选择对应架构
docker pull --platform linux/arm64 ubuntu

最佳实践

1. 避免使用latest标签，在生产环境指定明确版本

2. 优先使用摘要digest确保镜像不可变

3. 私有仓库镜像必须包含完整路径

镜像搜索与拉取

掌握从仓库获取镜像的技能

Part 2.1

docker search - 搜索镜像

docker search命令用于在Docker Hub中搜索公开的镜像。通过搜索可以发现官方镜像、社区镜像以及各种应用镜像。

docker search - 搜索镜像 Bash 
# 语法：docker search [OPTIONS] TERM

# 基础用法 - 搜索nginx镜像
$ docker search nginx

# 预期输出：
NAME                      DESCRIPTION                                     STARS     OFFICIAL   AUTOMATED
nginx                     Official build of Nginx.                        19500     [OK]       
jwilder/nginx-proxy       Automated Nginx reverse proxy for docker con…   2100                 [OK]
richarvey/nginx-php-fpm   Container with Nginx and PHP-FPM configured…   1000                 [OK]

# 参数说明：
# --filter=stars=N     只显示星数大于等于N的镜像
# --limit=N            限制返回结果数量（默认25）
# --no-trunc           不截断描述信息
# --format             格式化输出
docker search - 常用示例 Bash 
# 示例1：搜索高星级的Ubuntu镜像
$ docker search --filter=stars=1000 ubuntu

NAME      DESCRIPTION                   STARS     OFFICIAL   AUTOMATED
ubuntu    Ubuntu is a Debian-based…    15000     [OK]       

# 示例2：限制搜索结果数量
$ docker search --limit=5 mysql

NAME      DESCRIPTION                   STARS     OFFICIAL   AUTOMATED
mysql     MySQL is a widely used…      12000     [OK]       
mariadb   MariaDB Server is a…          5000      [OK]       

# 示例3：搜索官方镜像（stars>=1000且官方认证）
$ docker search --filter=stars=1000 --filter=is-official=true nginx

NAME      DESCRIPTION                   STARS     OFFICIAL   AUTOMATED
nginx     Official build of Nginx.      19500     [OK]       

# 示例4：格式化输出
$ docker search --format "{{.Name}} - {{.Description}}" nginx
搜索结果字段说明

- NAME：镜像名称

- DESCRIPTION：镜像描述

- STARS：用户点赞数

- OFFICIAL：[OK]表示官方镜像

- AUTOMATED：[OK]表示自动构建

Part 2.2

docker pull - 拉取镜像

docker pull命令用于从镜像仓库拉取镜像到本地。这是使用Docker的第一步，拉取的镜像会存储在本地镜像仓库中。

docker pull - 拉取镜像 Bash 
# 语法：docker pull [OPTIONS] NAME[:TAG|@DIGEST]

# 拉取最新版ubuntu镜像（默认latest标签）
$ docker pull ubuntu

Using default tag: latest
latest: Pulling from library/ubuntu
d5fd17ec1767: Pull complete
Digest: sha256:9a0bdde4188b896a372804be2384015e90e3f8493751292d5de9e3e5f8a6f3c8
Status: Downloaded newer image for ubuntu:latest
docker.io/library/ubuntu:latest

# 拉取指定版本的Ubuntu
$ docker pull ubuntu:22.04

22.04: Pulling from library/ubuntu
d5fd17ec1767: Pull complete
Digest: sha256:9a0bdde4188b896a372804be2384015e90e3f8493751292d5de9e3e5f8a6f3c8
Status: Downloaded newer image for ubuntu:22.04

# 拉取所有标签的镜像
$ docker pull --all-tags ubuntu

# 拉取指定平台的镜像
$ docker pull --platform linux/arm64 ubuntu:22.04
docker pull - 常用参数详解 Bash 
# 参数说明：
# -a, --all-tags        拉取仓库中所有标签的镜像
# --platform            指定平台（linux/amd64, linux/arm64, linux/arm/v7）
# -q, --quiet           静默模式，只输出镜像ID
# --disable-content-trust 跳过镜像签名验证（默认true）

# 示例：拉取多个常用镜像
$ docker pull nginx:alpine
$ docker pull python:3.11-slim
$ docker pull mysql:8.0
$ docker pull redis:7.2-alpine
$ docker pull node:18-alpine

# 示例：跨平台拉取
$ docker pull --platform linux/amd64 ubuntu:22.04
$ docker pull --platform linux/arm64/v8 ubuntu:22.04

# 示例：静默拉取
$ docker pull -q nginx:alpine
sha256:4b7c53f8a67a4f4a7e5e6f8a9b0c1d2e3f4a5b6c7d8e9f0a1b2c3d4e5f6a7b

Part 2.3

docker push - 推送镜像

docker push命令用于将本地镜像推送到镜像仓库。本节介绍如何推送到Docker Hub和私有仓库。

docker push - 推送镜像到Docker Hub Bash 
# 步骤1：登录Docker Hub
$ docker login
Username: your-username
Password: 
Login Succeeded

# 步骤2：给镜像打标签（格式：username/repository:tag）
$ docker tag myapp:latest your-username/myapp:v1.0

# 步骤3：推送到Docker Hub
$ docker push your-username/myapp:v1.0

# 预期输出：
The push refers to repository [docker.io/your-username/myapp]
a1b2c3d4e5f6: Pushing  10.24MB/100.24MB
...
v1.0: digest: sha256:abc123... size: 1234

# 步骤4：登出（可选）
$ docker logout
docker push - 推送到私有仓库 Bash 
# 步骤1：登录私有仓库
$ docker login registry.example.com
Username: admin
Password: 
Login Succeeded

# 步骤2：给镜像打标签（包含完整仓库地址）
$ docker tag myapp:latest registry.example.com/myteam/myapp:v1.0

# 步骤3：推送到私有仓库
$ docker push registry.example.com/myteam/myapp:v1.0

# 推送完成后登出
$ docker logout registry.example.com
注意事项

- 推送前必须先登录对应仓库

- 镜像标签格式必须匹配仓库地址

- 私有仓库需要相应权限才能推送

Part 3

镜像查看与管理

掌握本地镜像的管理操作

Part 3.1

docker images - 列出本地镜像

docker images（或docker image ls）命令用于列出本地已有的所有镜像，包括镜像ID、大小、标签等信息。

docker images - 列出本地镜像 Bash 
# 列出所有本地镜像
$ docker images

# 预期输出：
REPOSITORY          TAG           IMAGE ID       CREATED        SIZE
nginx               latest        a6bd71dd48d8   2 weeks ago    187MB
ubuntu              22.04         08d22c0f15e2   3 weeks ago    77.8MB
python              3.11-slim     7dcf5b8d3a5e   4 days ago     145MB
mysql               8.0           3218b38490ce   5 days ago     516MB
redis               alpine        c5e2e5c2a5b9   1 week ago     30MB

# 使用新格式命令
$ docker image ls

# 字段说明：
# REPOSITORY - 镜像仓库名
# TAG        - 标签版本
# IMAGE ID   - 镜像唯一ID（前12位）
# CREATED    - 创建时间
# SIZE       - 镜像大小
docker images - 常用参数 Bash 
# 参数说明：
# -a, --all          显示所有镜像（包括中间层）
# -q, --quiet        只显示镜像ID
# --digests          显示镜像摘要
# --filter           过滤输出

# 示例：显示所有镜像（包括中间层）
$ docker images -a

# 示例：只显示镜像ID
$ docker images -q
a6bd71dd48d8
08d22c0f15e2
7dcf5b8d3a5e

# 示例：过滤特定仓库的镜像
$ docker images nginx
$ docker images python:*

# 示例：显示带摘要信息
$ docker images --digests

REPOSITORY   TAG      DIGEST                                                                    IMAGE ID        CREATED        SIZE
nginx        latest   sha256:abc123...                                                         a6bd71dd48d8    2 weeks ago    187MB

# 示例：统计本地镜像数量
$ docker images -q | wc -l
12
镜像大小说明

- 镜像大小是镜像占用的磁盘空间

- 多个标签的同一镜像只占用一份空间

- 中间层镜像不显示在默认列表中

- 使用-a参数可查看所有层

Part 3.2

docker inspect - 查看镜像详情

docker inspect命令用于查看镜像的详细信息，包括创建时间、配置信息、层信息等。对于调试和故障排查非常有用。

docker inspect - 查看镜像详情 Bash

$ docker inspect ubuntu:22.04

        "Architecture": "amd64",

        "Size": 77832000,

            "Env": [

docker inspect - 格式化输出 Bash 
# 使用Go模板格式化输出，只显示特定字段
$ docker inspect --format '{{.Id}}' ubuntu:22.04

# 查看镜像的创建时间
$ docker inspect --format '{{.Created}}' ubuntu:22.04

使用场景

1. 查看配置：了解镜像的环境变量、端口、启动命令

2. 故障排查：检查镜像层信息和依赖

3. 自动化脚本：提取特定信息用于CI/CD

Part 3.3

docker tag - 给镜像打标签

docker tag命令用于给镜像创建一个新的标签（别名）。这个操作不会复制镜像，只是添加一个引用。

docker tag - 给镜像打标签 Bash 
# 语法：docker tag SOURCE_IMAGE[:TAG] TARGET_IMAGE[:TAG]

# 示例1：给镜像添加版本标签
$ docker tag ubuntu:22.04 myapp:1.0

# 示例2：添加多个版本标签
$ docker tag ubuntu:22.04 myapp:v1.0
$ docker tag ubuntu:22.04 myapp:v1.0.0
$ docker tag ubuntu:22.04 myapp:latest

# 示例3：添加完整仓库路径（用于推送）
$ docker tag myapp:v1.0 registry.example.com/myteam/myapp:v1.0

# 示例4：查看标签后的镜像列表
$ docker images | grep myapp

REPOSITORY                      TAG       IMAGE ID       CREATED        SIZE
myapp                           v1.0      08d22c0f15e2   3 weeks ago    77.8MB
myapp                           v1.0.0    08d22c0f15e2   3 weeks ago    77.8MB
myapp                           latest    08d22c0f15e2   3 weeks ago    77.8MB
registry.example.com/myteam/myapp  v1.0    08d22c0f15e2   3 weeks ago    77.8MB
标签操作说明

- docker tag不复制镜像，只是创建新的引用

- 同一镜像可以有多个标签，共享相同的IMAGE ID

- 删除标签不会删除镜像本身

- 推送前必须正确打标签以匹配仓库地址

Part 3.4

docker rmi - 删除本地镜像

docker rmi（或docker image rm）命令用于删除本地镜像。删除镜像前需要确保没有容器正在使用该镜像。

docker rmi - 删除本地镜像 Bash 
# 语法：docker rmi [OPTIONS] IMAGE [IMAGE...]

# 删除单个镜像
$ docker rmi ubuntu:22.04

# 删除多个镜像
$ docker rmi nginx:latest python:3.11

# 使用镜像ID删除
$ docker rmi 08d22c0f15e2

# 参数说明：
# -f, --force     强制删除（即使有容器使用）
# --no-prune      不删除未标记的父镜像

# 示例：强制删除镜像
$ docker rmi -f myapp:v1.0

# 示例：删除所有未使用的镜像
$ docker image prune

# 示例：删除所有悬空镜像（无标签的镜像）
$ docker image prune -a
docker rmi - 常见错误与处理 Bash 
# 错误1：镜像被容器使用
$ docker rmi nginx:latest

Error response from daemon: conflict: unable to remove repository reference 
"nginx:latest" (must force) - container 123abc456def is using its 
referenced image 605c77e624dd

# 解决方法：先删除容器，或强制删除
$ docker rm 123abc456def
$ docker rmi -f nginx:latest

# 错误2：镜像有多个标签
$ docker images | grep myapp
myapp  v1.0  08d22c0f15e2
myapp  v2.0  08d22c0f15e2

# 只删除v1.0标签，保留v2.0
$ docker rmi myapp:v1.0

# 删除镜像（所有标签都会被删除）
$ docker rmi myapp:v2.0
删除前检查

1. 使用docker ps -a检查是否有容器使用该镜像

2. 使用docker image ls确认要删除的镜像

3. 谨慎使用-f强制删除参数

4. 生产环境删除前建议备份或推送到仓库

Part 3.5

docker prune - 清理未使用资源

docker image prune命令用于清理本地镜像，删除不再使用的镜像以释放磁盘空间。

docker image prune - 清理镜像 Bash 
# 删除所有悬空镜像（dangling images）
# 悬空镜像：没有被任何标签引用的镜像
$ docker image prune

WARNING! This will remove all dangling images.
Are you sure? [y/N] y
Deleted: sha256:abc123...
Space reclaimed: 150MB

# 删除所有未使用的镜像
$ docker image prune -a

WARNING! This will remove all dangling images.
Are you sure? [y/N] y

# 参数说明：
# -a, --all       删除所有未使用的镜像（不只是悬空镜像）
# --filter         使用过滤条件
# -f, --force      不提示确认直接删除

# 示例：只清理超过24小时的悬空镜像
$ docker image prune --filter "until=24h"

# 示例：批量清理（自动确认）
$ docker image prune -af

# 示例：查看磁盘使用情况
$ docker system df

TYPE            TOTAL     ACTIVE    SIZE      RECLAIMABLE
Images          45        12        10.5GB    6.8GB (64%)
Containers      8         3         123MB     45MB (36%)
Local Volumes   5         2         1.2GB     500MB (41%)
清理策略建议

- 定期清理：建议每周执行一次清理

- 使用过滤器：清理特定条件的镜像

- 注意保留：不要清理正在使用的镜像

- CI/CD环境：每次构建后清理

Part 4

镜像高级操作

掌握镜像的导出、导入与高级管理

Part 4.1

docker save与docker load - 镜像归档

docker save用于将镜像导出为tar归档文件，docker load用于从归档文件恢复镜像。这常用于镜像的离线传输和备份。

docker save - 导出镜像 Bash 
# 语法：docker save [OPTIONS] IMAGE [IMAGE...]

# 导出单个镜像到文件
$ docker save nginx:alpine -o nginx-alpine.tar

# 导出多个镜像到单个文件
$ docker save ubuntu:22.04 python:3.11 mysql:8.0 -o images.tar

# 使用标准输出（常用于管道）
$ docker save nginx:latest | gzip > nginx-latest.tar.gz

# 导出时包含所有标签
$ docker save myapp -o myapp.tar

# 参数说明：
# -o, --output    指定输出文件（默认输出到标准输出）
# -q, --quiet     静默模式
docker load - 导入镜像 Bash 
# 语法：docker load [OPTIONS]

# 从文件导入镜像
$ docker load -i nginx-alpine.tar

# 或使用标准输入
$ docker load < nginx-alpine.tar

# 从gzip压缩文件导入
$ gunzip -c nginx-latest.tar.gz | docker load

# 参数说明：
# -i, --input     指定输入文件
# -q, --quiet     静默模式

# 预期输出：
Loaded image: nginx:alpine
Loaded image: nginx:latest
docker save/load - 实战示例 Bash 
# 场景1：备份生产环境镜像
$ docker save myapp:v1.0 myapp:v2.0 -o myapp-backup-$(date +%Y%m%d).tar

# 场景2：离线环境部署
# 在有网络的环境导出
$ docker save nginx:alpine redis:7.2 mysql:8.0 -o offline-images.tar
# 传输到离线服务器
$ scp offline-images.tar offline-server:/tmp/
# 在离线服务器导入
$ docker load -i /tmp/offline-images.tar

# 场景3：压缩传输减少带宽
$ docker save nginx:latest | gzip | ssh user@server "cat > nginx.tar.gz"
$ ssh user@server "gunzip -c nginx.tar.gz | docker load"

# 场景4：查看归档内容
$ tar -tvf myapp.tar | head -20
save vs export

| 特性 | docker save | docker export |

|------|------------|--------------|

| 适用对象 | 镜像 | 容器 |

| 保留历史 | ✅ 是 | ❌ 否 |

| 保留元数据 | ✅ 是 | ❌ 否 |

| 文件大小 | 较大 | 较小 |

Part 4.2

镜像标签管理

镜像标签管理是镜像管理的重要部分，包括添加标签、重命名标签和删除标签。这些操作对于镜像版本控制和仓库管理至关重要。

docker tag - 标签操作 Bash 
# 语法：docker tag SOURCE_IMAGE[:TAG] TARGET_IMAGE[:TAG]

# 为镜像添加版本标签
$ docker tag ubuntu:22.04 myapp:1.0

# 添加多个版本标签
$ docker tag ubuntu:22.04 myapp:v1.0
$ docker tag ubuntu:22.04 myapp:v1.0.0
$ docker tag ubuntu:22.04 myapp:latest

# 添加完整仓库路径标签（用于推送）
$ docker tag myapp:v1.0 registry.example.com/myteam/myapp:v1.0

# 查看标签后的镜像列表
$ docker images | grep myapp
docker tag - 实战场景 Bash 
# 场景1：多版本标签管理
# 开发版本
$ docker tag myapp:latest myapp:dev
# 测试版本
$ docker tag myapp:dev myapp:test
# 预发布版本
$ docker tag myapp:test myapp:staging
# 生产版本
$ docker tag myapp:staging myapp:v1.0.0
$ docker tag myapp:v1.0.0 myapp:prod

# 场景2：仓库标签管理
# 推送到Docker Hub
$ docker tag myapp:v1.0 your-username/myapp:v1.0
$ docker push your-username/myapp:v1.0

# 推送到私有仓库
$ docker tag myapp:v1.0 my-registry.com/team/myapp:v1.0
$ docker push my-registry.com/team/myapp:v1.0

# 场景3：删除标签
# 删除旧版本标签
$ docker rmi myapp:v0.9
标签命名最佳实践

1. 语义化版本：使用v1.0.0格式，便于版本管理

2. 环境标签：dev、test、staging、prod

3. Git提交SHA：便于追踪构建来源

4. 日期标签：2024-01-15，便于时间追溯

Part 4.3

镜像历史查看

docker history命令用于查看镜像的构建历史，包括每一层的创建指令、大小和创建时间。这对于理解镜像结构和优化镜像大小非常有用。

docker history - 查看镜像历史 Bash 
# 查看Ubuntu镜像的构建历史
$ docker history ubuntu:22.04

IMAGE          CREATED        CREATED BY                                      SIZE
605c77e624dd   2 weeks ago    /bin/sh -c #(nop)  CMD ["bash"]                0B
      2 weeks ago    /bin/sh -c #(nop)  ADD file:... in /            80.4MB

# 查看nginx镜像的详细历史
$ docker history nginx:latest

IMAGE          CREATED        CREATED BY                                       SIZE
605c77e624dd   2 weeks ago    /bin/sh -c #(nop)  CMD ["nginx"]                0B
a6bd71dd48d8   2 weeks ago    /bin/sh -c #(nop)  EXPOSE 80 443               0B
c9c3b8b7f4a3   2 weeks ago    /bin/sh -c #(nop)  COPY file:... in /etc/...    0.5MB
...

# 参数说明：
# --no-trunc     不截断输出
# -q, --quiet    只显示镜像ID
# --format       格式化输出
docker history - 分析镜像 Bash 
# 不截断输出，查看完整命令
$ docker history --no-trunc nginx:latest

# 只显示镜像层ID
$ docker history -q nginx:latest

# 格式化输出
$ docker history --format "{{.ID}}: {{.CreatedBy}}" nginx:latest

# 查看镜像各层大小占比
$ docker history nginx:latest --format "table {{.Size}}\t{{.CreatedBy}}"

# 场景：分析自定义镜像的构建层
$ docker build -t myapp:latest .
$ docker history myapp:latest
镜像优化提示

通过docker history分析镜像，可以发现：

1. 大层：找出占用空间大的层，考虑优化

2. 重复层：识别重复的安装操作

3. 清理时机：在构建时添加清理命令减少最终大小
