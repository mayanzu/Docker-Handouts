Part 0.1

Docker的定义与核心价值

Docker定义
Docker是一个开源的容器化平台，它允许开发者将应用程序及其依赖打包到一个轻量级、可移植的容器中，从而实现"一次构建，到处运行"的目标。Docker通过操作系统级虚拟化技术，在共享宿主机内核的前提下提供隔离的运行环境。

Docker的核心理念源自集装箱（Container）的思想：就像集装箱标准化了货物运输，Docker标准化了应用程序的打包和分发。无论应用运行在开发环境、测试环境还是生产环境，容器内的行为都保持一致。

1环境一致性

消除"在我机器上能跑"的问题，开发、测试、生产环境完全一致

2快速交付

从代码到部署的时间从数天缩短到数分钟，加速DevOps流程

3资源高效

相比虚拟机节省90%资源，单机可运行数百个容器实例

4可移植性

跨平台运行：Linux、Windows、macOS、云平台无缝迁移

Docker的核心价值主张

Build Once, Run Anywhere - 一次构建，到处运行

Ship Anywhere - 任意环境部署

Scale Easily - 轻松扩展伸缩

Part 0.2

Docker整体架构组成

Docker采用客户端-服务器（C/S）架构，由多个核心组件协同工作：

1Docker引擎（Engine）

Docker的核心运行时，包含Docker Daemon（守护进程）、Docker CLI（命令行接口）和containerd（容器运行时）

2Docker客户端（Client）

用户与Docker交互的接口，通过REST API与Daemon通信，支持本地和远程连接

3Docker守护进程（Daemon）

dockerd后台服务，监听API请求，管理镜像、容器、网络和存储卷

4Docker注册中心（Registry）

存储和分发Docker镜像的服务，如Docker Hub、私有仓库

Part 0.3

Docker三要素概述

Docker的三大核心概念是镜像（Image）、容器（Container）和仓库（Registry），它们构成了Docker技术体系的基础。

1镜像（Image）

只读模板，包含运行应用所需的所有内容：代码、运行时、库、配置文件等。镜像是容器的蓝图。

2容器（Container）

镜像的运行实例，提供隔离的运行环境。容器可以被创建、启动、停止、删除。

3仓库（Registry）

存储和分发镜像的中央存储，如Docker Hub、阿里云镜像仓库、私有仓库等。

Part 0.4

Docker镜像详解

镜像定义
Docker镜像是一个只读模板，采用分层存储（Layered）架构，每一层代表Dockerfile中的一条指令。镜像包含运行容器所需的所有内容：代码、运行时环境、库、环境变量和配置文件。
镜像分层结构示意 概念说明 
┌────────────────────────────────────────┐
│            应用层 (Application Layer)    │  ← 你的代码
│            COPY app.py /app/            │
├────────────────────────────────────────┤
│            依赖层 (Dependencies Layer)   │  ← 依赖库
│            RUN pip install flask        │
├────────────────────────────────────────┤
│            运行时层 (Runtime Layer)      │  ← Python运行时
│            FROM python:3.9              │
├────────────────────────────────────────┤
│            基础层 (Base Layer)           │  ← 操作系统
│            Ubuntu / Alpine / Debian     │
└────────────────────────────────────────┘

分层优势：
1. 共享层：多个镜像可共享相同的基础层，节省存储
2. 快速构建：只重建变化的层，未变化层使用缓存
3. 高效分发：只需传输差异层，加快镜像拉取
镜像命名规范

- 官方镜像：nginx、ubuntu、python（无命名空间前缀）

- 用户镜像：username/image-name（带用户名前缀）

- 私有仓库：registry.example.com/namespace/image:tag

- 标签格式：image:tag（默认latest）、image@sha256:digest

镜像常用操作

- docker pull：从仓库拉取镜像

- docker images：列出本地镜像

- docker rmi：删除本地镜像

- docker build：从Dockerfile构建镜像

- docker tag：给镜像打标签

Part 0.5

Docker容器详解

容器定义
容器是镜像的运行实例，是一个独立运行的软件单元。容器在镜像的基础上添加了一个可写层（Container Layer），所有运行时的修改都发生在这个可写层中，采用写时复制（Copy-on-Write）机制。
容器与镜像的关系 概念说明

│                    容器运行时结构                      │
├─────────────────────────────────────────────────────┤
│  ┌─────────────────────────────────────────────┐    │
│  │        可写层 (Container Layer)              │    │
│  │   - 运行时修改的文件                          │    │
│  │   - 应用产生的数据                            │    │
│  │   - 容器删除后数据丢失                        │    │
│  └─────────────────────────────────────────────┘    │
│                      ▲                              │
│                      │ 写时复制 (CoW)               │
│  ┌───────────────────┴───────────────────────┐      │
│  │        只读镜像层 (Image Layers)            │      │
│  │   Layer 3: 应用代码                         │      │
│  │   Layer 2: 依赖库                           │      │
│  │   Layer 1: 运行时环境                       │      │
│  │   Layer 0: 基础操作系统                     │      │
│  └─────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────┘

容器生命周期：
  Created → Running → Paused → Stopped → Deleted
     │         │         │         │         │
  docker   docker    docker    docker    docker
  create   start     pause     stop       rm
容器核心特性

1. 隔离性：进程、网络、文件系统、用户相互隔离

2. 轻量级：共享宿主机内核，启动秒级，资源占用少

3. 可移植：在任何Docker环境中行为一致

4. 可控制：可限制CPU、内存、IO等资源使用

数据持久化

容器删除后，可写层数据会丢失。如需持久化数据，应使用数据卷（Volume）或绑定挂载（Bind Mount）。

Part 0.6

Docker仓库详解

仓库定义
Docker仓库是存储和分发Docker镜像的服务。仓库分为公共仓库和私有仓库，开发者可以从仓库拉取镜像，也可以将本地镜像推送到仓库进行共享。
仓库类型 代表产品 特点
公共仓库 Docker Hub 官方镜像源，免费使用，有速率限制
云厂商仓库 阿里云、腾讯云、AWS ECR 国内访问快，企业级服务
私有仓库 Harbor、Nexus、Registry 自建服务，安全可控
仓库操作命令 Bash 
# 登录仓库
$ docker login registry.example.com
Username: your-username
Password: 
Login Succeeded

# 拉取镜像
$ docker pull nginx:latest
$ docker pull registry.example.com/myapp:v1.0

# 推送镜像（需先打标签）
$ docker tag myapp:v1.0 registry.example.com/myapp:v1.0
$ docker push registry.example.com/myapp:v1.0

# 搜索镜像
$ docker search nginx

# 登出仓库
$ docker logout registry.example.com
Docker Hub镜像命名规则

- nginx = docker.io/library/nginx:latest（官方镜像）

- username/nginx = docker.io/username/nginx:latest（用户镜像）

- registry.com/nginx = 私有仓库镜像

Part 0.7

Docker工作原理

Docker基于Linux内核的以下关键技术实现容器的隔离和资源限制：

技术 功能 作用
Namespace 命名空间隔离 隔离进程、网络、文件系统、用户等资源
Cgroups 控制组 限制和监控CPU、内存、IO等资源使用
UnionFS 联合文件系统 实现镜像分层存储和写时复制
Namespace隔离类型

- PID Namespace：进程ID隔离，容器内进程独立编号

- Network Namespace：网络隔离，独立网络栈

- Mount Namespace：文件系统挂载点隔离

- UTS Namespace：主机名和域名隔离

- IPC Namespace：进程间通信隔离

- User Namespace：用户和用户组隔离

Docker工作流程 流程说明 
1. 用户执行 docker run nginx
   ↓
2. Docker CLI 解析命令，通过REST API发送给Daemon
   ↓
3. Daemon 检查本地是否存在nginx镜像
   ↓
4. 不存在则从Registry拉取镜像（分层下载）
   ↓
5. 使用UnionFS叠加镜像层，创建可写层
   ↓
6. 通过Namespace创建隔离环境
   ↓
7. 通过Cgroups设置资源限制
   ↓
8. 启动容器进程，分配独立网络
   ↓
9. 容器运行中，用户可通过CLI管理

理解镜像的本质

Part 1.1

什么是Docker镜像？

镜像定义
Docker镜像是一个只读模板，包含了运行应用程序所需的所有内容：代码、运行时环境、系统工具、系统库和配置。镜像是容器的基础，容器是镜像的运行实例。

可以把镜像理解为类（Class），容器理解为对象（Object）。一个镜像可以创建多个容器实例，每个容器都是独立的运行环境。

镜像的组成

1. 基础镜像层：如ubuntu、alpine等操作系统基础

2. 依赖层：应用程序运行所需的库和工具

3. 应用层：应用程序代码和配置文件

4. 元数据：镜像的配置信息（环境变量、端口、启动命令等）

Part 1.2

镜像分层机制

Docker镜像采用联合文件系统（UnionFS）技术，由多个只读层（Layer）叠加而成。每一层代表Dockerfile中的一条指令。

分层优势

- 共享层：多个镜像可以共享相同的基础层，节省存储空间

- 快速构建：只重建变化的层，未变化层直接使用缓存

- 版本控制：每层有唯一ID，便于追踪和管理

Part 1.3

写时复制（CoW）

写时复制机制
写时复制（Copy-on-Write）是Docker镜像的核心技术。当容器启动时，Docker会在镜像的最上层添加一个可写层（Container Layer）。所有对文件的修改都发生在这个可写层，而不会改变底层的只读镜像层。

1读取文件

直接从镜像的只读层读取，性能高效

2修改文件

将文件从只读层复制到可写层，然后修改

3删除文件

在可写层创建删除标记，原文件仍保留在镜像层

4创建文件

直接在可写层创建新文件

理解要点

容器删除后，可写层数据会丢失。如需持久化数据，应使用数据卷（Volume）或绑定挂载（Bind Mount）。

Part 4.1

FROM指令

FROM指令
FROM指令指定构建镜像的基础镜像，必须是Dockerfile的第一条指令（除ARG外）。基础镜像可以是官方镜像、私有镜像或scratch（空镜像）。
FROM语法与示例 Dockerfile 
# 语法格式：
# FROM [--platform=] [:] [AS ]
# FROM [--platform=] [@] [AS ]

# 参数说明：
# --platform    指定平台（如linux/amd64, linux/arm64）
# image         基础镜像名称
# tag           镜像标签（默认latest）
# digest        镜像摘要（SHA256）
# AS name       多阶段构建时指定阶段名称

# 示例1：使用官方镜像的最新版
FROM ubuntu

# 示例2：使用指定版本的镜像
FROM ubuntu:22.04

# 示例3：使用指定平台的镜像
FROM --platform=linux/amd64 nginx:alpine

# 示例4：使用镜像摘要（确保使用特定版本）
FROM ubuntu@sha256:9a0bdde4188b896a372804be2384015e90e3f8493751292d5de9e3e5f8a6f3c8

# 示例5：多阶段构建，指定阶段名称
FROM golang:1.19 AS builder

# 示例6：从零开始构建（空镜像）
FROM scratch
常用基础镜像推荐

- ubuntu:22.04：完整的Ubuntu系统，适合通用应用

- alpine:latest：轻量级Linux，仅5MB，适合微服务

- debian:bullseye-slim：精简版Debian，平衡大小和功能

- scratch：空镜像，用于静态编译的二进制程序

Part 4.2

RUN指令

RUN指令
RUN指令在镜像构建过程中执行命令，并提交结果作为新层。用于安装软件包、下载文件、编译代码等构建时操作。
RUN语法与示例 Dockerfile 
# 语法格式：
# RUN                     # Shell格式（默认使用/bin/sh -c）
# RUN ["executable", "param1", "param2"]  # Exec格式（JSON数组）

# 参数说明：
# --mount=type=...    挂载缓存或秘密（BuildKit特性）
# --network=...       指定网络模式

# 示例1：Shell格式（推荐用于简单的shell命令）
RUN apt-get update && apt-get install -y nginx

# 示例2：Exec格式（推荐用于避免shell字符串处理问题）
RUN ["apt-get", "update"]
RUN ["apt-get", "install", "-y", "nginx"]

# 示例3：多行命令（使用&&连接，减少镜像层数）
RUN apt-get update && \
    apt-get install -y \
        curl \
        vim \
        git \
        wget && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*

# 示例4：使用管道
RUN wget -O - https://example.com/script.sh | bash

# 示例5：切换shell（使用其他shell执行）
SHELL ["/bin/bash", "-c"]
RUN echo "Using bash instead of sh"

# 示例6：使用BuildKit缓存挂载（加速重复构建）
# syntax=docker/dockerfile:1
RUN --mount=type=cache,target=/var/cache/apt \
    apt-get update && apt-get install -y nginx

# 示例7：使用秘密挂载（安全传递敏感信息）
RUN --mount=type=secret,id=mysecret cat /run/secrets/mysecret

# 示例8：指定网络模式
RUN --network=none echo "No network access"
RUN --network=host git clone https://github.com/example/repo.git
最佳实践

- 将多个相关命令用 && 连接，减少镜像层数

- 安装完成后清理缓存（apt-get clean、rm -rf /var/cache）

- 将不常变动的指令放在Dockerfile前面，充分利用缓存

COPY与ADD指令

COPY语法与示例 Dockerfile 
# 语法格式：
# COPY [--chown=:] [--chmod=] ... 
# COPY [--chown=:] [--chmod=] ["",... ""]

# 参数说明：
# --chown       设置文件所有者和组
# --chmod       设置文件权限（BuildKit特性）
# --from        从其他构建阶段或镜像复制
# src           源文件/目录（相对构建上下文）
# dest          目标路径（镜像内路径）

# 示例1：复制单个文件
COPY app.py /app/

# 示例2：复制多个文件到目录
COPY app.py config.py requirements.txt /app/

# 示例3：复制目录
COPY src/ /app/src/

# 示例4：使用通配符
COPY *.txt /app/
COPY app.* /app/

# 示例5：设置文件所有者
COPY --chown=www-data:www-data app.py /app/

# 示例6：设置文件权限
COPY --chmod=755 entrypoint.sh /app/

# 示例7：从其他构建阶段复制（多阶段构建）
COPY --from=builder /app/build /app/

# 示例8：从其他镜像复制
COPY --from=nginx:latest /etc/nginx/nginx.conf /app/nginx.conf

# 示例9：保留空格的文件名（使用JSON数组格式）
COPY ["my file.txt", "/app/"]
COPY ["config file.json", "settings.json", "/app/config/"]
ADD语法与示例 Dockerfile 
# 语法格式：
# ADD [--chown=:] [--chmod=] ... 
# ADD [--chown=:] [--chmod=] ["",... ""]

# ADD与COPY的区别：
# 1. ADD可以自动解压tar文件（gzip, bzip2, xz）
# 2. ADD可以从URL下载文件

# 示例1：复制本地tar文件并自动解压
ADD archive.tar.gz /app/

# 示例2：从URL下载文件（不推荐，使用curl或wget替代）
ADD https://example.com/file.txt /app/

# 示例3：从URL下载tar并自动解压
ADD https://example.com/archive.tar.gz /app/

# 示例4：使用通配符
ADD *.tar.gz /app/

# 示例5：设置文件所有者
ADD --chown=root:root config.tar.gz /etc/

# 示例6：保留权限的复制
ADD --chmod=644 config.ini /app/
COPY vs ADD选择建议

优先使用COPY，行为明确可预测。仅在需要自动解压tar文件时使用ADD。从URL下载文件建议使用RUN配合curl/wget，更透明可控。

ENV与ARG指令

ENV语法与示例 Dockerfile 
# 语法格式：
# ENV             # 单个变量
# ENV = ...       # 多个变量

# 参数说明：
# key           环境变量名
# value         环境变量值

# 示例1：设置单个环境变量
ENV NODE_VERSION 18.17.0

# 示例2：设置多个环境变量

# 示例5：设置默认环境变量（可被覆盖）

ENV LOG_LEVEL=info

# 示例6：使用环境变量构建其他变量
ENV BASE_URL=https://api.example.com
ENV API_URL=$BASE_URL/v1

# 运行时的效果：
# 容器启动后，这些环境变量会保留在容器中
# 可以通过 docker run -e 覆盖
# $ docker run -e DATABASE_URL=postgres://prod:5432/mydb myapp
ARG语法与示例 Dockerfile 
# 语法格式：
# ARG [=]

# 参数说明：
# name          参数名
# default value 默认值（可选）

# 示例1：定义构建参数
ARG VERSION=latest

# 示例2：使用构建参数
FROM node:$VERSION

# docker build --build-arg VERSION=16 -t myapp:16 .

FROM node:$VERSION
ARG VERSION              # 需要重新定义才能在FROM后使用
RUN echo "Building version: $VERSION"

# 示例7：预定义ARG（无需声明即可使用）
# TARGETPLATFORM - 目标平台（如linux/amd64）
# TARGETOS - 目标操作系统
# TARGETARCH - 目标架构
# TARGETVARIANT - 目标变体
# BUILDPLATFORM - 构建平台
# BUILDOS - 构建操作系统
# BUILDARCH - 构建架构
# BUILDVARIANT - 构建变体

FROM alpine

# 构建命令示例：
# $ docker build --build-arg NODE_VERSION=16 --build-arg APP_ENV=development -t myapp .
ENV vs ARG区别

- ARG：构建时变量，不会保留在最终镜像中

- ENV：运行时变量，会保留在容器中，可被覆盖

- 敏感信息（密码、密钥）应使用ARG而非ENV

镜像优化技巧

1. 选择合适的基础镜像

- alpine：最小化，约5MB，适合静态编译语言

- debian:slim：平衡大小和功能，约30MB

- distroless：Google出品，无shell，超安全

- 避免使用latest标签，指定明确版本

2. 减少镜像层数

- 将多个RUN命令用 && 连接

- 清理缓存和临时文件在同一RUN中完成

- 避免不必要的COPY操作

优化前后对比 Dockerfile 
# 优化前（多层，体积大）
FROM ubuntu:22.04
RUN apt-get update
RUN apt-get install -y curl
RUN apt-get install -y vim
RUN apt-get install -y git
RUN rm -rf /var/lib/apt/lists/*

# 优化后（单层，体积小）
FROM ubuntu:22.04
RUN apt-get update && \
    apt-get install -y curl vim git && \
    rm -rf /var/lib/apt/lists/*

# 优化前镜像大小：约250MB
# 优化后镜像大小：约180MB

# 进一步优化（使用slim版本）
FROM debian:bullseye-slim
RUN apt-get update && \
    apt-get install -y --no-install-recommends curl vim git && \
    rm -rf /var/lib/apt/lists/*

# 最终镜像大小：约90MB
3. 利用构建缓存

- 将不常变动的指令放在Dockerfile前面

- COPY依赖文件（package.json）先于源代码

- 使用.dockerignore排除不需要的文件
