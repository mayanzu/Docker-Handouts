Part 0.1

Docker的定义与核心价值

Docker定义
Docker是一个开源的容器化平台，它允许开发者将应用程序及其依赖打包到一个轻量级、可移植的容器中，从而实现"一次构建，到处运行"的目标。Docker通过操作系统级虚拟化技术，在共享宿主机内核的前提下提供隔离的运行环境。

Docker的核心理念源自集装箱（Container）的思想：就像集装箱标准化了货物运输，Docker标准化了应用程序的打包和分发。无论应用运行在开发环境、测试环境还是生产环境，容器内的行为都保持一致。

1环境一致性

消除"在我机器上能跑"的问题，开发、测试、生产环境完全一致

2快速交付

从代码到部署的时间从数天缩短到数分钟，加速DevOps流程

3资源高效

相比虚拟机节省90%资源，单机可运行数百个容器实例

4可移植性

跨平台运行：Linux、Windows、macOS、云平台无缝迁移

Docker的核心价值主张

Build Once, Run Anywhere - 一次构建，到处运行

Ship Anywhere - 任意环境部署

Scale Easily - 轻松扩展伸缩

Part 0.2

Docker整体架构组成

Docker采用客户端-服务器（C/S）架构，由多个核心组件协同工作：

1Docker引擎（Engine）

Docker的核心运行时，包含Docker Daemon（守护进程）、Docker CLI（命令行接口）和containerd（容器运行时）

2Docker客户端（Client）

用户与Docker交互的接口，通过REST API与Daemon通信，支持本地和远程连接

3Docker守护进程（Daemon）

dockerd后台服务，监听API请求，管理镜像、容器、网络和存储卷

4Docker注册中心（Registry）

存储和分发Docker镜像的服务，如Docker Hub、私有仓库

Part 0.3

Docker三要素概述

Docker的三大核心概念是镜像（Image）、容器（Container）和仓库（Registry），它们构成了Docker技术体系的基础。

1镜像（Image）

只读模板，包含运行应用所需的所有内容：代码、运行时、库、配置文件等。镜像是容器的蓝图。

2容器（Container）

镜像的运行实例，提供隔离的运行环境。容器可以被创建、启动、停止、删除。

3仓库（Registry）

存储和分发镜像的中央存储，如Docker Hub、阿里云镜像仓库、私有仓库等。

Part 0.4

Docker镜像详解

镜像定义
Docker镜像是一个只读模板，采用分层存储（Layered）架构，每一层代表Dockerfile中的一条指令。镜像包含运行容器所需的所有内容：代码、运行时环境、库、环境变量和配置文件。
镜像分层结构示意 概念说明 
┌────────────────────────────────────────┐
│            应用层 (Application Layer)    │  ← 你的代码
│            COPY app.py /app/            │
├────────────────────────────────────────┤
│            依赖层 (Dependencies Layer)   │  ← 依赖库
│            RUN pip install flask        │
├────────────────────────────────────────┤
│            运行时层 (Runtime Layer)      │  ← Ubuntu系统
│            FROM ubuntu:22.04            │
├────────────────────────────────────────┤
│            基础层 (Base Layer)           │  ← 操作系统
│            Ubuntu / Alpine / Debian     │
└────────────────────────────────────────┘

分层优势：
1. 共享层：多个镜像可共享相同的基础层，节省存储
2. 快速构建：只重建变化的层，未变化层使用缓存
3. 高效分发：只需传输差异层，加快镜像拉取
镜像命名规范

- 官方镜像：ubuntu、alpine、debian（无命名空间前缀）

- 用户镜像：username/image-name（带用户名前缀）

- 私有仓库：registry.example.com/namespace/image:tag

- 标签格式：image:tag（默认latest）、image@sha256:digest

镜像常用操作

- docker pull：从仓库拉取镜像

- docker images：列出本地镜像

- docker rmi：删除本地镜像

- docker build：从Dockerfile构建镜像

- docker tag：给镜像打标签

Part 0.5

Docker容器详解

容器定义
容器是镜像的运行实例，是一个独立运行的软件单元。容器在镜像的基础上添加了一个可写层（Container Layer），所有运行时的修改都发生在这个可写层中，采用写时复制（Copy-on-Write）机制。
容器与镜像的关系 概念说明 
┌─────────────────────────────────────────────────────┐
│                    容器运行时结构                      │
├─────────────────────────────────────────────────────┤
│  ┌─────────────────────────────────────────────┐    │
│  │        可写层 (Container Layer)              │    │
│  │   - 运行时修改的文件                          │    │
│  │   - 应用产生的数据                            │    │
│  │   - 容器删除后数据丢失                        │    │
│  └─────────────────────────────────────────────┘    │
│                      ▲                              │
│                      │ 写时复制 (CoW)               │
│  ┌───────────────────┴───────────────────────┐      │
│  │        只读镜像层 (Image Layers)            │      │
│  │   Layer 3: 应用代码                         │      │
│  │   Layer 2: 依赖库                           │      │
│  │   Layer 1: 运行时环境                       │      │
│  │   Layer 0: 基础操作系统                     │      │
│  └─────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────┘

容器生命周期：
  Created → Running → Paused → Stopped → Deleted
     │         │         │         │         │
  docker   docker    docker    docker    docker
  create   start     pause     stop       rm
容器核心特性

1. 隔离性：进程、网络、文件系统、用户相互隔离

2. 轻量级：共享宿主机内核，启动秒级，资源占用少

3. 可移植：在任何Docker环境中行为一致

4. 可控制：可限制CPU、内存、IO等资源使用

数据持久化

容器删除后，可写层数据会丢失。如需持久化数据，应使用数据卷（Volume）或绑定挂载（Bind Mount）。

Part 0.6

Docker仓库详解

仓库定义
Docker仓库是存储和分发Docker镜像的服务。仓库分为公共仓库和私有仓库，开发者可以从仓库拉取镜像，也可以将本地镜像推送到仓库进行共享。
仓库类型 代表产品 特点
公共仓库 Docker Hub 官方镜像源，免费使用，有速率限制
云厂商仓库 阿里云、腾讯云、AWS ECR 国内访问快，企业级服务
私有仓库 Harbor、Nexus、Registry 自建服务，安全可控
Docker Hub镜像命名规则

- ubuntu = docker.io/library/ubuntu:latest（官方镜像）

- username/ubuntu = docker.io/username/ubuntu:latest（用户镜像）

- registry.com/ubuntu = 私有仓库镜像

Part 0.6.1

docker login - 仓库登录

docker login命令用于登录到Docker仓库。登录后才能从私有仓库拉取镜像或向私有仓库推送镜像。

docker login - 登录仓库 Bash 
# 登录到Docker Hub
$ docker login

# 登录到私有仓库
$ docker login registry.example.com

# 预期输出：
Username: your-username
Password:
Login Succeeded

# 登录后，凭证会保存在 ~/.docker/config.json
参数说明

- 无参数：登录Docker Hub（默认）

- registry URL：登录到指定的私有仓库

安全提示

密码不会以明文形式保存，Docker使用加密方式存储凭证。

Part 0.6.2

docker pull - 拉取镜像

docker pull命令用于从仓库拉取镜像到本地。如果本地已有该镜像，会使用已存在的版本。

docker pull - 拉取镜像 Bash 
# 拉取官方镜像（默认tag为latest）
$ docker pull ubuntu:latest

# 拉取特定版本标签
$ docker pull ubuntu:22.04

# 拉取私有仓库镜像
$ docker pull registry.example.com/myapp:v1.0

# 拉取所有标签（如果仓库支持）
$ docker pull -a ubuntu
参数说明

- NAME[:TAG|@DIGEST]：镜像名称和标签

- -a, --all-tags：下载所有标签的镜像

- --platform：指定平台（多平台镜像）

镜像拉取流程

1. 解析镜像名称，确定仓库地址

2. 向仓库API请求镜像清单

3. 从仓库下载各个镜像层

4. 组合所有层创建本地镜像

Part 0.6.3

docker tag 与 docker push - 推送镜像

docker push命令用于将本地镜像推送到远程仓库。推送前需要先用docker tag给镜像打上目标仓库的标签。

docker tag 与 docker push - 推送镜像 Bash 
# 1. 给镜像打标签（格式：registry/namespace/image:tag）
$ docker tag myapp:v1.0 registry.example.com/myapp:v1.0

# 2. 推送到私有仓库
$ docker push registry.example.com/myapp:v1.0

# 3. 推送成功输出示例：
# The push refers to repository [registry.example.com/myapp]
# v1.0: digest: sha256:abc123... size: 2048
docker tag参数说明

- SOURCE_IMAGE：源镜像名称或ID

- TARGET_IMAGE：目标镜像名称（通常包含仓库地址）

docker push参数说明

- NAME[:TAG]：要推送的镜像名称

- -a, --all-tags：推送所有标签

- --disable-content-trust：跳过镜像签名验证

注意事项

推送前必须先登录到目标仓库，且当前用户有推送权限。

Part 0.6.4

docker search 与 docker logout

docker search命令用于在Docker Hub搜索公开镜像。docker logout命令用于从已登录的仓库登出。

docker search - 搜索镜像 Bash 
# 搜索镜像
$ docker search ubuntu

# 预期输出：
NAME                DESCRIPTION                STARS    OFFICIAL    AUTOMATED
ubuntu              Ubuntu is a Debian-based…  12345    [OK]
...


# 搜索私有仓库镜像（需先登录）
$ docker search registry.example.com/myapp

# 参数说明：
# --limit 100：限制搜索结果数量
# --filter stars=100：筛选stars大于100的镜像
docker logout - 登出仓库 Bash 
# 登出Docker Hub
$ docker logout

# 登出私有仓库
$ docker logout registry.example.com

# 登出后，凭证会从 ~/.docker/config.json 中移除
仓库操作完整流程

1. docker login - 登录仓库

2. docker pull - 拉取镜像

3. docker tag - 打标签

4. docker push - 推送镜像

5. docker search - 搜索镜像

6. docker logout - 登出仓库

Part 0.7

Docker工作原理

Docker基于Linux内核的以下关键技术实现容器的隔离和资源限制：

技术 功能 作用
Namespace 命名空间隔离 隔离进程、网络、文件系统、用户等资源
Cgroups 控制组 限制和监控CPU、内存、IO等资源使用
UnionFS 联合文件系统 实现镜像分层存储和写时复制
Namespace隔离类型

- PID Namespace：进程ID隔离，容器内进程独立编号

- Network Namespace：网络隔离，独立网络栈

- Mount Namespace：文件系统挂载点隔离

- UTS Namespace：主机名和域名隔离

- IPC Namespace：进程间通信隔离

- User Namespace：用户和用户组隔离

Docker工作流程 流程说明 
1. 用户执行 docker run ubuntu
   ↓
2. Docker CLI 解析命令，通过REST API发送给Daemon
   ↓
3. Daemon 检查本地是否存在ubuntu镜像
   ↓
4. 不存在则从Registry拉取镜像（分层下载）
   ↓
5. 使用UnionFS叠加镜像层，创建可写层
   ↓
6. 通过Namespace创建隔离环境
   ↓
7. 通过Cgroups设置资源限制
   ↓
8. 启动容器进程，分配独立网络
   ↓
9. 容器运行中，用户可通过CLI管理

Part 0.5.1

Docker服务状态查看

在使用Docker之前，需要了解如何管理Docker服务的运行状态。systemctl status docker命令用于查看Docker服务的运行状态。

systemctl status - 查看Docker服务状态 Bash 
# 查看Docker服务状态
$ sudo systemctl status docker

# 预期输出：
● docker.service - Docker Application Container Engine
     Loaded: loaded (/lib/systemd/system/docker.service; enabled; vendor preset: enabled)
     Active: active (running) since Mon 2024-01-15 10:00:00 UTC; 1h 30min ago
       Docs: https://docs.docker.com
   Main PID: 1234 (dockerd)
      Tasks: 25
     Memory: 120.5M
     CGroup: /system.slice/docker.service
             └─1234 /usr/bin/dockerd -H fd:// --containerd=/run/containerd/containerd.sock
输出信息解读

- Loaded：服务是否已加载到系统，enabled表示开机自启

- Active：服务运行状态，active (running) 表示正常运行中

- Main PID：主进程ID，1234 表示dockerd进程

- Tasks：服务包含的线程数

- Memory：服务使用的内存

服务状态说明

- active (running)：服务正在运行

- inactive (dead)：服务已停止

- failed：服务启动失败，需检查日志

- enabled：已设置开机自启

Docker服务启停管理

使用systemctl命令可以启动、停止和重启Docker服务。

systemctl start/stop/restart - 服务启停控制 Bash 
# 启动Docker服务
$ sudo systemctl start docker

# 停止Docker服务
$ sudo systemctl stop docker

# 重启Docker服务（停止后自动启动）
$ sudo systemctl restart docker

# 重载配置（不中断服务，仅重载配置）
$ sudo systemctl reload docker
# 注意：reload不一定支持所有服务
使用场景

- start：服务未运行且需要启动时

- stop：需要临时停止服务时

- restart：修改配置后需要重新加载，或服务异常时

Docker服务开机自启管理

systemctl enable/disable命令用于设置Docker服务开机自启动。systemctl daemon-reload用于重新加载systemd配置。

systemctl enable/disable - 开机自启设置 Bash 
# 设置Docker服务开机自启动
$ sudo systemctl enable docker

# 取消Docker服务开机自启动
$ sudo systemctl disable docker

# 检查服务是否为开机自启
$ systemctl is-enabled docker
enabled
systemctl daemon-reload - 重新加载配置 Bash 
# 修改Docker配置文件后，需要重新加载systemd配置
$ sudo systemctl daemon-reload

# 然后重启Docker服务使配置生效
$ sudo systemctl restart docker
常见配置修改场景

修改/etc/docker/daemon.json后需要执行：

1. sudo systemctl daemon-reload - 重载systemd配置

2. sudo systemctl restart docker - 重启Docker服务

Docker命令体系概览

Docker命令分为管理命令（Management Commands）和顶层命令（Commands）两大类，理解命令分类有助于快速定位所需命令。

命令类别 管理对象 常用命令示例
docker image 镜像管理 ls, pull, push, rm, build, tag
docker container 容器管理 ls, run, start, stop, rm, exec, logs
docker network 网络管理 ls, create, rm, connect, disconnect
docker volume 存储卷管理 ls, create, rm, inspect, prune
docker system 系统管理 info, df, prune, events
新旧命令格式对比 Bash 
# Docker 1.13+ 引入了新的管理命令格式，两种格式等效：

# 镜像操作
docker images          ≈  docker image ls
docker rmi ubuntu       ≈  docker image rm ubuntu
docker pull ubuntu      ≈  docker image pull ubuntu

# 容器操作
docker ps              ≈  docker container ls
docker rm container    ≈  docker container rm container
docker run ubuntu       ≈  docker container run ubuntu

# 推荐使用新的管理命令格式，语义更清晰
# 本课程将同时介绍两种格式，以适应不同环境
