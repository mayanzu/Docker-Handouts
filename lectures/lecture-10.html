1.1

什么是Docker Compose？

Docker Compose定义
Docker Compose是一个用于定义和运行多容器Docker应用的工具。通过一个YAML文件，用户可以定义服务、网络、数据卷等资源，然后通过一条命令启动所有服务。

想象一个典型的Web应用：前端Nginx、后端Python/Node.js、数据库MySQL、缓存Redis。如果分别启动4个容器，需要4条docker run命令，而且还要处理它们之间的依赖关系。Compose就是来解决这个问题的。

在前面的课程中，我们学习了如何使用 docker run 命令启动单个容器。但在实际开发中，一个完整的应用往往由多个容器组成。每次启动应用时，都要逐个输入冗长的命令，不仅效率低下，还容易出错。Docker Compose的出现，就是为了让多容器应用的管理变得简单高效。

生活类比：Compose就像餐厅的套餐菜单

想象你去餐厅吃饭，有两种点餐方式：

方式1：单点（类似docker run）

- "我要一份宫保鸡丁"

- "我要一碗米饭"

- "我要一份青菜"

- "我要一碗汤"

每道菜都要单独点，容易遗漏，而且服务员要记录多次。

方式2：套餐（类似Docker Compose）

- "我要A套餐"

套餐已经定义好了包含哪些菜品，一句话就能点完，不会遗漏，服务员也更容易记录。Docker Compose就是为你的应用定义一份"套餐菜单"，一条命令就能启动所有需要的容器。

1一键启动

通过一条命令启动所有服务，无需手动操作

2环境隔离

每个项目有独立的环境，互不干扰

3配置复用

配置文件版本化，方便团队协作

4资源管理

统一管理网络、数据卷等资源

Compose的核心价值

Docker Compose不仅仅是简化命令输入，它的核心价值在于：

1. 声明式配置：用配置文件描述"想要什么"，而不是"怎么做"

2. 环境一致性：开发、测试、生产环境使用同一份配置，减少"在我机器上能跑"的问题

3. 版本控制：配置文件可以提交到Git，团队成员共享同一份环境定义

4. 快速复现：新成员加入团队，只需一条命令就能搭建完整的开发环境

1.2

为什么需要Docker Compose？

多容器管理的痛点
在实际开发和生产环境中，一个完整的应用通常由多个容器组成。手动管理这些容器会带来诸多问题：命令冗长、容易出错、难以复现、协作困难。

假设你要手动启动一个包含Web服务器、数据库和缓存的应用，你需要执行以下步骤：

手动启动多个容器的繁琐过程 Bash 
# 第一步：创建网络（让容器可以互相通信）
$ docker network create my-app-network

# 第二步：启动数据库容器
$ docker run -d \
  --name my-db \
  -e MYSQL_ROOT_PASSWORD=secret \
  -v db-data:/var/lib/mysql \
  --network my-app-network \
  mysql:8.0

# 第三步：启动Redis缓存容器
$ docker run -d \
  --name my-redis \
  --network my-app-network \
  redis:alpine

# 第四步：启动后端应用容器
$ docker run -d \
  --name my-api \
  -e DB_HOST=my-db \
  -e REDIS_HOST=my-redis \
  --network my-app-network \
  my-api-image

# 第五步：启动前端Nginx容器
$ docker run -d \
  --name my-web \
  -p 80:80 \
  --network my-app-network \
  nginx:alpine
思考

上面的命令有什么问题？首先，命令非常长，容易打错；其次，启动顺序很重要，如果先启动应用再启动数据库，应用会连接失败；最后，如果要停止或重启，需要逐个操作。Docker Compose正是为了解决这些问题而诞生的。

手动管理的痛点详解

痛点1：命令冗长易错

每个容器的启动命令可能有十几个参数，一个字母打错就会导致启动失败。而且这些命令很难记忆，通常需要保存在文档中，每次启动都要复制粘贴。

痛点2：启动顺序依赖

数据库必须先启动，应用才能连接；如果顺序错了，应用会报错退出。手动管理时，你需要记住哪个服务依赖哪个服务，这在服务数量增多时变得非常困难。

痛点3：环境不一致

开发人员A在自己电脑上启动了5个容器，开发人员B可能只启动了4个，导致环境不一致。这种差异会引发"在我机器上能跑"的经典问题。

痛点4：难以维护

当需要修改某个容器的配置（如端口、环境变量）时，需要重新输入整条命令。如果有10个容器，就要修改10次，容易遗漏。

手动管理 vs Compose管理对比

停止服务：手动需要逐个执行 docker stop 和 docker rm；Compose只需 docker compose down

查看日志：手动需要 docker logs 容器名 逐个查看；Compose只需 docker compose logs 查看所有服务

团队协作：手动需要口头或文档说明启动步骤；Compose只需分享一个 docker-compose.yml 文件

环境复现：手动在新机器上容易遗漏步骤；Compose在任何装有Docker的机器上都能一键复现

手动编排 vs Compose编排实战对比

实战场景
假设我们要部署一个电商网站，包含：Nginx（前端）、Node.js（后端API）、MySQL（数据库）、Redis（缓存）。下面分别展示手动编排和Compose编排的完整过程。
手动编排：启动命令（共需执行10+条命令） Bash 
# 1. 创建专用网络
$ docker network create ecommerce-network

# 2. 启动MySQL数据库（必须先启动，因为其他服务依赖它）
$ docker run -d \
  --name ecommerce-mysql \
  -e MYSQL_ROOT_PASSWORD=admin123 \
  -e MYSQL_DATABASE=shop \
  -v mysql-data:/var/lib/mysql \
  --network ecommerce-network \
  mysql:8.0

# 3. 启动Redis缓存
$ docker run -d \
  --name ecommerce-redis \
  --network ecommerce-network \
  redis:alpine

# 4. 启动Node.js后端（依赖MySQL和Redis）
$ docker run -d \
  --name ecommerce-api \
  -e DB_HOST=ecommerce-mysql \
  -e DB_PASSWORD=admin123 \
  -e REDIS_HOST=ecommerce-redis \
  -p 3000:3000 \
  --network ecommerce-network \
  my-node-app:latest

# 5. 启动Nginx前端（依赖后端API）
$ docker run -d \
  --name ecommerce-web \
  -p 80:80 \
  -v ./nginx.conf:/etc/nginx/nginx.conf:ro \
  --network ecommerce-network \
  nginx:alpine

# 6. 查看所有容器状态（需要逐个确认）
$ docker ps

# 7. 如果某个容器启动失败，查看日志
$ docker logs ecommerce-api
$ docker logs ecommerce-mysql

# 8. 停止所有服务（需要逐个停止）
$ docker stop ecommerce-web ecommerce-api ecommerce-redis ecommerce-mysql
$ docker rm ecommerce-web ecommerce-api ecommerce-redis ecommerce-mysql

# 9. 删除网络
$ docker network rm ecommerce-network
步骤1：创建项目目录 Bash 
# 创建电商项目目录
$ mkdir ecommerce-project
$ cd ecommerce-project
步骤2：创建docker-compose.yml文件 Bash 
# 使用文本编辑器创建配置文件（以下三种方式任选其一）

# 方式1：使用vim编辑器
$ vim docker-compose.yml

# 方式2：使用nano编辑器（更适合新手）
$ nano docker-compose.yml

# 方式3：使用cat命令直接写入（适合复制粘贴）
$ cat > docker-compose.yml << 'EOF'
# 在这里粘贴下面的配置内容
# 输入完成后按 Ctrl+D 保存
EOF
步骤3：编写docker-compose.yml配置内容 YAML 
version: "3.8"

services:
  web:
    image: nginx:alpine
    ports:
      - "80:80"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
    depends_on:
      - api
    networks:
      - ecommerce-net

  api:
    image: my-node-app:latest
    ports:
      - "3000:3000"
    environment:
      - DB_HOST=mysql
      - DB_PASSWORD=admin123
      - REDIS_HOST=redis
    depends_on:
      - mysql
      - redis
    networks:
      - ecommerce-net

  mysql:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: admin123
      MYSQL_DATABASE: shop
    volumes:
      - mysql-data:/var/lib/mysql
    networks:
      - ecommerce-net

  redis:
    image: redis:alpine
    networks:
      - ecommerce-net

networks:
  ecommerce-net:

volumes:
  mysql-data:
Compose操作：一条命令完成所有操作 Bash 
# 启动所有服务（自动创建网络、按依赖顺序启动）
$ docker compose up -d

# 查看所有服务状态（一目了然）
$ docker compose ps

# 查看所有服务日志（统一输出）
$ docker compose logs -f

# 停止并删除所有服务（一键清理）
$ docker compose down
对比总结

命令数量：手动需要10+条命令，Compose只需1条

出错概率：手动容易打错容器名、漏掉参数；Compose配置一次，多次复用

启动顺序：手动需要人工控制顺序；Compose自动按depends_on启动

团队协作：手动需要写文档说明；Compose直接分享配置文件

可维护性：手动修改需要重新输入命令；Compose修改文件后重新up即可

生活类比：手动编排 vs Compose编排

手动编排就像组织一场音乐会，你需要：

- 逐个打电话通知每位乐手到场时间

- 记住谁先到、谁后到（小提琴手必须在指挥之前到）

- 每次演出都要重复这个过程

- 如果有新乐手加入，需要重新协调所有人

Compose编排就像有一份演出计划表：

- 计划表上写明了所有乐手的到场时间和顺序

- 只需要发布一次计划表，所有人按表行动

- 每次演出都按同一份计划表执行，不会出错

- 修改计划表后，所有人自动按新计划行动

Compose安装与基本用法

安装说明
Docker Desktop for Windows/Mac已内置Compose。如果使用Linux或Docker Engine，需要单独安装Compose。
Linux系统安装Compose（详细步骤） Bash 
# 步骤1：下载Docker Compose二进制文件
# 从GitHub官方仓库下载最新版本（WSL2中curl通常已预装）
$ sudo curl -L \
  "https://github.com/docker/compose/releases/latest/download/docker-compose-$(uname -s)-$(uname -m)" \
  -o /usr/local/bin/docker-compose

# 如果curl不可用，可以使用wget替代：
# $ sudo wget -O /usr/local/bin/docker-compose \
#   "https://github.com/docker/compose/releases/latest/download/docker-compose-$(uname -s)-$(uname -m)"

# 步骤2：赋予可执行权限
# 让系统可以运行这个程序
$ sudo chmod +x /usr/local/bin/docker-compose

# 步骤3：创建软链接（可选，但推荐）
# 这样可以在任何目录直接使用docker-compose命令
$ sudo ln -s /usr/local/bin/docker-compose /usr/bin/docker-compose

# 步骤4：验证安装是否成功
$ docker-compose --version
# 预期输出类似：docker-compose version 2.20.0, build xxx
验证安装（所有系统通用） Bash 
# 检查Docker Compose版本（新版命令）
$ docker compose version

# 预期输出：
# Docker Compose version v2.20.0

# 旧版本命令（docker-compose）也可用
$ docker-compose version
基本命令速查表 Bash 
# 启动所有服务（后台运行，推荐）
# -d 表示 detached（分离模式），即在后台运行
$ docker compose up -d

# 启动并实时查看日志（前台运行，调试用）
$ docker compose up

# 停止所有服务（保留容器和数据）
$ docker compose stop

# 停止并删除所有容器（数据卷保留）
$ docker compose down

# 查看服务状态
$ docker compose ps

# 查看某个服务的日志
$ docker compose logs [服务名称]

# 实时跟踪日志（类似 tail -f）
$ docker compose logs -f [服务名称]

# 进入容器内部调试（类似docker exec）
$ docker compose exec [服务名称] bash
注意

新版本使用docker compose（空格，作为Docker的子命令），旧版本使用docker-compose（连字符，独立命令）。两者功能相同，建议优先使用新版命令。

Compose版本演进历史

V1版本（docker-compose）：

- 独立的Python程序，需要单独安装

- 命令格式：docker-compose up（连字符）

- 配置文件版本：1.x、2.x、3.x

V2版本（docker compose）：

- 集成到Docker CLI中，作为子命令

- 命令格式：docker compose up（空格）

- 使用Go语言重写，性能更好

- 向后兼容旧版配置文件

如何选择：

- Docker Desktop用户：已内置V2版本，直接使用docker compose

- Linux用户：Docker Engine 20.10+已包含V2，优先使用docker compose

- 旧系统：如果只有V1版本，使用docker-compose也可以

2.1

YAML是什么？从零开始学

YAML定义
YAML（YAML Ain't Markup Language）是一种人类可读的数据序列化格式。它的设计目标是让人们更容易读写配置文件。YAML使用缩进来表示层级关系，类似于Python的语法风格。

可以把YAML理解为一种"有格式的文本"，用来描述数据的结构。就像填写表格一样，YAML通过简单的规则让计算机理解你想要表达的信息。Docker Compose使用YAML格式来定义多容器应用的配置。

YAML与JSON的关系

如果你听说过JSON，可以把YAML理解为JSON的"简化版"。YAML更易于人类阅读和编写，而JSON更易于机器解析。两者可以互相转换。

YAML vs JSON 同一份数据的两种写法 YAML / JSON 
# YAML 写法（人类友好，缩进表示层级）
person:
  name: 张三
  age: 25
  is_student: false
  hobbies:
    - 编程
    - 阅读
    - 游泳
  address:
    city: 北京
    zipcode: "100000"

# 对应的 JSON 写法（机器友好，大括号表示层级）
{
  "person": {
    "name": "张三",
    "age": 25,
    "is_student": false,
    "hobbies": ["编程", "阅读", "游泳"],
    "address": {
      "city": "北京",
      "zipcode": "100000"
    }
  }
}
YAML的其他使用场景

YAML不仅用于Docker Compose，还广泛应用于各种配置场景：

- CI/CD流水线：GitHub Actions、GitLab CI、Travis CI的配置文件都是YAML格式

- Kubernetes：K8s的部署文件（Deployment、Service等）使用YAML定义

- Ansible：自动化运维工具Ansible的Playbook使用YAML编写

- Helm：Kubernetes包管理工具Helm的Chart配置文件

- Spring Boot：Java框架Spring Boot支持application.yml作为配置文件

- 静态网站生成器：Jekyll、Hugo等工具的文章 front matter 使用YAML

YAML vs JSON 对比总结 对比表

─────────────────────────────────────────────────────────────

# 结论：
# - 如果是人类编写和阅读的配置文件 → 优先选择 YAML
# - 如果是程序之间的数据交换 → 优先选择 JSON
# - Docker Compose 选择 YAML 正是因为配置文件需要人类频繁编辑

YAML在Compose中的应用

Compose文件格式
docker-compose.yml是一个YAML格式的配置文件，用于定义服务、服务网络、数据卷等资源。文件名固定为docker-compose.yml或docker-compose.yaml。

在Docker Compose中，YAML文件描述了"要运行哪些容器"、"每个容器用什么镜像"、"容器之间如何通信"、"数据存储在哪里"等关键信息。理解YAML语法是编写Compose文件的基础。

# 创建项目目录
$ mkdir my-compose-project
$ cd my-compose-project

# 创建docker-compose.yml配置文件
$ vim docker-compose.yml
# 输入以下配置内容
步骤2：docker-compose.yml基本结构详解 YAML 
# 版本声明（可选，但建议指定）
# 不同版本支持不同的功能特性
version: "3.8"

# 服务定义（核心部分）
# services是一个字典，每个键对应一个服务

  # 服务名称：web（自定义，用于标识服务）
  web:

    image: nginx:alpine
    # 端口映射：将容器的80端口映射到宿主机的8080端口
    ports:
      - "8080:80"
    # 连接的网络
    networks:
      - frontend

  # 服务名称：db（数据库服务）
  db:
    image: mysql:8.0
    # 环境变量：设置MySQL的root密码
    environment:
      MYSQL_ROOT_PASSWORD: secret
    # 数据卷挂载：持久化存储数据库数据
    volumes:
      - db-data:/var/lib/mysql
    networks:
      - backend

# 网络定义
# 自定义网络，用于容器间通信
networks:
  frontend:
  backend:

# 数据卷定义
# 命名数据卷，用于持久化存储
volumes:
  db-data:
核心概念

- services：定义要运行的容器，每个服务对应一个容器。这是Compose文件的核心部分。

- networks：定义容器间通信的网络。同一网络中的容器可以通过服务名互相访问。

- volumes：定义持久化存储。即使容器被删除，数据卷中的数据仍然存在。

生活类比：Compose文件就像建筑施工图

想象你要建造一栋房子，需要一份完整的施工图：

services（服务） = 房间

- 客厅（web服务）：接待访客

- 厨房（api服务）：处理业务逻辑

- 储藏室（db服务）：存储数据

每个房间有自己的功能，但都是房子的一部分。

networks（网络） = 走廊和门

- 走廊连接各个房间，让人可以在房间之间移动

- 有些房间之间有门（同一网络），可以直接互通

- 有些房间之间没有门（不同网络），需要绕路

volumes（数据卷） = 保险柜

- 即使房子拆了重建（容器删除重建），保险柜里的贵重物品（数据）依然保留

- 保险柜可以从旧房子搬到新房子（数据卷可以挂载到新容器）

为什么使用YAML格式？

Docker Compose选择YAML格式而不是JSON或XML，主要原因是：

1. 人类可读性强：没有大量的括号和引号，结构清晰

2. 支持注释：可以在配置文件中添加说明，方便团队理解

3. 简洁：相同的配置，YAML比JSON少30%的字符

4. 广泛应用：Kubernetes、CI/CD工具都使用YAML，学习一次多处受益

服务定义详解

服务定义原理
服务（service）定义了一个容器运行时配置。当执行docker compose up时，Compose会为每个服务启动一个容器。服务名称就是容器名称的前缀。

服务定义中包含：使用哪个镜像、暴露哪些端口、挂载哪些数据卷、设置哪些环境变量等。这些配置与docker run命令的参数一一对应。可以把服务定义理解为"给docker run命令写一份配置文件"。

服务定义完整示例（逐行解释） YAML 
services:
  web:
    # 镜像：指定容器基于哪个镜像创建
    # 格式：仓库名/镜像名:标签（标签省略时默认latest）
    image: nginx:alpine

    # 容器名称：自定义容器的名字（可选）
    # 如果不指定，Compose会自动生成：目录名_服务名_序号
    container_name: my-web-server

    # 端口映射：将容器端口暴露到宿主机
    # 格式："宿主机端口:容器端口"
    # 这样外部用户可以通过宿主机IP+端口访问容器服务
    ports:
      - "8080:80"    # 访问宿主机的8080端口，会转发到容器的80端口
      - "8443:443"   # HTTPS端口映射

    # 环境变量：向容器内部传递配置信息
    # 容器内的应用程序可以读取这些变量来改变行为
    environment:
      - NODE_ENV=production    # 告诉Node.js运行在生成环境
      - DB_HOST=db             # 告诉应用数据库地址是"db"（服务名）

    # 数据卷挂载：将宿主机目录或数据卷映射到容器内部
    # 用于持久化数据或共享代码
    volumes:
      # 宿主机目录挂载（开发时常用，代码修改立即生效）
      - ./html:/usr/share/nginx/html
      # 命名数据卷挂载（生产环境推荐，数据持久化）
      - nginx-logs:/var/log/nginx

    # 依赖服务：控制启动顺序
    # Compose会先启动api和db，再启动web
    depends_on:
      - api
      - db

    # 网络连接：指定服务加入哪些网络
    # 同一网络中的服务可以互相访问
    networks:
      - frontend    # 连接前端网络（面向用户）
      - backend     # 连接后端网络（内部通信）

    # 重启策略：容器退出时如何处理
    # unless-stopped：除非手动停止，否则总是重启
    # always：总是重启
    # on-failure：仅在非零退出码时重启
    restart: unless-stopped

  api:
    image: python:3.11-slim
    # 覆盖默认启动命令
    # 容器启动时执行这个命令，而不是镜像的默认命令
    command: python app.py
    # 工作目录：容器内命令执行的默认目录
    working_dir: /app
    volumes:
      - ./app:/app
    depends_on:
      - db
      - redis
    networks:
      - backend

服务配置常用参数速查

- image：指定镜像，如 nginx:alpine

- build：从Dockerfile构建，如 build: ./app

- ports：端口映射，如 - "8080:80"

- volumes：数据卷挂载，如 - ./data:/app/data

- environment：环境变量，如 KEY: value

- depends_on：依赖服务，控制启动顺序

- networks：连接的网络

- restart：重启策略，如 unless-stopped

- command：覆盖默认启动命令

- working_dir：设置工作目录

服务定义与docker run命令的对应关系

Compose的服务定义本质上是docker run命令的配置化。下面是对应关系：

image: nginx:alpine ↔ docker run nginx:alpine

ports: - "8080:80" ↔ docker run -p 8080:80

volumes: - ./data:/app ↔ docker run -v ./data:/app

environment: - KEY=value ↔ docker run -e KEY=value

networks: - mynet ↔ docker run --network mynet

restart: unless-stopped ↔ docker run --restart unless-stopped

可以看到，Compose只是把命令行参数转换成了配置文件格式，更易于管理和维护。

环境变量与配置管理

环境变量的作用
环境变量是向容器内部传递配置信息的一种方式。通过环境变量，可以在不修改镜像的情况下改变容器的行为，例如数据库连接地址、密码、调试模式等。

在实际项目中，开发环境、测试环境和生产环境的配置往往不同。使用环境变量可以将配置与代码分离，实现"一份代码，多处部署"。Docker Compose提供了多种设置环境变量的方式。

环境变量的三种设置方式 YAML 
services:
  web:
    image: myapp:latest
    
    # 方式1：直接在docker-compose.yml中定义（适合简单场景）
    environment:
      - NODE_ENV=production
      - DB_HOST=db
      - DB_PORT=3306

  api:
    image: myapi:latest
    
    # 方式2：键值对形式（更简洁）
    environment:
      API_KEY: "123456789"
      DEBUG_MODE: "false"

  db:
    image: mysql:8.0
    
    # 方式3：从.env文件加载（推荐，适合敏感信息）
    # 在docker-compose.yml同级目录创建.env文件
    # .env文件内容示例：
    # MYSQL_ROOT_PASSWORD=mysecretpassword
    # MYSQL_DATABASE=myapp
    env_file:
      - .env
    # 这样密码不会直接写在配置文件中，更安全
.env文件示例 文本 
# .env 文件
# 这个文件存储环境变量，不会被提交到版本控制（应加入.gitignore）

# 数据库配置
MYSQL_ROOT_PASSWORD=my_secret_password_123
MYSQL_DATABASE=production_db
MYSQL_USER=app_user
MYSQL_PASSWORD=app_password_456

# 应用配置

安全提示

密码等敏感信息不要直接写在docker-compose.yml中，应该使用.env文件，并将.env加入.gitignore，避免泄露到代码仓库。

环境变量的优先级

当多个地方都定义了同一个环境变量时，优先级从高到低为：

1. docker-compose.yml中的environment：直接写在配置文件中的变量

2. .env文件：通过env_file加载的变量

3. 宿主机环境变量：运行docker compose命令时的shell环境变量

例如，如果docker-compose.yml中设置了DB_HOST=mysql，即使.env文件中设置了DB_HOST=postgres，最终容器内的DB_HOST也是mysql。

生活类比：环境变量就像遥控器的频道设置

想象你有一台电视机（容器），它可以接收不同的频道（配置）：

硬编码配置（不使用环境变量）：

- 电视机出厂时就固定了只能看CCTV-1

- 如果想看其他频道，必须拆开电视机重新焊接电路板

环境变量配置：

- 电视机出厂时预留了遥控器接口

- 通过遥控器（环境变量）可以随时切换频道

- 同一台电视机，在家里看新闻（开发环境），在办公室看财经（生产环境）

- 不需要修改电视机本身（不需要重新构建镜像）

.env文件的最佳实践

1. 创建.env.example模板：提交到Git，包含所有需要的变量名，但值为空或示例值

2. .env加入.gitignore：真实的.env文件不提交，避免泄露密码

3. 团队成员各自创建.env：根据.env.example创建自己的.env文件

4. 生产环境使用密钥管理：不要在服务器上直接存储.env，使用AWS Secrets Manager等工具

2.5

构建镜像与Dockerfile

build配置原理
除了使用已有的镜像，Compose还可以从Dockerfile构建镜像。当服务需要定制化配置时，可以指定build上下文，Compose会自动构建镜像。

使用build时，Compose会先执行docker build，然后用构建好的镜像启动容器。这对于需要根据代码变化重新构建的场景非常有用。在开发过程中，当你的代码发生变化时，只需要重新执行docker compose up --build即可重新构建并启动服务。

使用Dockerfile构建（详细配置） YAML 
services:
  web:
    # 完整build配置
    build:
      # context：构建上下文，指定包含Dockerfile的目录
      # Docker会将这个目录下的所有文件发送给Docker守护进程
      context: ./webapp
      
      # dockerfile：指定Dockerfile的文件名（默认就是Dockerfile）
      dockerfile: Dockerfile
      
      # args：构建参数，对应Dockerfile中的ARG指令
      # 可以在构建时传递变量值
      args:
        VERSION: 1.0
        NODE_ENV: production
    
    # 镜像名称：构建完成后给镜像打标签
    # 如果不指定，镜像名会是：目录名_服务名
    image: my-webapp:1.0

  # 简写形式（如果Dockerfile在当前目录且就叫Dockerfile）
  api:
    build: ./api
    ports:
      - "5000:5000"

什么时候使用build？

- 当你需要运行自己编写的代码时（如Python、Node.js应用）

- 当官方镜像不能满足需求，需要安装额外软件包时

- 当需要在镜像中嵌入配置文件或静态资源时

- 开发环境中代码频繁变化，需要频繁重新构建时

image vs build 的选择策略

使用image的场景：

- 标准服务：MySQL、Redis、Nginx等成熟的开源软件

- 不需要定制：官方镜像已经满足需求

- 快速启动：直接拉取镜像比构建快得多

使用build的场景：

- 自研应用：公司内部开发的业务系统

- 需要定制：需要安装特定版本的依赖包

- 开发调试：代码频繁修改，需要快速重新构建

混合使用：

- 数据库、缓存等基础服务使用image

- 业务应用使用build

- 这是最常见的组合方式

构建缓存优化技巧

Docker构建镜像时会使用缓存，合理利用缓存可以大幅提升构建速度：

1. 依赖文件先复制：先COPY requirements.txt，再RUN pip install，最后COPY代码

2. 不变的层放前面：系统包安装放在前面，代码复制放在后面

3. 使用.dockerignore：排除node_modules、.git等不需要的文件

4. 多阶段构建：编译阶段和运行阶段分离，减小最终镜像体积

LEMP栈介绍
LEMP栈是指Linux + Nginx + MySQL + PHP的Web运行环境。这是目前最流行的Web开发环境之一，广泛应用于PHP网站和应用的部署。使用Compose可以一键部署完整的LEMP环境。

      - app-net

  # 作用：通过Web界面管理MySQL数据库

    ports:
      # 通过 http://localhost:8080 访问phpMyAdmin

    environment:
      # 告诉phpMyAdmin MySQL服务的位置

      # 依赖MySQL服务
      - mysql
    networks:
      - app-net

4.1

服务依赖与启动顺序

depends_on原理
depends_on用于声明服务间的依赖关系。Compose会确保被依赖的服务先启动，后依赖的服务后启动。但它不等待依赖服务"就绪"，只等待其"启动"。

depends_on解决的问题是启动顺序，但不会等待数据库完全就绪（接受连接）才启动应用。对于需要等待服务真正就绪的场景，需要配合健康检查（healthcheck）使用。可以把depends_on理解为"告诉Compose哪个服务应该先启动"，而不是"等待服务完全准备好"。

服务依赖示例（启动顺序） YAML 
services:
  web:
    image: nginx:alpine
    depends_on:
      - api
      - db
    # 启动顺序：api和db先启动，web后启动
    # 但web不会等待api和db"就绪"，只是等待它们"启动"

  api:
    image: myapi:latest
    depends_on:
      - db
      - redis
    # 启动顺序：db和redis最先启动，api第二启动

  db:
    image: mysql:8.0
    # db最先启动（没有依赖其他服务）

  redis:
    image: redis:alpine
    # redis第二启动（没有依赖其他服务）
depends_on的局限性

- depends_on只控制启动顺序，不等待服务就绪

- 例如数据库容器启动了，但数据库服务可能还在初始化

- 应用容器可能会因为连接不上数据库而报错退出

- 解决方案：使用健康检查（healthcheck）或重试机制

生活类比：depends_on就像排队等号

想象你去银行办业务，需要先取号，再等叫号：

depends_on的行为（只管排队顺序）：

- 系统确保你在A先生之后取号

- 但不保证A先生已经办完业务

- 你可能取到号了，但A先生还在柜台办理

- 如果你的业务依赖A先生办完（比如A是你的担保人），你还是要等

healthcheck的行为（等待真正就绪）：

- 不仅确保A先生先取号

- 还会等待A先生办完业务、拿到回执

- 确认A先生完全办好了，你才开始办理

- 这样你的业务就不会因为依赖未满足而失败

为什么depends_on不等待服务就绪？

这是Docker Compose的设计哲学：

1. 解耦关注点：启动顺序是编排层的责任，服务就绪是应用层的责任

2. 避免假设：Compose不知道"就绪"的定义（数据库可能需要初始化，Web服务可能需要预热）

3. 灵活性：应用可以自己实现重试逻辑，或使用healthcheck精确控制

4. 快速失败：如果依赖服务真的有问题，应用快速失败比无限等待更好

4.2

健康检查专题讲解

健康检查（Healthcheck）
健康检查是Docker提供的一种机制，用于检测容器内的服务是否正常运行。通过定期执行检查命令，Docker可以判断容器是"健康"、"不健康"还是"正在启动"。

健康检查解决了depends_on的局限性问题。当配置了健康检查后，其他服务可以等待依赖服务真正"健康"后再启动，而不是仅仅等待容器启动。这对于数据库等需要初始化时间的服务非常重要。

健康检查配置详解 YAML 
services:
  db:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: secret
    # 健康检查配置
    healthcheck:
      # test：执行的检查命令
      # 这里使用mysqladmin ping检查MySQL是否响应
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      
      # interval：检查间隔时间（多久检查一次）
      interval: 5s
      
      # timeout：单次检查的超时时间
      # 如果命令在3秒内没有返回，认为检查失败
      timeout: 3s
      
      # retries：失败重试次数
      # 连续失败3次后，认为容器不健康
      retries: 3
      
      # start_period：启动宽限期
      # 容器启动后10秒内，检查失败不计入重试次数
      # 给服务足够的初始化时间
      start_period: 10s

  web:
    image: myapp:latest
    depends_on:
      db:
        # 只有当db健康检查通过后，web才会启动
        condition: service_healthy
    # 这样web服务会等待MySQL完全就绪后才启动
常见服务的健康检查命令 YAML

查看健康状态

使用docker compose ps可以查看容器的健康状态。状态包括：starting（启动中）、healthy（健康）、unhealthy（不健康）。

健康检查的工作原理

健康检查是一个定时执行的命令，Docker根据命令的退出码判断容器健康状态：

- 退出码0：健康（healthy）

- 退出码1：不健康（unhealthy）

- 退出码2：保留，不使用

状态转换流程：

1. 容器启动 → starting状态

2. 在start_period内，检查失败不计入重试次数

3. start_period后，开始正式检查

4. 连续成功1次 → healthy状态

5. 连续失败retries次 → unhealthy状态

生活类比：健康检查就像体检

想象容器是一个员工，健康检查就是定期体检：

interval（检查间隔） = 体检频率

- 每5秒检查一次，就像每5秒量一次体温

- 频率太高浪费资源，频率太低发现问题太晚

timeout（超时时间） = 体检时限

- 如果3秒内没有完成检查，认为检查失败

- 就像体检时如果医生3秒内没有得到回应，认为员工昏迷了

retries（重试次数） = 确诊标准

- 连续3次体检不合格，才认定员工真的生病了

- 避免偶然的网络抖动导致误判

start_period（启动宽限期） = 新员工适应期

- 新员工刚入职10秒内，体检不合格不算数

- 给服务足够的初始化时间

健康检查的最佳实践

1. 检查命令要轻量：不要执行复杂的业务逻辑，只检查服务是否响应

2. 设置合理的超时：timeout应该比正常响应时间长2-3倍

3. 避免过于频繁：interval不要太短，5-10秒是合理的范围

4. 给足启动时间：start_period要覆盖服务的最长初始化时间

5. 记录检查日志：健康检查失败时，应该有日志可查

服务扩展与副本

服务扩展原理
Compose支持水平扩展，即为一个服务启动多个容器副本（replicas）。这可以实现负载均衡和高可用。当访问量增加时，可以通过扩展服务来提高处理能力。

当扩展服务时，Compose会启动多个容器，它们共享同一个网络和数据卷（如果是挂载的宿主机目录，则共享同一份数据）。副本数由--scale参数指定。需要注意的是，如果多个容器绑定同一端口，需要使用负载均衡器（如Nginx）来分发请求。

服务扩展命令 Bash 
# 扩展web服务到3个副本
$ docker compose up -d --scale web=3

# 查看扩展后的服务状态
$ docker compose ps
# 会看到web服务有3个容器在运行

# 同时扩展多个服务
$ docker compose up -d --scale web=3 --scale api=2

# 注意：如果多个容器绑定同一端口，需要使用负载均衡器
# 或者使用ports配置中的随机端口（不推荐用于生产）
负载均衡配置（Nginx + 多副本） YAML 
services:
  # Nginx作为负载均衡器
  nginx:
    image: nginx:alpine
    ports:
      # 只有Nginx暴露端口给用户访问
      - "80:80"
    volumes:
      # 挂载Nginx配置文件
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
    depends_on:
      - web

  # Web应用服务（3个副本）
  web:
    image: myweb:latest
    # 不暴露端口（通过Nginx反向代理访问）
    # 每个副本会自动分配不同的内部端口
    deploy:
      replicas: 3
    # 或者使用命令行：--scale web=3

# nginx.conf配置示例：
# upstream backend {
#     # Compose会自动将web服务名解析到所有副本
#     server web:5000;
# }
# 
# server {
#     listen 80;
#     location / {
#         proxy_pass http://backend;
#     }
# }
扩展的注意事项

- 扩展适用于无状态服务（如Web应用），不适用于有状态服务（如数据库）

- 多个副本共享数据卷时，要考虑并发写入的问题

- 生产环境中建议使用Kubernetes等更强大的编排工具

- 扩展后需要使用负载均衡器分发请求

什么是无状态服务？

无状态服务：每次请求都是独立的，不依赖之前的请求

- 例子：Web服务器、API服务器

- 特点：可以随意增加或减少副本数量

- 扩展方式：水平扩展（增加更多相同的容器）

有状态服务：需要保存状态，请求之间有关联

- 例子：数据库、消息队列

- 特点：不能简单地增加副本（会导致数据不一致）

- 扩展方式：垂直扩展（增加单个容器的资源）或使用集群模式

生活类比：扩展服务就像增加收银台

想象一个超市，顾客排队结账：

无状态服务扩展（收银台）：

- 平时2个收银台够用

- 周末人多，开放6个收银台

- 每个收银台功能完全相同，顾客去哪个都一样

- 这就是Web服务的水平扩展

有状态服务扩展（银行柜台）：

- 你在1号柜台办理业务，办到一半

- 不能随便切换到2号柜台继续办理

- 因为1号柜台的工作人员记录了你的办理进度

- 这就是数据库等有状态服务，不能简单扩展

网络与数据卷管理

网络隔离原理
Compose会自动为项目创建一个默认网络，所有服务通过服务名相互访问。通过自定义网络，可以实现服务间的隔离，提高安全性。

在同一网络中的容器可以通过容器名互相访问（DNS解析）。不同网络的容器不能直接通信，必须通过网关或端口映射。这种网络隔离机制可以保护敏感服务（如数据库）不被外部直接访问。

自定义网络配置（网络隔离示例） YAML 
services:
  # 前端服务（面向用户）
  frontend:
    image: nginx:alpine
    ports:
      - "80:80"
    networks:
      # 只连接web网络
      - web-net
    # frontend只能访问web-net中的服务

  # 后端API服务
  backend:
    image: myapi:latest
    networks:
      # 同时连接两个网络
      - web-net    # 接收前端请求
      - db-net     # 访问数据库
    # backend是桥梁，连接前端和数据库

  # 数据库服务（敏感服务）
  database:
    image: mysql:8.0
    networks:
      # 只连接db网络
      - db-net
    # database只能被db-net中的服务访问
    # 外部无法直接访问数据库，提高了安全性

networks:
  web-net:
    driver: bridge
    # 默认bridge网络，允许外部访问
  
  db-net:
    driver: bridge
    internal: true
    # internal: true 表示完全隔离
    # 不允许外部访问，只允许同一网络内的容器通信
数据卷配置详解 YAML 
services:
  db:
    image: mysql:8.0
    volumes:
      # 类型1：命名数据卷（推荐用于生产环境）
      # 由Docker管理，存储在宿主机特定位置
      # 优点：易于备份、迁移，可以跨容器共享
      - mysql-data:/var/lib/mysql
      
      # 类型2：宿主机目录挂载（推荐用于开发环境）
      # 将宿主机的./data目录映射到容器的/var/lib/mysql
      # 优点：可以直接在宿主机查看和修改文件
      - ./data:/var/lib/mysql
      
      # 类型3：只读挂载（适合配置文件）
      # ro = read only，容器只能读取不能修改
      # 防止容器意外修改配置
      - ./config:/etc/mysql/conf.d:ro
      
      # 类型4：临时卷（容器删除后数据丢失）
      # 适合临时文件、缓存等不需要持久化的数据
      - /tmp

volumes:
  mysql-data:
    # 可以指定驱动和选项
    driver: local
    # driver_opts:
    #   type: nfs
    #   o: addr=192.168.1.100,rw
    #   device: ":/path/to/dir"

- 命名数据卷：Docker管理，适合生产环境，易于备份

- 只读挂载：保护配置文件不被修改

生活类比：网络隔离就像公司的门禁系统

想象一栋办公楼，有不同的安全区域：

web-net（前台网络） = 公共区域

- 访客可以进入前台（frontend服务）

- 前台可以联系办公区（backend服务）

- 但访客不能直接进入办公区

db-net（后台网络） = 机密区域

- 只有持有特殊门卡的员工（backend服务）才能进入

- 数据库（database服务）在机密区域内

- 访客和前台人员都无法进入

backend服务 = 持有两张门卡的员工

- 可以在前台接待访客（连接web-net）

- 也可以进入机密区域查询数据（连接db-net）

- 起到桥梁作用，但不会让访客直接接触机密信息

网络隔离的实际应用场景

场景1：多租户系统

- 租户A的服务在network-a中

- 租户B的服务在network-b中

- 两个租户的数据完全隔离，互不干扰

场景2：开发测试环境隔离

- 开发环境的服务在dev-network中

- 测试环境的服务在test-network中

- 避免开发环境的错误影响测试环境

场景3：微服务安全分层

- 公开API在public-network中

- 内部服务在internal-network中

- 数据库在db-network中

- 层层隔离，提高安全性

- 书架在你家（宿主机），位置固定

- 购物时临时存放商品（临时数据）

- 结账后（容器删除）购物车就清空了

常见问题排查指南

故障排查思路
在使用Docker Compose时，遇到问题不要慌张。按照"查看日志→检查配置→验证网络→检查端口"的顺序排查，大多数问题都能快速定位。
问题1：服务启动后立即退出

现象：执行docker compose up -d后，容器状态显示Exited

原因：通常是应用启动失败（如配置错误、依赖服务未就绪）

解决：

1. 查看日志：docker compose logs [服务名]

2. 检查环境变量是否正确设置

3. 确认依赖服务是否已启动

4. 使用docker compose run --rm [服务名] sh手动调试

问题2：端口冲突

现象：报错"bind: address already in use"

原因：宿主机上该端口已被其他程序占用

解决：

1. 查看端口占用：sudo lsof -i :80 或 sudo ss -tlnp | grep :80

2. 停止占用端口的程序，或修改docker-compose.yml中的端口映射

3. 例如将"80:80"改为"8080:80"

问题3：容器间无法通信

现象：应用无法连接到数据库或其他服务

原因：可能是网络配置错误、服务名拼写错误、或者服务未加入同一网络

解决：

1. 确认所有服务都加入了同一个网络

2. 检查连接地址是否使用了正确的服务名（不是localhost或127.0.0.1）

3. 使用docker compose exec [服务名] ping -c 3 [目标服务名]测试网络连通性（-c 3表示发送3个包后停止）

4. 检查防火墙设置是否阻止了容器间通信

问题4：数据卷挂载失败

现象：容器启动后，挂载的目录为空或权限不足

原因：可能是宿主机目录不存在、路径错误、或SELinux限制

解决：

1. 确认宿主机目录存在：ls -la ./data

2. 检查路径是否正确（相对路径从docker-compose.yml所在目录开始）

3. 在WSL2中，确保文件在WSL文件系统内（/home/username/project），而非Windows挂载路径（/mnt/c/）

4. 如果是权限问题，添加 :Z 或 :z 标签（SELinux环境）

问题5：镜像拉取失败

现象：报错 "pull access denied" 或 "manifest unknown"

原因：镜像名称错误、镜像不存在、或需要登录Docker Hub

解决：

1. 检查镜像名称拼写是否正确

2. 确认镜像标签存在：docker pull nginx:alpine 测试

3. 如果是私有镜像，先登录：docker login

4. 配置国内镜像加速器（如阿里云、DaoCloud）加速拉取

问题6：环境变量未生效

现象：容器内读取不到设置的环境变量

原因：可能是.env文件位置错误、格式问题、或被覆盖

解决：

1. 确认.env文件与docker-compose.yml在同一目录

2. 检查.env文件格式（KEY=VALUE，不能有空格）

3. 在容器内验证：docker compose exec [服务名] env | grep KEY

4. 检查docker-compose.yml中是否同时设置了environment覆盖了.env

最佳实践与注意事项

生产环境建议
Docker Compose非常适合开发环境、测试环境和小型生产环境。对于大型生产环境，建议使用Kubernetes等更强大的编排工具。但掌握Compose是学习容器编排的基础。
最佳实践清单

1. 使用.env文件管理敏感信息：密码、密钥等不要硬编码在docker-compose.yml中

2. 指定镜像版本：使用nginx:1.24而不是nginx:latest，避免版本升级导致意外问题

3. 配置健康检查：为关键服务配置healthcheck，确保服务真正就绪

4. 使用命名数据卷：便于备份和管理，避免数据丢失

5. 设置重启策略：使用restart: unless-stopped，确保服务意外退出后自动重启

6. 限制资源使用：为服务设置内存和CPU限制，防止单个服务耗尽系统资源

生产环境配置示例 YAML 
version: "3.8"

services:
  web:
    image: nginx:1.24-alpine  # 指定具体版本
    ports:
      - "80:80"
    volumes:
      - ./html:/usr/share/nginx/html:ro
    networks:
      - app-net
    # 资源限制
    deploy:
      resources:
        limits:
          cpus: '0.5'        # 最多使用50% CPU
          memory: 512M       # 最多使用512MB内存
        reservations:
          cpus: '0.25'       # 预留25% CPU
          memory: 256M       # 预留256MB内存
    # 健康检查
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost"]
      interval: 30s
      timeout: 10s
      retries: 3
    # 重启策略
    restart: unless-stopped
    # 日志限制
    logging:
      driver: "json-file"
      options:
        max-size: "10m"
        max-file: "3"

networks:
  app-net:
    driver: bridge

volumes:
  app-data:
    driver: local
学习建议

掌握Docker Compose后，建议继续学习Docker Swarm或Kubernetes，了解更强大的容器编排能力。Compose是入门容器编排的最佳起点，它的概念和语法与Kubernetes有很多相似之处。

Compose vs Kubernetes 对比

Docker Compose：

- 适用场景：单机部署、开发测试环境、小型生产环境

- 学习曲线：平缓，几小时即可上手

- 配置复杂度：简单，一个YAML文件

- 功能：基础的多容器编排

Kubernetes：

- 适用场景：大规模集群、高可用生产环境、跨主机部署

- 学习曲线：陡峭，需要数周甚至数月

- 配置复杂度：复杂，多个YAML文件（Deployment、Service、Ingress等）

- 功能：自动扩缩容、自愈、滚动更新、服务发现、负载均衡等

学习路径建议：

1. 先掌握Docker基础（镜像、容器、网络、数据卷）

2. 学习Docker Compose（多容器编排）

3. 了解Docker Swarm（Docker原生集群方案，可选）

4. 深入学习Kubernetes（生产级容器编排）

- 使用Docker Compose完全够用

- 配合Docker Swarm可以实现简单的高可用

- 成本低，维护简单

中型项目（3-10台服务器）：

- 可以考虑Kubernetes，但要评估学习成本

大型项目（10+台服务器）：

- 强烈建议使用Kubernetes

- 自动化运维能力是必需的

- 投入学习成本是值得的

清理命令对比

docker compose down - 只删除当前项目的容器和网络，保留数据卷和镜像

docker compose down -v - 删除容器、网络和数据卷，保留镜像

docker compose down -v --rmi all - 删除所有资源包括镜像

docker container prune - 删除所有已停止的容器

docker system prune - 删除所有未使用的容器、网络、镜像

docker system prune --volumes - 删除所有未使用的资源包括数据卷

清理的最佳实践

1. 实验后立即清理：完成实验后立即执行 docker compose down -v

2. 定期检查资源：每周执行 docker system df 查看磁盘占用

3. 保留重要数据：清理前确认数据卷中没有重要数据

4. 分步清理：先清理容器，再清理网络，最后清理数据卷和镜像

5. 使用标签：为实验容器添加标签，方便批量清理

生产环境注意事项

在生产环境中，永远不要使用 docker system prune -a -f --volumes。这会删除所有未运行的容器和数据，可能导致数据丢失。生产环境应该有明确的清理策略和备份机制。
