数据卷基础概念

理解容器数据存储的核心问题

Part 1.1

为什么容器数据会丢失？

容器文件系统特性
Docker容器的文件系统具有临时性和分层结构。当容器被删除时，容器内的所有数据都会丢失。
模拟数据丢失场景（Ubuntu容器） Bash 
# 启动一个Ubuntu容器，在容器中创建重要数据
$ docker run -it --name test-container ubuntu:22.04 bash

# 在容器中创建测试文件
root@abc123def456:/# mkdir -p /app/data
root@abc123def456:/# echo "这是重要数据" > /app/data/info.txt
root@abc123def456:/# ls -la /app/data/

# 退出容器并删除
root@abc123def456:/# exit
$ docker rm test-container

# 重新创建容器尝试访问数据
$ docker run -it --name new-container ubuntu:22.04 bash
root@xyz789abc012:/# cat /app/data/info.txt
cat: /app/data/info.txt: No such file or directory
# 数据已经丢失！
核心问题

✓ 容器删除后数据无法恢复

✓ 容器层数据读写性能较差（多层叠加）

✓ 多个容器无法共享同一份数据

Part 1.2

什么是Docker数据卷？

数据卷（Volume）
Docker数据卷是一个可供容器使用的特殊目录，它绕过容器文件系统，直接挂载到宿主机的文件系统中。数据卷的生命周期独立于容器，可以跨多个容器共享和重用。
数据卷工作原理 图示 
宿主机文件系统                        Ubuntu容器
┌────────────────────┐            ┌────────────────────┐
│ /var/lib/docker/   │            │ /app/data          │
│ volumes/my-data/   │◄──────────►│ (挂载点)           │
│ _data/             │   挂载     │                    │
│  - info.txt        │            │  - info.txt        │
│  - logs/           │            │  - logs/           │
└────────────────────┘            └────────────────────┘
     数据持久化保存                    容器内访问

1独立生命周期

删除容器不影响数据卷

2高性能读写

绕过容器层，直接与宿主机交互

3跨容器共享

可同时挂载给多个容器

4双向同步

修改立即生效

Part 1.3

数据卷的存储位置

Linux系统数据卷默认位置

数据卷默认存储在 /var/lib/docker/volumes/ 目录下（WSL环境路径可能不同）

查看Ubuntu容器的数据卷信息 Bash 
# 创建一个带数据卷的Ubuntu容器
$ docker run -d --name ubuntu-data -v my-data:/data ubuntu:22.04 sleep 3600

# 查看数据卷详情
$ docker volume inspect my-data
[
    {
        "CreatedAt": "2024-01-15T10:30:00Z",
        "Driver": "local",
        "Mountpoint": "/var/lib/docker/volumes/my-data/_data",
        "Name": "my-data",
        "Scope": "local"
    }
]

# 【推荐】使用docker命令查看数据卷实际路径（适用于所有环境）
$ docker volume inspect my-data --format '{{ .Mountpoint }}'
/var/lib/docker/volumes/my-data/_data

# 在容器中写入数据
$ docker exec ubuntu-data sh -c 'echo "Hello from container" > /data/test.txt'

# 在宿主机上验证（使用docker cp替代直接访问路径，更通用）
$ docker cp ubuntu-data:/data/test.txt - | cat
Hello from container
WSL环境特别说明

• 路径差异：WSL中Docker Desktop的数据卷可能存储在WSL2虚拟磁盘内，直接访问 /var/lib/docker/volumes/ 可能失败

• 推荐做法：使用 docker volume inspect 查看实际路径，或使用 docker cp 进行文件操作

• 替代方案：使用绑定挂载（Bind Mount）将WSL用户目录挂载到容器，便于直接访问

Windows/Mac系统说明

• Windows：C:\ProgramData\docker\volumes\（Docker Desktop）

• Mac：通过Linux虚拟机存储，路径与Linux类似

• WSL：Docker Desktop集成模式下路径可能不同，建议使用docker命令

Part 1.4

三种存储方式对比

Docker提供了三种不同的数据存储方式：

存储类型说明数据持久性适用场景
数据卷（Volume）Docker管理的特殊目录持久，独立于容器生产数据库、日志
绑定挂载（Bind Mount）宿主机任意目录持久，依赖宿主机配置文件、开发源码
tmpfs挂载内存中的文件系统临时，容器停止即丢失敏感数据、临时缓存
容器存储层容器的可写层随容器删除而丢失临时测试数据
选择建议

• 生产数据库（MySQL）→ 使用数据卷

• 配置文件（应用配置）→ 使用绑定挂载

• 敏感信息（密钥）→ 使用tmpfs挂载

Part 2.1

docker volume create - 创建数据卷

docker volume create命令详解 Bash 
# 语法：docker volume create [OPTIONS] [VOLUME_NAME]

# 1. 创建命名数据卷（最常用的方式）
$ docker volume create my-ubuntu-html

# 2. 创建带标签的数据卷（便于管理）
$ docker volume create \
  --label env=production \
  --label app=webapp \
  app-production-data

# 3. 查看创建的数据卷
$ docker volume ls
DRIVER    VOLUME NAME
local     my-ubuntu-html
local     app-production-data

# 4. 查看数据卷详情
$ docker volume inspect my-ubuntu-html
命名数据卷 vs 匿名数据卷

• 命名数据卷：有明确名称，便于管理和识别

• 匿名数据卷：自动生成随机ID，难以管理

• 推荐：始终使用命名数据卷

Part 2.2

docker volume ls/inspect/rm - 查看和管理

查看数据卷 Bash 
# 列出所有数据卷
$ docker volume ls

# 列出数据卷（过滤）
$ docker volume ls -f name=app

# 查看数据卷详细信息
$ docker volume inspect my-ubuntu-html

# 使用Go模板格式化输出
$ docker volume inspect --format '{{.Mountpoint}}' my-ubuntu-html
/var/lib/docker/volumes/my-ubuntu-html/_data
删除数据卷 Bash 
# 删除指定数据卷（必须是未使用的）
$ docker volume rm my-ubuntu-html

# 清理所有未使用的数据卷（谨慎使用！）
$ docker volume prune
WARNING! This will remove all local volumes not used by at least one container.
Are you sure you want to continue? [y/N] y

# 查看清理结果
$ docker volume ls
删除注意事项

• 无法删除正在被容器使用的数据卷

• 删除容器不会自动删除关联的数据卷

Part 2.3

docker run -v 挂载数据卷详解

docker run -v 是最常用的数据卷挂载方式：

语法：-v volume-name:container-path（推荐） Bash 
# 使用已存在的数据卷
$ docker run -d --name ubuntu-vol-1 \
  -v my-ubuntu-html:/app/data \
  ubuntu:22.04 sleep 3600

# 自动创建数据卷（如果不存在）
$ docker run -d --name ubuntu-vol-2 \
  -v app-data:/app/data \
  ubuntu:22.04 sleep 3600

# 参数说明：
# -v 数据卷名称:容器挂载点
#   数据卷名称：可以是已存在的或自动创建的
#   容器挂载点：容器内的目标路径
语法：-v /host/path:/container/path（绑定挂载） Bash 
# 绑定挂载宿主机目录
$ docker run -d --name ubuntu-bind-1 \
  -v /home/user/app/data:/app/data \
  ubuntu:22.04 sleep 3600

# 自定义应用配置
$ docker run -d --name ubuntu-bind-2 \
  -v /home/user/app/config:/app/config \
  -v /home/user/app/logs:/app/logs \
  ubuntu:22.04 sleep 3600
Part 2.4

docker run --mount 挂载详解

--mount 是更明确的挂载方式，语法更清晰，推荐在生产环境使用：

--mount 挂载数据卷 Bash 
# 基本语法：--mount type=volume,source=卷名,target=容器路径

# 使用数据卷挂载
$ docker run -d --name ubuntu-mount-1 \
  --mount type=volume,source=app-data,target=/app/data \
  ubuntu:22.04 sleep 3600

# 只读挂载（readonly）
$ docker run -d --name ubuntu-mount-2 \
  --mount type=volume,source=app-data,target=/app/data,readonly \
  ubuntu:22.04 sleep 3600

# 同时挂载多个数据卷
$ docker run -d --name ubuntu-mount-3 \
  --mount type=volume,source=app-data,target=/app/data \
  --mount type=volume,source=app-logs,target=/app/logs \
  ubuntu:22.04 sleep 3600
对比：-v 和 --mount 对比 
# -v 语法（简洁）
$ docker run -d --name web -v app-data:/app ubuntu:22.04 sleep 3600

# --mount 语法（明确，推荐）
$ docker run -d --name web --mount type=volume,source=app-data,target=/app ubuntu:22.04 sleep 3600

# 为什么推荐 --mount？
# 1. 语法更清晰，明确指定 type、source、target
# 2. 错误提示更详细
# 3. 支持更多选项（readonly、volume-opt等）
Part 2.5

查看容器挂载信息

查看Ubuntu容器挂载信息 Bash 
# 创建带多个数据卷的Ubuntu容器
$ docker run -d --name ubuntu-multi-vol \
  -v app-data:/app/data \
  -v app-logs:/app/logs \
  ubuntu:22.04 sleep 3600

# 查看容器所有信息
$ docker inspect ubuntu-multi-vol

# 只查看挂载信息（Mounts部分）
$ docker inspect --format '{{json .Mounts}}' ubuntu-multi-vol | jq
[
  {
    "Type": "volume",
    "Name": "app-data",
    "Source": "/var/lib/docker/volumes/app-data/_data",
    "Destination": "/app/data",
    "RW": true
  }
]
字段说明 解释 
Type: volume          # 挂载类型（volume/bind/tmpfs）
Name: app-data      # 数据卷名称
Source: /var/lib/... # 宿主机上的路径（Mountpoint）
Destination: /app/data  # 容器内的挂载点
RW: true              # 读写权限（true=可写，false=只读）
Part 2.6

WSL环境实验指南

WSL用户使用建议

• 推荐使用绑定挂载：将WSL用户目录（如 ~/docker-data）挂载到容器，便于直接访问和管理

• 避免直接访问Docker数据卷路径：WSL中 /var/lib/docker/volumes/ 可能无法直接访问

• 使用docker cp进行文件传输：在宿主机和容器之间复制文件，不依赖具体路径

• 查看实际路径：使用 docker volume inspect 卷名 --format '{{ .Mountpoint }}' 获取真实路径

WSL推荐做法：使用绑定挂载 Bash 
# 在WSL用户目录创建数据目录
$ mkdir -p ~/docker-data/app-data
$ mkdir -p ~/docker-data/app-logs

# 使用绑定挂载启动容器（推荐）
$ docker run -d --name ubuntu-wsl \
  -v ~/docker-data/app-data:/app/data \
  -v ~/docker-data/app-logs:/app/logs \
  ubuntu:22.04 sleep 3600

# 在WSL中直接查看和修改数据
$ ls ~/docker-data/app-data/
$ echo "Test data" > ~/docker-data/app-data/test.txt

# 在容器中验证
$ docker exec ubuntu-wsl cat /app/data/test.txt
Test data
使用docker cp进行文件操作（通用方法） Bash 
# 从容器复制文件到宿主机（WSL适用）
$ docker cp ubuntu-wsl:/app/data/test.txt ~/test.txt

# 从宿主机复制文件到容器（WSL适用）
$ docker cp ~/test.txt ubuntu-wsl:/app/data/

# 复制整个目录
$ docker cp ubuntu-wsl:/app/data ~/backup-data

# 使用 - 作为标准输出/输入
$ docker cp ubuntu-wsl:/app/data/test.txt - | cat
$ echo "New content" | docker cp - ubuntu-wsl:/app/data/newfile.txt

挂载类型详解

深入理解三种挂载方式的特点和使用场景

Part 3.1

绑定挂载（Bind Mounts）详解

绑定挂载
绑定挂载将宿主机上的任意目录或文件挂载到容器中。容器可以完全访问宿主机指定路径的内容。
绑定挂载应用配置文件（Ubuntu容器示例） Bash 
# 创建宿主机上的配置目录
$ mkdir -p ~/app-config ~/app-data ~/app-logs

# 创建自定义应用配置文件
$ cat > ~/app-config/default.conf << 'EOF'
server {
    listen 80;
    server_name localhost;
    root /app/data;
    
    location / {
        try_files $uri $uri/ =404;
    }
    
    location /health {
        return 200 "OK";
        add_header Content-Type text/plain;
    }
}
EOF

# 创建网页文件
$ echo "
Custom Website via Bind Mount
" > ~/app-data/index.html

# 启动Ubuntu使用绑定挂载
$ docker run -d --name ubuntu-bind-demo \
  -p 8080:80 \
  -v ~/app-config:/app/config:ro \
  -v ~/app-data:/app/data:ro \
  -v ~/app-logs:/app/logs \
  ubuntu:22.04 sleep 3600

# 测试
$ curl http://localhost:8080
$ curl http://localhost:8080/health
使用场景

• 开发环境：挂载源代码，修改实时生效

• 配置文件：使用宿主机编辑器修改配置

• 日志收集：宿主机直接访问容器日志

Part 3.2

tmpfs挂载详解

tmpfs挂载
tmpfs挂载将数据存储在宿主机的内存中，而不是磁盘上。读写速度极快，但容器停止后数据会丢失。
tmpfs挂载示例（Ubuntu容器） Bash 
# 创建带tmpfs挂载的Ubuntu容器
$ docker run -d --name ubuntu-tmpfs-demo \
  --tmpfs /app/secrets \
  --tmpfs /app/cache \
  ubuntu:22.04 sleep 3600

# 使用--mount语法指定更多选项
$ docker run -d --name ubuntu-tmpfs-mount \
  --mount type=tmpfs,target=/app/secrets,tmpfs-size=512m,tmpfs-mode=0770 \
  ubuntu:22.04 sleep 3600

# 参数说明：
# tmpfs-size: 限制内存大小（默认无限制）
# tmpfs-mode: 文件权限模式（默认1777）

# 验证tmpfs挂载
$ docker exec ubuntu-tmpfs-demo df -h | grep tmpfs
tmpfs  512M     0  512M   0% /app/secrets
tmpfs  512M     0  512M   0% /app/cache

# 写入数据到tmpfs
$ docker exec ubuntu-tmpfs-demo sh -c 'echo "sensitive data" > /app/secrets/token.txt'
$ docker exec ubuntu-tmpfs-demo cat /app/secrets/token.txt

# 重启容器后数据丢失
$ docker restart ubuntu-tmpfs-demo
$ docker exec ubuntu-tmpfs-demo cat /app/secrets/token.txt
cat: /app/secrets/token.txt: No such file or directory

1高性能

基于内存，读写速度极快

2数据安全

容器停止后自动清空

3内存占用

会占用宿主机内存

Part 4.1

实战：Ubuntu数据持久化完整案例

生产环境应用部署（Ubuntu环境） Bash 
# 步骤1：创建必要的数据卷
$ docker volume create app-web-data
$ docker volume create app-config
$ docker volume create app-logs

# 步骤2：初始化配置文件（通过Ubuntu容器）
$ docker run --rm -v app-config:/config ubuntu:22.04 sh -c 'cat > /config/default.conf << "EOF"
server {
    listen 80;
    server_name localhost;
    root /app/data;
    access_log /app/logs/access.log;

EOF'

# 步骤3：初始化网页内容
$ docker run --rm -v app-web-data:/html ubuntu:22.04 sh -c 'cat > /html/index.html << "EOF"
<!DOCTYPE html><html><head><title>Production Ubuntu App</title></head>
<body><h1>Welcome to Production Ubuntu App</h1></body></html>
EOF'

# 步骤4：启动生产Ubuntu容器
$ docker run -d --name ubuntu-prod \
  -p 80:80 \
  -v app-web-data:/app/data:ro \
  -v app-config:/app/config:ro \
  -v app-logs:/app/logs \
  --restart unless-stopped \
  ubuntu:22.04 sleep 3600

# 步骤5：验证
$ curl http://localhost

Part 4.2

数据卷权限管理

数据卷权限问题是一个常见的坑，需要注意UID/GID匹配：

权限问题排查（Ubuntu容器示例） Bash 
# 问题：应用无法写入日志
$ docker run -d --name ubuntu-perm-check -v app-logs:/app/logs ubuntu:22.04 sleep 3600
$ docker logs ubuntu-perm-check
# 错误：Permission denied

# 排查方法1：查看容器内进程用户
$ docker exec ubuntu-perm-check id
uid=101(ubuntu) gid=1000(ubuntu) groups=1000(ubuntu)

# 排查方法2：查看数据卷宿主机权限（原生Linux环境）
# WSL用户请使用 docker volume inspect 查看实际路径
$ ls -la /var/lib/docker/volumes/app-logs/_data/
drwxr-xr-x 2 root root 999 root 4096 Jan 15 10:30 .

# 解决方案1：修改宿主机权限（原生Linux环境）
# WSL用户建议使用绑定挂载替代数据卷，或直接在容器内修改权限
$ sudo chown -R 101:101 /var/lib/docker/volumes/app-logs/_data

# 解决方案2：使用Z标签（SELinux）
$ docker run -d --name ubuntu-selinux-fix \
  -v app-logs:/app/logs:Z \
  ubuntu:22.04 sleep 3600

# 解决方案3：使用ro只读挂载日志目录
$ docker run -d --name ubuntu-ro-mount \
  -v ubuntu-app:/app/data:ro \
  ubuntu:22.04 sleep 3600
常见容器用户UID参考 Bash 
# Ubuntu容器
$ docker run --rm ubuntu:22.04 id
uid=101(ubuntu) gid=1000(ubuntu)

# Ubuntu容器
$ docker run --rm ubuntu:22.04 id
uid=0(root) gid=0(root)

# MySQL容器
$ docker run --rm mysql:8.0 id

Part 4.3

数据卷备份与恢复

数据安全至关重要，定期备份数据卷是生产环境的必要操作：

备份数据卷（使用Ubuntu容器） Bash 
# 创建测试数据
$ docker run -d --name ubuntu-backup-demo \
  -v app-backup-data:/data \
  ubuntu:22.04 sleep 3600
$ docker exec ubuntu-backup-demo sh -c 'echo "Important data $(date)" > /data/backup.txt'

# 方法1：备份到宿主机
$ docker run --rm -v app-backup-data:/data -v $(pwd):/backup ubuntu:22.04 \
  tar czf /backup/ubuntu-backup-$(date +%Y%m%d).tar.gz -C /data .

# 验证备份
$ ls -lh ubuntu-backup-*.tar.gz

# 方法2：备份到指定目录
$ mkdir -p ~/backups
$ docker run --rm \
  -v app-backup-data:/data \
  -v ~/backups:/backup \
  ubuntu:22.04 \
  tar czf /backup/ubuntu-backup-latest.tar.gz -C /data .
恢复数据卷 Bash 
# 步骤1：创建新数据卷
$ docker volume create app-restore-data

# 步骤2：从备份恢复
$ docker run --rm \
  -v app-restore-data:/data \
  -v ~/backups:/backup \
  ubuntu:22.04 \
  tar xzf /backup/ubuntu-backup-latest.tar.gz -C /data

# 步骤3：验证恢复
$ docker run -d --name ubuntu-restore-verify \
  -v app-restore-data:/app/data \
  ubuntu:22.04 sleep 3600
$ docker exec ubuntu-restore-verify cat /app/data/backup.txt
Part 4.4

数据卷使用最佳实践总结

推荐做法

  • 使用命名数据卷而非匿名卷
  • 不需要写入的数据使用只读挂载（:ro）
  • 为数据卷设置合理的备份策略
  • 定期使用 docker volume prune 清理
  • 监控数据卷的空间使用情况
  • 生产环境使用专门的存储驱动
  • 注意数据卷的权限和SELinux问题

避免做法

  • 在容器内直接存储重要数据
  • 使用绑定挂载访问系统敏感目录
  • 多个容器写入同一数据卷时未考虑并发
  • 忽略数据卷的权限问题
  • 生产环境使用匿名卷
  • 删除容器前未备份重要数据
  • 使用 docker volume prune 前未确认

实验二

动手实验：Ubuntu+配置文件管理

实验 Ubuntu自定义配置与多容器数据共享

实验目的

掌握应用配置文件的绑定挂载、多个容器共享数据卷、自定义网页

实验步骤

1创建宿主机目录结构
2编写自定义应用配置
3创建网页内容
4启动Ubuntu并测试
5修改配置验证热更新
6清理环境
实验命令 Bash 
# 步骤1：创建宿主机目录结构
$ mkdir -p ~/ubuntu-project/{html,config,logs}
$ cd ~/ubuntu-project

# 步骤2：编写应用配置文件
$ cat > config/default.conf << 'EOF'
server {
    listen 80;
    server_name localhost;
    root /app/data;
    location / { index index.html; }
    location /status { stub_status on; access_log off; }
    location /api { return 200 '{"status":"ok"}'; add_header Content-Type application/json; }
}
EOF

# 步骤3：创建网页内容
$ cat > html/index.html << 'EOF'
<!DOCTYPE html>
<html><head><title>Ubuntu Custom Config</title></head>
<body><h1>Welcome to Custom Ubuntu App</h1>
<p>Configuration managed via bind mount</p>
<ul><li><a href="/status">Server Status</a></li><li><a href="/api">API</a></li></ul>
</body></html>
EOF

# 步骤4：启动Ubuntu容器
$ docker run -d --name ubuntu-exp2 \
  -p 8080:80 \
  -v $(pwd)/config:/app/config:ro \
  -v $(pwd)/html:/app/data:ro \
  -v $(pwd)/logs:/app/logs \
  ubuntu:22.04 sleep 3600

# 步骤5：测试
$ curl http://localhost:8080
$ curl http://localhost:8080/status
$ curl http://localhost:8080/api

# 步骤6：修改配置验证热更新
$ cat > config/default.conf << 'EOF'
server {
    listen 80;
    server_name localhost;
    root /app/data;
    location / { index index.html; }
    location /status { stub_status on; access_log off; }
    location /api { return 200 '{"version":"2.0","status":"healthy"}'; add_header Content-Type application/json; }
    location /time { return 200 "{\"time\":\"$(date -Iseconds)\"}"; add_header Content-Type application/json; }
}
EOF

# 重新加载应用配置
$ docker exec ubuntu-exp2 cat /app/config/app.conf

# 验证更新
$ curl http://localhost:8080/api
$ curl http://localhost:8080/time

# 步骤7：清理
$ docker stop ubuntu-exp2 && docker rm ubuntu-exp2
$ rm -rf ~/ubuntu-project
Part 4.5

常见问题与解决方案

问题1：权限被拒绝 问题 
# 症状：容器无法写入数据卷
$ docker logs ubuntu-exp1
# [alert] open() failed (13: Permission denied)

# 解决方案：使用Z标签
$ docker stop ubuntu-exp1 && docker rm ubuntu-exp1
$ docker run -d --name ubuntu-exp1-fixed \
  -v app-logs:/app/logs:Z \
  ubuntu:22.04 sleep 3600
问题2：数据卷空间不足 问题 
# 排查
$ docker system df -v | grep -A 5 "Local Volumes"

# 解决方案1：清理未使用的卷
$ docker volume prune -f

# 解决方案2：扩大Docker存储空间
# Docker Desktop -> Settings -> Resources -> Disk image size
问题3：Windows路径问题 问题 
# 错误：invalid mount path

# 解决方案1：使用绝对路径
docker run -v //c/users/project:/app ubuntu:22.04

# 解决方案2：使用PowerShell变量
docker run -v ${PWD}:/app ubuntu:22.04

# 解决方案3：Docker Desktop设置共享驱动器
# Settings -> Resources -> File Sharing
