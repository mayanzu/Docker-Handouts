预备

本讲需要的预备知识

1集群是什么

集群（Cluster）就是把多台机器联合起来，当成一台"超级电脑"使用。你不再关心"应用跑在哪台机器上"，而是把任务交给集群，由它统一调度。Kubernetes就是管理这个集群的"大脑"。

2声明式是什么

命令式是"一步步告诉怎么做"（先启动A，再启动B）；声明式是"只说要什么结果"（我要3个副本一直运行）。K8s是声明式的：你写下期望，它负责实现并持续维持。

3控制器是什么

控制器（Controller）是一个不停循环检查的组件：它对比"你想要的"和"现在实际的"，发现不一致就动手纠正。容器挂了它自动重启，这就是K8s"自愈"的来源。

本讲涉及的工具，先混个脸熟

- Kubernetes（K8s）：容器集群编排系统，本讲主角。"K8s"是把"K"和"s"之间的8个字母缩写的写法。

- kubectl：和K8s集群对话的命令行工具，发音"cube-control"。你的所有指令都通过它发给集群。

- minikube：在你自己电脑上跑一个"单节点K8s集群"的工具，专为学习和开发设计。

- Pod：K8s里最小的部署单位，可以理解为"一个或一组紧密协作的容器"。

一句话串起来：用 kubectl 把"期望"发给 minikube 搭的 K8s 集群，集群帮你把应用跑成 Pod。

热身

3 分钟快速演示：你的第一个Pod

先不管原理，跑一遍感受 K8s 和 docker run 的差别。假设你已经 minikube start 成功。

Step 1：让集群跑一个Nginx Bash 
# 注意：这里不是docker run，而是把"我要一个nginx"告诉集群
$ kubectl create deployment my-nginx --image=nginx:alpine

# 查看Pod——集群已经帮你把nginx跑起来了
$ kubectl get pods
# NAME                        READY   STATUS    RESTARTS   AGE
# my-nginx-xxxxxxxxx-yyyyy     1/1     Running   0          20s
Step 2：见证"自愈"——删掉它试试 Bash 
# 手动删除这个Pod
$ kubectl delete pod my-nginx-xxxxxxxxx-yyyyy

# 立刻再查，会发现——又冒出来一个新的Pod！
$ kubectl get pods
# my-nginx-xxxxxxxxx-zzzzz     1/1     Running   0          5s
等等！删掉的Pod为什么自己回来了？

用 docker rm 删掉容器，它就永远没了。但在 K8s 里，你之前声明的期望是"要有一个my-nginx一直运行"。

你删掉一个，集群发现"实际只有0个，期望是1个"，于是自动补上一个新的。这就是 K8s 的核心——声明式 + 自愈。

它不像 Docker 那样"执行一次命令"，而是持续努力维持你声明的状态。带着"它凭什么能做到"这个问题，进入 Part 1。

Part 1

为什么需要Kubernetes

从单机Compose到集群编排

1.1

Docker Compose的天花板

单机编排的局限
第十讲的 Docker Compose 解决了"在一台机器上编排多个容器"的问题。但它有一个根本前提：所有容器都跑在同一台机器上。一旦业务增长，单机就会触及天花板。

想象你用 Compose 部署了一个电商网站，运行得很好。但随着用户增多，新的问题接踵而至——这些问题，Compose 都无能为力，而它们正是 Kubernetes 要解决的。

场景 Docker Compose（单机） Kubernetes（集群）
机器扛不住 只能升级这一台机器（纵向扩展，有上限） 加机器进集群，横向扩展，几乎无上限
机器宕机 整台机器挂了，服务全停 自动把容器调度到其他健康机器上
容器崩溃 靠restart策略重启，能力有限 控制器持续监控，自动重建并维持副本数
流量高峰 手动改配置、手动扩容 支持自动扩缩容（HPA），按负载增减副本
滚动更新 需手动停旧启新，易中断 内置滚动更新与回滚，不中断服务
1.2

Kubernetes是什么

Kubernetes定义
Kubernetes是一个开源的容器编排平台，用于自动化容器化应用的部署、扩缩容和管理。它最初由Google基于内部Borg系统的经验开发，2014年开源，现由CNCF（云原生计算基金会）托管，是云原生时代事实上的标准。

"Kubernetes"源自希腊语，意为"舵手"或"领航员"，负责调度和管理成千上万个容器在集群中有序运行。常缩写为 K8s（K和s之间有8个字母）。

1自动调度

根据资源情况，自动决定容器跑在哪台机器上

2自愈能力

容器/节点故障时自动重建、迁移，维持期望状态

3弹性伸缩

按负载自动增减副本数，应对流量波动

4服务发现

内置DNS与负载均衡，容器间通过名字互相访问

K8s不是要取代Docker

初学者常误以为"学了K8s就不用Docker了"。其实两者是不同层次的工具：

- Docker：负责"造容器"——打镜像、跑单个容器，是容器运行时层面。

- Kubernetes：负责"管一大堆容器"——在多台机器上调度、编排，是编排层面。

K8s 底层仍然需要一个容器运行时来真正跑容器（如 containerd）。你用 Docker 构建的镜像，照样能在 K8s 里运行。可以说：Docker 管"一个"，K8s 管"一群"。

K8s适合什么场景？

适合：微服务架构、需要高可用、需要弹性伸缩、多团队多环境、跨多台服务器的中大型系统。

不一定需要：个人项目、单机小应用、几个容器就能搞定的场景——这时 Docker Compose 反而更简单。不要为了用K8s而用K8s，它的复杂度是有成本的。

1.3

声明式与期望状态

命令式 vs 声明式
命令式（Imperative）：你下达一条条具体指令，告诉系统"怎么做"。声明式（Declarative）：你描述"想要的最终状态"，系统自己想办法达成并维持。Kubernetes 是声明式的，这是理解它一切行为的钥匙。
两种风格的对比 对比 
  命令式（像Docker）：一步步指挥
    docker run nginx        # 启动第1个
    docker run nginx        # 启动第2个
    docker run nginx        # 启动第3个
    # 如果挂了一个，你得自己发现并再run一次

  声明式（像K8s）：只声明结果
    "我要3个nginx副本一直运行"   （写在YAML里）
    # 挂了一个？K8s自动补到3个
    # 想要5个？把数字改成5，它自动多起2个
    # 你永远只描述"要什么"，不关心"怎么做"
期望状态与控制循环
你声明的目标叫期望状态（Desired State），集群当前的真实情况叫实际状态（Current State）。K8s内部的控制器不停运行一个控制循环（Control Loop）：对比两者，发现差异就采取行动，让实际状态不断向期望状态靠拢。
控制循环：K8s永不停歇的"找茬-纠正" 流程 
        ┌─────────────────────────────┐
        │   期望状态：3个nginx副本      │
        └──────────────┬──────────────┘
                       │ 对比
                       v
        ┌─────────────────────────────┐
        │   实际状态：现在只有2个       │   ← 有一个崩溃了
        └──────────────┬──────────────┘
                       │ 发现差异（少1个）
                       v
        ┌─────────────────────────────┐
        │   采取行动：再创建1个         │
        └──────────────┬──────────────┘
                       │ 循环往复（每时每刻）
                       └──────► 回到对比

  这个循环一刻不停，所以K8s能"自愈"——
  这正是热身演示里"删掉Pod又自己回来"的原因。

学习集群时，minikube是"麻雀虽小五脏俱全"

用 minikube 搭的本地集群，通常把控制平面和工作节点合并在一个节点上（单节点集群）。组件一个不少，只是没有"多台机器"的物理隔离。理解了架构，再上多节点集群就水到渠成。

2.1

控制平面组件

控制平面（Control Plane）
控制平面是集群的"大脑"，由四个核心组件组成，负责接收请求、存储状态、调度决策、维持期望。它本身不跑业务容器，只做管理。
① API Server——集群的唯一入口

所有组件、所有 kubectl 命令，都必须经过 API Server。它对外接收请求、做认证鉴权，对内是其他组件交流的中转站。

关键：其他组件之间不直接对话，全部通过 API Server。这种"中心化"设计让集群清晰可控。

② etcd——集群的数据库

etcd 是一个分布式键值数据库，保存集群的全部状态：有哪些Pod、跑在哪、配置是什么……它是集群的"唯一事实来源"。

关键：etcd 里存的是"期望状态"。API Server 把你的声明写进 etcd，其他组件读 etcd 来干活。etcd 数据丢了，集群就失忆了，所以生产环境必须定期备份。

③ Scheduler——调度器

当一个新 Pod 需要运行，Scheduler 负责决定它该放到哪个工作节点。它会综合考虑：哪个节点资源够（CPU/内存）、有没有特殊要求（标签、亲和性）等，挑出最合适的节点。

④ Controller Manager——控制器管理器

它运行着各种控制器，每个控制器盯着一类资源跑"控制循环"。比如 Deployment 控制器盯着"副本数对不对"，节点控制器盯着"节点还活着吗"。

这就是Part 1.3控制循环的执行者：发现实际≠期望，就动手纠正。K8s的"自愈"能力，主要由这里实现。

2.2

工作节点组件

工作节点（Worker Node）
工作节点是真正运行业务容器的机器。每个工作节点上都有三个关键角色：kubelet（节点代理）、kube-proxy（网络代理）、容器运行时（真正跑容器）。
① kubelet——节点代理

kubelet 是每个工作节点上的代理，听 API Server 指挥。它负责：接收"在本节点运行某Pod"的指令、调用容器运行时把容器拉起来、持续向控制平面汇报本节点和Pod的健康状况。

② kube-proxy——节点上的"网络交警"

kube-proxy 负责本节点的网络规则，实现 Service 的负载均衡和转发。当你访问一个 Service，它确保请求被正确转发到后端某个 Pod。

铺垫：Service 的细节是第十四讲的内容，这里只需知道"容器间通信、负载均衡靠它落地"。

③ 容器运行时（Container Runtime）

真正"跑容器"的底层程序，常见的是 containerd。kubelet 通过标准接口（CRI）调用它来创建、启动、停止容器。

历史小知识：早期 K8s 直接用 Docker 作为运行时，1.24 版本起移除了内置的 Docker 支持（dockershim），改用更轻量的 containerd。但你用 Docker 构建的镜像完全不受影响，因为镜像格式是统一的（OCI标准）。

控制平面 vs 工作节点，一句话区分

控制平面"动脑"（决策、存储、调度），工作节点"动手"（真正跑容器）。kubectl 把命令发给控制平面的 API Server，API Server 协调各组件，最终由工作节点的 kubelet 把容器跑起来。

2.3

一个Pod的诞生全过程

把前面的组件串起来，看一个最经典的问题：当你执行 kubectl apply 要部署一个应用，集群内部到底发生了什么？理解这条链路，整个架构就活了。

从"声明"到"运行"的完整旅程 分步流程 
  场景：你执行 kubectl apply -f deployment.yaml（要3个nginx）

  第1步：kubectl → API Server
    kubectl 把YAML（期望状态）发给 API Server
    API Server 认证、校验后，把它写入 etcd

  第2步：Controller Manager 发现新任务
    Deployment控制器从 API Server 得知"要3个nginx"
    当前实际是0个，于是创建3个Pod对象（仅登记，还没真正运行）
    这3个Pod此时状态是 Pending（待调度）

  第3步：Scheduler 调度
    Scheduler 发现有3个Pending的Pod没着落
    根据各节点资源，决定每个Pod放到哪个节点
    把"Pod→节点"的绑定结果写回 API Server（存入etcd）

  第4步：kubelet 启动容器
    目标节点上的 kubelet 从 API Server 得知"你要跑这个Pod"
    调用容器运行时（containerd）拉取nginx镜像、启动容器
    Pod 状态变为 Running

  第5步：持续汇报与维持
    kubelet 不断把Pod健康状况报回 API Server
    控制器持续对比：少了就补，多了就删，永远维持3个
这条链路揭示的三个设计精髓

1. 一切围绕API Server：所有组件只和它对话，通过读写 etcd 间接协作，互不直接依赖。

2. 状态驱动，而非命令驱动：没有谁"命令"kubelet干活，而是各组件不断观察状态、响应变化。

3. 职责分离：登记（控制器）、决策放哪（调度器）、落地执行（kubelet）各司其职，清晰解耦。

为什么Pod会卡在Pending？

看懂上面的流程就明白：Pod 长期 Pending，通常是第3步调度失败——没有节点满足它的资源要求（如内存不够），Scheduler 找不到合适的节点放它。这是后面故障排查的高频问题。

Part 3

核心对象概念

Pod、Node、Namespace

3.1

Pod——最小调度单位

Pod定义
Pod是Kubernetes中最小的部署和调度单位。它是一个或多个容器的组合，这些容器共享同一个网络（同一个IP）和存储卷，总是被一起调度到同一个节点、一起创建、一起销毁。

这是初学者的第一个关键认知转变：在 Docker 里，最小单位是"容器"；在 K8s 里，最小单位是"Pod"。K8s 不直接管理容器，而是管理 Pod。大多数情况下，一个 Pod 里就一个容器，但 Pod 这层抽象给了 K8s 更大的灵活性。

Pod内部结构示意 示意图

  │   ┌────────────┐      ┌────────────┐                 │
  │   │  容器A      │      │  容器B(可选)│                 │
  │   │  主应用     │      │  辅助/Sidecar│                │
  │   └────────────┘      └────────────┘                 │
  │         │                    │                       │
  │         └──── 共享网络 lo ────┘  ← 彼此用localhost互访 │
  │         └──── 共享存储卷 ──────┘                       │
  └──────────────────────────────────────────────────────┘

  同一Pod内的容器：用 localhost 就能互相通信
  不同Pod之间：用各自的Pod IP 通信
Pod内的容器如何协作

Pod是K8s的最小部署单位，内部可以包含一个或多个容器：

- 一个Pod里可以有一个或多个容器

- 同一Pod内的容器共享网络和存储，彼此通过 localhost 互通

- Pod是整体调度、整体销毁的最小单位，不会只启动或销毁其中一部分容器

这呼应了第九讲学过的 Container 网络模式——同Pod容器共享网络栈，正是那个原理在K8s里的体现。

什么时候一个Pod放多个容器？

多数 Pod 只有一个容器。但当两个容器紧密耦合、必须同生共死时，会放进同一个Pod，典型是 Sidecar（边车）模式：

- 日志收集：主容器跑应用，边车容器收集它的日志并外发

- 代理：主容器跑业务，边车容器做网络代理（如服务网格的 Envoy）

- 内容同步：主容器是Web服务器，边车定时从Git拉取最新网页内容

重要：一般不直接创建Pod

虽然能用 kubectl run 直接创建一个裸 Pod，但生产中几乎从不这么做。因为裸 Pod 没有"自愈"——节点挂了它就没了，不会自动重建。我们总是通过 Deployment 等控制器来间接管理 Pod，由控制器保证副本数。这是第十三讲的核心内容。

3.2

Node与Namespace

Node（节点）
Node 是集群中的一台机器（物理机或虚拟机），是 Pod 真正运行的地方。每个 Node 上都装有 kubelet、kube-proxy 和容器运行时。Node 是 K8s 的"资源池"——所有 Pod 都从 Node 提供的 CPU、内存里分配资源。
查看集群节点 Bash 
# 查看集群有哪些节点
$ kubectl get nodes
# NAME       STATUS   ROLES           AGE   VERSION
# minikube   Ready    control-plane   5m    v1.30.0
# （minikube是单节点，既是控制平面又是工作节点）

# 查看节点详情（资源、状态、上面跑了哪些Pod）
$ kubectl describe node minikube
Namespace（命名空间）
Namespace 是集群内部的逻辑隔离分区，用来把一个物理集群"切分"成多个互不干扰的虚拟空间。不同 Namespace 里可以有同名资源，便于多团队、多环境共用一个集群。
Namespace常用操作 Bash 
# 查看所有命名空间
$ kubectl get namespaces
# NAME              STATUS   AGE
# default           Active   10m   ← 你不指定时默认用这个
# kube-system       Active   10m   ← K8s自身组件运行在这里
# kube-public       Active   10m

# 创建一个命名空间
$ kubectl create namespace dev

# 在指定命名空间里查看Pod（-n 指定命名空间）
$ kubectl get pods -n kube-system
# 能看到 coredns、etcd、kube-apiserver 等系统组件
Node 与 Namespace 的区别

Node（节点）：提供真实计算资源（CPU/内存）的机器，集群有几台机器就有几个Node。

Namespace（命名空间）：对集群资源做逻辑分组，例如 dev、test 各占一个命名空间，互不干扰；不同命名空间下可以存在同名资源而不冲突。

注意：Namespace 是逻辑隔离，不是物理隔离。同一个Node上可以跑着不同Namespace的Pod。

kube-system：千万别乱动

kube-system 命名空间里跑着 K8s 自己的核心组件（API Server、etcd、CoreDNS等）。初学者不要在这里创建或删除东西，误删系统组件会导致集群瘫痪。你的应用应该放在 default 或自建的命名空间里。

3.3

控制器与Service概览

Pod、Node、Namespace 是地基。在它们之上，K8s 还有一组更高层的对象，本讲先建立印象，第十三、十四讲再深入。理解它们"是干什么的"，比记住细节更重要。

Deployment（部署）——下一讲主角
Deployment 是最常用的工作负载控制器，负责管理一组相同的 Pod 副本。你声明"我要3个nginx"，Deployment 就保证始终有3个在运行，并支持滚动更新、回滚、扩缩容。前面热身演示里"删掉Pod又回来"，正是 Deployment 在背后兜底。
Service（服务）——第十四讲主角
Pod 是"短命"的——会被销毁重建，IP 随时变化。Service 给一组 Pod 提供一个固定的访问入口（稳定IP和DNS名），并自动负载均衡到后端健康的 Pod，解决了"Pod IP 总变、没法稳定访问"的问题。
核心对象的层次关系 层次图 
  Deployment（我要3个nginx，并保证它们一直在）
       │ 管理
       v
  ReplicaSet（具体维持3个副本的控制器，由Deployment创建）
       │ 管理
       v
  Pod、Pod、Pod（3个真正运行的最小单位）
       │ 运行在
       v
  Node（机器）
       ▲
       │ 对外暴露访问入口
  Service（给这3个Pod一个固定入口 + 负载均衡）
本讲到此为"概念建立"，命令在Part 4

Part 3 的目标是让你脑海里有一张"K8s对象地图"。不必纠结于如何写 Deployment/Service 的YAML（那是13、14讲的事）。接下来 Part 4 我们搭好集群，亲手用 kubectl 操作这些对象，把概念落到实处。

4.4

kubectl核心命令

kubectl定义
kubectl 是与集群交互的命令行工具，本质是 API Server 的客户端——你敲的每条命令，都被翻译成 HTTP 请求发给 API Server。掌握几个高频命令，就能应付日常80%的操作。
查看类命令（最常用） Bash 
# get：列出某类资源（pods/nodes/services/deployments...）
$ kubectl get pods                 # 当前命名空间的Pod
$ kubectl get pods -A              # 所有命名空间（-A = --all-namespaces）
$ kubectl get pods -o wide         # 显示更多信息（IP、所在节点）
$ kubectl get all                  # 查看常见资源的总览

# describe：查看某个资源的详细信息和事件（排错神器）
$ kubectl describe pod 
# 输出底部的 Events 部分，是排查问题的第一现场

# logs：查看Pod里容器的日志（等价于docker logs）
$ kubectl logs 
$ kubectl logs -f           # -f 实时跟踪

# exec：进入Pod里的容器（等价于docker exec）
$ kubectl exec -it  -- sh   # 注意中间的 --
命令的统一格式

kubectl 命令几乎都遵循同一个套路：

kubectl [动作] [资源类型] [资源名] [选项]

- 动作：get（看列表）、describe（看详情）、create/apply（创建）、delete（删除）、logs、exec…

- 资源类型：pod、node、service(svc)、deployment(deploy)、namespace(ns)…（括号是简写）

例如 kubectl get svc -n dev = 查看dev命名空间的所有Service。记住这个格式，命令就不用死记。

exec命令里的 -- 是什么？

kubectl exec -it mypod -- sh 中间的 -- 用来分隔"kubectl的参数"和"要在容器里执行的命令"。-- 后面的 sh 是在容器里跑的。漏写 -- 是新手常见错误。

4.5

命令式：快速创建资源

操作 K8s 有两种风格，对应 Part 1.3 讲的命令式与声明式。先看命令式——用一条条 kubectl 命令直接创建，适合快速实验和临时操作。

命令式部署与暴露一个Nginx Bash 
# 1. 创建一个Deployment（它会保证Pod一直在）
$ kubectl create deployment web --image=nginx:alpine

# 2. 查看创建出来的Deployment和Pod
$ kubectl get deployments
$ kubectl get pods

# 3. 扩容到3个副本（声明"我现在要3个"）
$ kubectl scale deployment web --replicas=3
$ kubectl get pods    # 现在能看到3个web-xxx的Pod

# 4. 把Deployment暴露为Service，对外提供访问入口
$ kubectl expose deployment web --port=80 --type=NodePort

# 5. 用minikube打开这个服务（自动在浏览器访问）
$ minikube service web
# 看到Nginx欢迎页，说明从Service到Pod的链路通了
这几条命令复习了Part 3的对象

- create deployment → 创建 Deployment（站长），它自动建出 Pod

- scale --replicas=3 → 修改期望副本数，控制器自动补齐到3个

- expose ... --type=NodePort → 创建 Service（客服电话），给3个Pod一个统一入口

短短几行，就把"声明式 + 控制器 + 服务发现"全跑了一遍。

命令式的缺点

命令式快，但不可复现、难维护：一堆命令散落各处，换个环境得重敲一遍，也没法存进 Git 做版本管理。这正是生产环境更偏爱声明式的原因——见下一页。

4.6

声明式：用YAML管理

声明式（kubectl apply）
声明式是 K8s 的推荐做法：把期望状态写进 YAML 文件，用 kubectl apply -f 提交给集群。文件即"事实"，可版本化、可复现、可审阅。这与第十讲的 docker-compose.yml 一脉相承——都是"用文件描述期望"。
deployment.yaml（逐行注释） YAML 
apiVersion: apps/v1          # 用哪个API版本
kind: Deployment             # 对象类型：Deployment
metadata:
  name: web                  # 这个Deployment叫web
spec:
  replicas: 3                # 期望状态：3个副本（核心！）
  selector:
    matchLabels:
      app: web               # 管理带 app=web 标签的Pod
  template:                  # Pod模板：副本长什么样
    metadata:
      labels:
        app: web             # 给Pod打上 app=web 标签
    spec:
      containers:
      - name: nginx          # 容器名
        image: nginx:alpine  # 用哪个镜像
        ports:
        - containerPort: 80  # 容器监听的端口
应用与更新 Bash 
# 提交期望状态给集群（创建或更新）
$ kubectl apply -f deployment.yaml

# 想改成5个副本？改YAML里的replicas: 5，再apply一次即可
# K8s会自动对比差异，只做必要的变更（多起2个Pod）
$ kubectl apply -f deployment.yaml

# 查看声明式对象的rollout状态
$ kubectl rollout status deployment/web

# 删除：用同一份文件，确保删干净
$ kubectl delete -f deployment.yaml
命令式 vs 声明式，如何选？

命令式（create/scale/expose）：适合临时实验、快速验证、学习探索。优点是快，缺点是不可复现。

声明式（apply -f yaml）：适合正式项目、团队协作、生产部署。YAML文件提交到Git，谁都能复现同一套环境，改配置只需改文件再apply。

最佳实践：学习时多用命令式快速感受，真正部署时一律用声明式YAML。这和第十讲"用docker-compose.yml而非一堆docker run"的道理完全一致。

apply 的"幂等"特性

kubectl apply 可以反复执行而不会出错：第一次创建，之后每次都只把实际状态调整到与文件一致。这种"执行多少次结果都一样"的特性叫幂等，是声明式的一大优势——你不用关心"现在是第几次执行"。

步骤6预期：Pod数量变为4，且无需重建已有Pod——K8s只做"差量"调整。

对比第十讲，体会进步

第十讲用 Compose 也能跑3个Nginx，但删掉一个不会自动恢复，机器宕机就全没了。这里的 Deployment 会持续维持副本数，这正是从"单机编排"迈向"集群编排"的本质飞跃。

5.1

常见问题排查指南

排查总原则
K8s 排查的万能起手式是两条命令：kubectl get pods（看状态）和 kubectl describe pod <名>（看详情和Events）。describe 输出底部的 Events 几乎总能告诉你问题出在哪。
问题1：Pod一直处于 Pending

现象：get pods 显示 STATUS=Pending，长时间不变。

原因：调度失败，没有节点能容纳它（最常见是资源不足）。

解决：

1. kubectl describe pod <名> 看 Events，常见提示 "Insufficient cpu/memory"

2. 给 minikube 分配更多资源重启：minikube start --memory=4g --cpus=2

3. 或降低Pod的资源请求（resources.requests）

问题2：ImagePullBackOff / ErrImagePull

现象：Pod 状态为 ImagePullBackOff。

原因：拉取镜像失败——镜像名拼错、标签不存在、或网络拉不到。

解决：

1. 检查 image 拼写和标签是否正确

2. kubectl describe pod <名> 看 Events 里的具体拉取错误

3. 国内网络可配置镜像加速，或先 docker pull 验证镜像能拉到

问题3：CrashLoopBackOff

现象：Pod 反复重启，状态 CrashLoopBackOff。

原因：容器启动后很快就崩溃退出，K8s 不停重启它。通常是应用本身报错、配置缺失、依赖未就绪。

解决：

1. kubectl logs <名> 看容器日志，定位崩溃原因（最关键）

2. 若已重启多次，看上一次的日志：kubectl logs <名> --previous

3. 检查环境变量、配置、启动命令是否正确

问题4：Service访问不通

现象：Pod 都Running，但通过Service访问不到。

原因：常见是 Service 的 selector 标签和 Pod 的 labels 对不上，导致Service没选中任何Pod。

解决：

1. kubectl describe svc <名> 看 Endpoints，若为空说明没选中Pod

2. 核对 Service 的 selector 与 Deployment 模板里的 labels 完全一致

3. minikube 下用 minikube service <名> 而非直接访问 NodePort

排查命令速查 Bash 
$ kubectl get pods -o wide          # 看状态、IP、所在节点
$ kubectl describe pod <名>          # 看详情，重点看Events
$ kubectl logs <名>                  # 看容器日志
$ kubectl logs <名> --previous       # 看崩溃前的日志
$ kubectl get events --sort-by=.lastTimestamp  # 看全局事件
$ kubectl describe svc <名>          # 看Service的Endpoints
总结

本讲核心要点回顾

核心知识点

1. K8s解决Compose的单机局限：跨机调度、自愈、弹性伸缩、滚动更新

2. 声明式是核心思想：你声明期望状态，控制循环持续让现实逼近期望

3. 架构分两层：控制平面（API Server/etcd/Scheduler/Controller Manager）动脑，工作节点（kubelet/kube-proxy/运行时）动手

4. 核心对象：Pod是最小单位，Node是机器，Namespace是逻辑隔离，Deployment管副本，Service给入口

5. kubectl是唯一入口工具：get/describe/logs/exec四件套，apply -f提交YAML

6. minikube用于本地学习，一条命令搭起五脏俱全的单节点集群

从Docker到K8s的认知跃迁

- 管理单位：从"容器"变成"Pod"

- 操作方式：从"命令式docker run"变成"声明式apply YAML"

- 可靠性：从"挂了就没了"变成"控制器自愈，持续维持期望"

- 范围：从"一台机器"变成"一个集群"

抓住"声明期望 + 控制循环"这条主线，K8s 再多的概念都有了归属。

清理

实验环境清理

为什么要清理？
minikube 集群会持续占用内存、CPU和磁盘（尤其它本身就是一个大容器/虚拟机）。完成实验后及时清理，可以释放资源、避免下次启动冲突。
删除本讲创建的资源 Bash 
# 进入项目目录
$ cd ~/k8s-first

# 用同一份YAML删除（声明式清理，干净彻底）
$ kubectl delete -f service.yaml
$ kubectl delete -f deployment.yaml

# 或按类型删除命令式创建的资源
$ kubectl delete deployment web 2>/dev/null
$ kubectl delete service web web-svc 2>/dev/null

# 确认已清空
$ kubectl get all
停止或删除整个集群 Bash 
# 方式1：只停止集群，保留数据（下次start可继续）
$ minikube stop

# 方式2：彻底删除集群，释放全部资源（推荐课后执行）
$ minikube delete

# 验证清理
$ minikube status      # 应提示集群不存在或已停止
$ docker ps -a         # minikube相关容器应已移除
⚠️ 提示

minikube delete 会删除整个本地集群及其中所有资源（包括你部署的应用），不可恢复。下次做实验重新 minikube start 即可，会是一个全新的干净集群。

如果只是当天课程结束、明天还要继续，用 minikube stop 即可，保留环境更省事。
