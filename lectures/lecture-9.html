本讲需要的预备知识

1IP地址是什么

就像身份证号是人的唯一标识，IP地址是计算机在网络中的唯一标识。目前广泛使用的 IPv4 由 4 个数字组成，每个数字范围 0~255，之间用点分隔，如 192.168.1.1。

IP地址的构成：网络号 + 主机号

例：192.168.1.100 中，192.168.1 是网络号（相当于小区地址），100 是主机号（相当于门牌号）。同一个网络号的电脑可以直接互通，就像同一个小区里的住户。

CIDR 记法：子网掩码

IP 地址后面的 /24 叫 CIDR 记法，表示子网掩码。/24 = 255.255.255.0，意思是前 24 位是网络号，后 8 位是主机号。所以 10.0.2.0/24 表示所有 10.0.2.xxx 的地址在同一个网段。

私网IP vs 公网IP

私网IP：家庭/公司内部使用，无法被外网直接访问

172.17.0.2 — Docker 容器常用的网段（Docker 自动分配）

192.168.x.x — 家庭路由器常用网段

10.x.x.x — 大型内网/云服务器常用网段

公网IP：互联网上的唯一地址（如 8.8.8.8），全球任何地方都能访问。

2端口是什么

一台电脑可能有多个程序同时上网。端口就像大楼的房间号，帮操作系统把网络流量分配给正确的程序。

例如：Nginx通常占用80端口，MySQL占用3306端口。你访问网页时，实际上是在访问服务器的80端口。

3宿主机是什么

宿主机就是运行Docker的那台电脑。容器运行在宿主机上，就像租客住在一栋公寓里。外部访问容器服务时，实际上是先访问宿主机，再由宿主机转发给容器。

本讲常用的几个命令（先混个脸熟） Bash 
# ping：测试两台电脑是否能连通（像敲门的"咚咚咚"）
 ping 172.17.0.2

# curl：像浏览器一样访问网页，返回HTML内容
 curl http://localhost:8080

# docker ps：查看正在运行的容器
 docker ps

# docker exec：在容器内部执行命令
 docker exec nginx-1 ping -c 3 nginx-api

本讲案例：Nginx是什么

Nginx
Nginx（发音"engine-x"）是一款高性能的Web服务器和反向代理服务器。简单来说，它就像餐厅的前台接待员：顾客（浏览器）向它点菜（请求网页），它负责把订单送到后厨（后端服务），再把做好的菜（网页内容）端给顾客。

在本讲中，我们用Nginx作为统一案例，因为它安装简单、启动即提供一个可访问的网页服务，非常适合用来验证网络是否通、端口是否映射正确。你不需要会配置Nginx，只需要知道：启动Nginx容器后，用浏览器或 curl 访问它，如果能看到"Welcome to nginx!"页面，就说明网络是通的。

 curl http://localhost:8080

<html>
<head><title>Welcome to nginx!</title></head>
<body>
<h1>Welcome to nginx!</h1>
</body>
</html>

# 停止并删除这个体验容器
 docker stop my-nginx && docker rm my-nginx

3. 贴近实际：Nginx是生产环境最常用的Web服务器，学会它的容器网络配置有实际价值

1.1

网络命名空间——容器的独立网络世界

网络命名空间（Network Namespace）
Linux内核提供的一种资源隔离机制。每个网络命名空间拥有独立的网络设备、IP地址、路由表和防火墙规则。就像每栋写字楼有独立的水电系统，互不干扰。

当你启动一个Docker容器时，Docker会为其创建一个新的网络命名空间。容器内的 eth0 网卡，只在这个命名空间内可见，宿主机默认看不到它。我们可以通过WSL中的 ip netns 命令手动体验这个过程。

WSL演示：创建独立的网络命名空间 Bash 
# 查看当前的网络接口（默认在"根命名空间"里）
# lo = loopback（本地回环），自己访问自己用的
# eth0 = 以太网卡，连接外部网络用的
 ip addr
1: lo: ... inet 127.0.0.1         # 本地回环地址，127.0.0.1永远代表"本机"
2: eth0: ... inet 172.21.176.1    # WSL的网卡，有自己的IP

# sudo = 以管理员身份运行（需要权限）
# ip netns = 操作网络命名空间的命令
# add ns-1 = 创建一个叫 ns-1 的新命名空间
 sudo ip netns add ns-1

# ip netns exec ns-1 xxx = 在 ns-1 这个命名空间里执行 xxx 命令
# 在 ns-1 里查看——竟然只有 lo，没有 eth0！
 sudo ip netns exec ns-1 ip addr
1: lo: <LOOPBACK> state DOWN ...   # ns-1 自己的回环接口

# 结论：ns-1 是一个完全独立的网络世界
# 它有自己的 lo，但没有任何网卡，所以无法访问任何其他电脑
# 这就是容器的网络隔离本质——每个容器都是"另起一张网"
关键理解

容器的网络隔离不是"没有网络"，而是拥有一张完全独立的网络。Docker做的第一件事，就是像 ip netns add ns-1 一样，为每个容器创建一个独立的网络命名空间。容器内的 eth0 只在容器内部可见，宿主机和其他容器都看不到。

小知识：为什么需要 sudo

创建网络命名空间需要Linux内核权限，就像安装软件需要管理员权限一样。在WSL中用 sudo 可以临时获取管理员权限。

1.2

veth pair——连接两个命名空间的虚拟网线

veth pair（虚拟以太网对）
veth pair 是一对虚拟网卡，像一根网线的两端：数据从一端进入，必定从另一端出来。Docker用它连接容器的网络命名空间和宿主机，就像用网线把电脑连到交换机。
WSL演示：手动创建veth pair连接命名空间 Bash 
# 创建一对虚拟网卡（就像拉一根网线，两头各插一台电脑）
# veth-1    = 插在宿主机这头
# veth-1-peer = 插在 ns-1 容器那头
# 数据从一头进去，必然从另一头出来
 sudo ip link add veth-1 type veth peer name veth-1-peer && \
sudo ip link set veth-1-peer netns ns-1 && \
sudo ip addr add 10.0.0.1/24 dev veth-1 && \
sudo ip link set veth-1 up && \
sudo ip netns exec ns-1 ip addr add 10.0.0.2/24 dev veth-1-peer && \
sudo ip netns exec ns-1 ip link set veth-1-peer up

# 测试连通性！从宿主机 ping ns-1 的 IP
# ping = 发一个"数据包"过去，看有没有回应（像敲门看有没有人）
# -c 3 = 只发3次就停下
 ping -c 3 10.0.0.2
64 bytes from 10.0.0.2: time=0.042 ms   # 通了！延迟0.042毫秒
WSL手动创建 vs Docker自动创建 对照表 
  我们手动做的：              Docker自动做的：
  ----------------            ------------------
  ip netns add ns-1    ===>   创建容器的网络命名空间
  ip link add veth...  ===>   创建veth pair（一对虚拟网卡）
  ip link set ... netns ===>  将一端放入容器命名空间
  ip addr add ...      ===>   分配IP地址给容器
核心结论

Docker容器的网络，本质上就是我们在WSL中手动演示的这套流程的自动化封装。

1.3

网桥——让多个容器互联互通

网桥（Bridge）
功能类似于交换机。Docker默认创建的 docker0 网桥将所有 veth 的一端连接在一起，使容器之间可以像同一个局域网内的电脑一样直接通信。

如果有三个、五个容器怎么办？不可能每两个之间都拉一根veth pair。网桥就是解决方案：所有veth的一端都插到网桥上，网桥负责转发数据包。

WSL演示：创建网桥连接多个命名空间 Bash 
# 创建第一个命名空间 ns-2（模拟第一个容器）
 sudo ip netns add ns-2

# 创建第二个命名空间 ns-3（模拟第二个容器）
 sudo ip netns add ns-3

# 创建网桥 br-1（相当于一台交换机，可以插很多网线）
 sudo ip link add br-1 type bridge
 sudo ip addr add 10.0.1.1/24 dev br-1   # 给网桥分配IP（作为网关）
 sudo ip link set br-1 up                 # 开启网桥

# 创建第一根 veth 网线，一端连 ns-2，一端连网桥
 sudo ip link add veth-2 type veth peer name veth-2-peer && \
sudo ip link set veth-2 master br-1 && \
sudo ip link set veth-2 up && \
sudo ip link set veth-2-peer netns ns-2 && \
sudo ip netns exec ns-2 ip addr add 10.0.1.2/24 dev veth-2-peer && \
sudo ip netns exec ns-2 ip link set veth-2-peer up

# 创建第二根 veth 网线，一端连 ns-3，一端连网桥
 sudo ip link add veth-3 type veth peer name veth-3-peer && \
sudo ip link set veth-3 master br-1 && \
sudo ip link set veth-3 up && \
sudo ip link set veth-3-peer netns ns-3 && \
sudo ip netns exec ns-3 ip addr add 10.0.1.3/24 dev veth-3-peer && \
sudo ip netns exec ns-3 ip link set veth-3-peer up

# 测试：ns-2 和 ns-3 通过网桥互通！
 sudo ip netns exec ns-2 ping -c 3 10.0.1.3
64 bytes from 10.0.1.3: time=0.051 ms   # 通了！两个容器可以互相通信

# 查看网桥上连着哪些"网线"（veth）
 bridge link
2: veth-2 state UP : master br-1  # veth-2 连着 br-1
4: veth-3 state UP : master br-1  # veth-3 也连着 br-1
网桥拓扑示意 架构图 
         +------------------+
         |     br-1 网桥     |  <-- 相当于交换机，可以插很多网线
         |   10.0.1.1（网关） |
         +--+------+------+-+
            |      |
       veth-2    veth-3       <-- 两根"网线"插在网桥上
            |      |
         +--+---+--+---+
         | ns-2 | ns-3 |       <-- 两个独立的"电脑"（命名空间）
         |10.0.1|10.0.1|
         |.2    |.3    |
         +------+------+
为什么需要网桥

如果没有网桥，每两个容器之间都要拉一根专属的veth线——3个容器要3根线，10个容器要45根线。有了网桥，就像装了一台交换机，所有容器都插到交换机上，随便哪两个都能互通。

1.4

NAT——容器如何访问外网

NAT（网络地址转换）
容器的IP（如 172.17.0.2）是私网地址，外网无法直接访问。NAT负责在私网IP与宿主机公网IP之间做地址转换。

ns-4 和 ns-5 可以互相通信，但还不能访问外网。因为 10.0.2.0/24 是私网地址——外网路由器不认识这个地址，就像寄信时写了"某小区3号楼"但没有写城市名，快递员不知道往哪送。

解决方案是NAT（网络地址转换）：把容器发出的数据包的"发件人地址"从私网IP改成宿主机的公网IP，外网的服务器就能把响应送回来。这就像快递员帮你把"某小区3号楼"改成"北京市朝阳区某小区3号楼"。

WSL2 环境说明

WSL2 的根本限制：WSL2 使用 Hyper-V 虚拟网络，Hyper-V NAT 无法转发带有"伪装源 IP"的数据包。因此即配置了 iptables MASQUERADE，从自定义 namespace 发起的流量也无法真正出海。这是 WSL2 架构限制，非命令错误。

WSL2 替代测试：可验证数据包是否到达 WSL 宿主机 eth0（内网互通），但无法 ping 通外部公网 IP。

Linux 原生环境：所有命令均可完整复现，可直接 ping 通 8.8.8.8。

【WSL】完整演示：创建网络 + 配置 NAT Bash 
# ===== 第一步：创建网络拓扑（使用独立网段 10.0.2.x） =====

# 创建两个命名空间
 sudo ip netns add ns-4
 sudo ip netns add ns-5

# 创建网桥 br-2
 sudo ip link add br-2 type bridge
 sudo ip addr add 10.0.2.1/24 dev br-2
 sudo ip link set br-2 up

# 创建第一根 veth 网线，连 ns-4
 sudo ip link add veth-4 type veth peer name veth-4-peer && \
sudo ip link set veth-4 master br-2 && \
sudo ip link set veth-4 up && \
sudo ip link set veth-4-peer netns ns-4 && \
sudo ip netns exec ns-4 ip addr add 10.0.2.2/24 dev veth-4-peer && \
sudo ip netns exec ns-4 ip link set veth-4-peer up

# 创建第二根 veth 网线，连 ns-5
 sudo ip link add veth-5 type veth peer name veth-5-peer && \
sudo ip link set veth-5 master br-2 && \
sudo ip link set veth-5 up && \
sudo ip link set veth-5-peer netns ns-5 && \
sudo ip netns exec ns-5 ip addr add 10.0.2.3/24 dev veth-5-peer && \
sudo ip netns exec ns-5 ip link set veth-5-peer up

# 验证：ns-4 和 ns-5 互通
 sudo ip netns exec ns-4 ping -c 1 10.0.2.3

# ===== 第二步：配置 NAT，让容器能访问外网 =====

# 开启 IP 转发
 sudo sysctl -w net.ipv4.ip_forward=1

# 配置 SNAT 规则（MASQUERADE 自动把源 IP 替换成 eth0 的 IP）
 sudo iptables -t nat -A POSTROUTING \
  -s 10.0.2.0/24 -o eth0 -j MASQUERADE

# 配置默认网关
 sudo ip netns exec ns-4 ip route add default via 10.0.2.1
 sudo ip netns exec ns-5 ip route add default via 10.0.2.1

# ===== 第三步：测试（WSL 专用） =====
# 查看 WSL 宿主机 eth0 的 IP
 ip addr show eth0
# inet 172.x.x.x   ← 替换为你的实际 IP

# 验证数据包是否到达宿主机
 sudo ip netns exec ns-4 ping -c 3 172.x.x.x
# 如果能通 → SNAT 生效，数据包到达宿主机

64 bytes from 8.8.8.8: time=15.2 ms   # 成功！可以真正访问公网
数据包旅程：容器如何访问公网 分步流程 
  场景：ns-4（IP: 10.0.2.2）想 ping 通公网 IP（8.8.8.8）

  第一层：容器 → 宿主机（第一次 SNAT）
    ns-4 发出的数据包：来源IP 10.0.2.2，目标IP 8.8.8.8
    到达宿主机虚拟网桥后，内核做第一次 SNAT
    把来源IP从 10.0.2.2 替换成 宿主机的内网IP（172.21.176.1）
    此时数据包进入"宿主机所在的网络"（云VPC或局域网）

  第二层：宿主机 → 公网（第二次 SNAT）
    带着源IP 172.21.176.1 的数据包，来到了出口路由器（VPC网关）
    网关发现这是私网地址，外网不认，于是做第二次 SNAT
    把源IP替换成服务器的真实公网IP（如 47.100.x.x）
    数据包这才真正"出海"进入互联网

  第三层：到达目标服务器
    8.8.8.8 收到的请求，来自 IP 47.100.x.x（公网IP）
    它根本不知道底下还藏着 172.21.176.1 的宿主机
    更不知道最底层还有个 10.0.2.2 的容器

  响应沿原路返回：公网IP → VPC网关NAT回172.21.176.1 → 宿主机NAT回10.0.2.2
SNAT vs DNAT 的区别 对比 
  SNAT（Source NAT）——改源头
    场景：容器访问外网
    作用：把"发件人"从容器IP改成宿主机IP
    目的：让外网的响应能够送回来（外网不认识私网IP）

  DNAT（Destination NAT）——改目标
    场景：外网访问容器
    作用：把"收件人"从宿主机IP改成容器IP
    目的：让外部请求能够送达正确的容器

  生活中的类比：
  SNAT = 你寄信时，快递员把你的家庭地址改成公司地址，
         这样回信能寄到公司再转给你
  DNAT = 别人寄信到公司，前台根据门牌号转发给你
Docker网络全景

网络命名空间提供隔离（每个容器有独立网络）

→ veth pair提供连接（虚拟网线连接容器和宿主机）

→ 网桥提供交换（多容器互联互通）

→ NAT提供内外互通（让容器能访问外网，也让外网能访问容器）

这四层机制共同构成了Docker容器网络的基础。

Part 2

Bridge网络模式

默认bridge与自定义bridge网络

2.1

默认Bridge网络

默认Bridge网络
Docker安装后自动创建的 bridge 网络，网桥名为 docker0，网段通常为 172.17.0.0/16。所有未指定网络的容器都会自动连接到这个网桥。它是Docker的默认网络模式。

假设我们要运行一个Nginx容器对外提供Web服务。不指定网络时，Docker会自动将其接入默认bridge网络。

演示：启动两个Nginx容器（默认bridge） Bash 
# 启动第一个Nginx容器
 docker run -d --name nginx-1 nginx:alpine

# 启动第二个Nginx容器
 docker run -d --name nginx-2 nginx:alpine

# 查看容器分配的IP地址
 docker inspect --format '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}' nginx-1
172.17.0.2
 docker inspect --format '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}' nginx-2
172.17.0.3

# 从 nginx-1 通过 IP 访问 nginx-2
 docker exec nginx-1 ping -c 3 172.17.0.3
PING 172.17.0.3 (172.17.0.3): 56 data bytes
64 bytes from 172.17.0.3: seq=0 ttl=64 time=0.1 ms

# 但通过容器名无法访问（默认bridge无DNS）
 docker exec nginx-1 ping -c 3 nginx-2
ping: bad address 'nginx-2'
默认bridge的核心局限

默认bridge网络中的容器只能通过IP地址互相访问，不能通过容器名访问。因为默认bridge没有内置DNS服务发现机制。

2.2

自定义Bridge与DNS服务发现

自定义Bridge网络
用户手动创建的bridge网络。与默认bridge最大的区别是：自定义bridge内置了DNS服务发现。同一自定义网络中的容器可以通过容器名互相解析和访问，就像同一个公司的同事可以通过姓名找到对方，而不需要记住对方的手机号（IP地址）。

在实际项目中，我们通常会创建自定义网络来部署应用。比如一个Web前端和一个API后端，它们需要频繁通信，通过名字访问远比硬编码IP更可靠。

docker network create 用于创建一个自定义bridge网络。与默认bridge不同，自定义bridge内置DNS服务，同一网络中的容器可以通过容器名互相解析。

演示：自定义网络中的Nginx互相访问 Bash 
# 创建自定义bridge网络（网络名可自定义，这里用 my-net）
 docker network create my-net

# --network 参数：指定容器加入哪个网络

# 启动前端Nginx，接入自定义网络
 docker run -d --name nginx-web --network my-net nginx:alpine

# 启动API服务Nginx，接入同一网络
 docker run -d --name nginx-api --network my-net nginx:alpine

# 通过容器名直接访问！DNS自动解析
 docker exec nginx-web ping -c 3 nginx-api
PING nginx-api (172.18.0.3): 56 data bytes
64 bytes from 172.18.0.3: seq=0 ttl=64 time=0.1 ms

# 直接wget测试HTTP访问
 docker exec nginx-web wget -qO- http://nginx-api
<!DOCTYPE html>...（返回nginx-api的欢迎页）

# 查看Docker内置的DNS服务器地址
 docker exec nginx-web cat /etc/resolv.conf
nameserver 127.0.0.11
核心结论

默认bridge：只能IP通信，无DNS，适合单机简单测试。

自定义bridge：支持容器名DNS解析，适合多容器应用部署，是生产环境的首选。

Part 3

其他网络模式

Host、None、Container模式对比

3.1

Host模式——与宿主机共享网络

Host网络模式
容器不再拥有独立的网络命名空间，而是直接使用宿主机的网络栈。容器看到的网卡、IP地址、端口与宿主机完全一致。性能最好（无NAT开销），但隔离性最差。
演示：Host模式下的Nginx Bash 
# 启动Nginx使用host网络（无需-p端口映射！）
 docker run -d --name nginx-host --network host nginx:alpine

# 查看容器IP——和宿主机完全一样
 docker exec nginx-host ip addr
# 看到的 eth0 IP 就是宿主机的IP

# 直接访问宿主机的80端口即可访问Nginx
 curl http://localhost
<!DOCTYPE html>...（Nginx欢迎页）

# 注意：host模式下 -p 端口映射无效！
# 因为容器直接使用宿主机的端口
 docker run -d --name nginx-host2 --network host nginx:alpine
# 第二个容器会启动失败：80端口已被占用
Docker Desktop / WSL 特别提醒

在 Docker Desktop 或 WSL 环境中，--network host 的"宿主机"是 Linux 虚拟机，而非 Windows/macOS 系统本身。因此在 Windows/macOS 浏览器中访问 localhost 不会生效。

Linux 原生环境无此限制，可直接在浏览器访问 localhost。

【WSL / Docker Desktop】验证 Host 模式 Bash 
# 在 WSL/Docker Desktop 中，通过临时容器验证 host 模式是否生效：
 docker run --rm --network host alpine wget -qO- http://localhost
# 如果能看到 HTML 代码，说明 Nginx 成功占用了"底层虚拟机"的 80 端口
# --rm 表示容器退出后自动删除
【Linux 原生】验证 Host 模式 Bash 
# Linux 原生环境：直接在宿主机浏览器访问 localhost 即可
# 在宿主机浏览器打开 http://localhost
# 或在宿主机本地 curl 验证：
 curl http://localhost
<!DOCTYPE html>...
# 注意：host 模式下 -p 端口映射无效！
适用场景

对网络性能要求极高的场景（如高吞吐量的代理、负载均衡），且端口不会冲突时使用。

3.2

None模式——完全断网

None网络模式
容器拥有独立的网络命名空间，但不配置任何网络接口（除本地回环 lo 外）。容器完全无法联网，也无法被外部访问。就像一个被拔掉网线的电脑。
演示：None模式下的Nginx Bash 
# 启动无网络的Nginx容器
 docker run -d --name nginx-none --network none nginx:alpine

# 查看网络接口——只有lo，没有eth0
 docker exec nginx-none ip addr
1: lo: <LOOPBACK,UP,LOWER_UP> mtu 65536 ...
    inet 127.0.0.1/8 scope host lo
# 没有 eth0 网卡

# 无法访问外网
 docker exec nginx-none ping -c 3 baidu.com
ping: bad address 'baidu.com'

# 也无法从外部访问（没有网络接口）
 curl http://localhost
# 连接失败——因为没有端口映射，也没有eth0

# Nginx进程虽然运行着，但没人能访问到它
 docker exec nginx-none ps
PID   USER     TIME  COMMAND
    1 root      0:00 nginx: master process nginx -g ...
适用场景

离线批处理、安全隔离计算、不需要网络的后台数据处理任务。完全断网可以最大限度降低攻击面。

3.3

Container模式——共享网络栈

Container网络模式
新容器与指定容器共享同一个网络命名空间。两个容器拥有完全相同的IP地址、网卡配置和端口空间，就像两台电脑共用一根网线和同一个网卡。
演示：Container模式——Redis + redis-cli Bash 
# 启动 Redis 容器作为"网络基础"
 docker run -d --name redis-server redis:alpine

# 启动 redis-cli 容器，共享 redis-server 的网络
# 这个容器没有自己的网卡，用的是 redis-server 的
 docker run -it --rm --network container:redis-server \
  redis:alpine redis-cli -h 127.0.0.1

# 现在你在 redis-cli 里，直接连到 redis-server 的 localhost
127.0.0.1:6379> PING
PONG                    # 成功！不需要指定 IP，因为共享网络栈

# 退出：Ctrl+P 然后 Ctrl+Q
端口冲突实验 Bash 
# 在 Container 模式下，两个容器共享同一个端口空间
# 如果两个容器都想用同一个端口，会冲突

# 先启动一个 Redis（占用 6379 端口）
 docker run -d --name redis1 redis:alpine

# 尝试让另一个容器也用 6379 端口（会失败）
 docker run -d --name redis2 \
  --network container:redis1 \
  redis:alpine redis-server --port 6379

# 报错：Failed to bind to port ...
# 因为 redis1 已经占用了 6379，redis2 没法再用
适用场景

Sidecar模式：主容器运行服务，Sidecar容器做日志收集或监控，共享网络使得 Sidecar 可以直接通过 localhost 访问主容器，无需额外网络配置。

Part 4

端口映射

外部网络如何访问容器服务

4.1

端口映射原理——DNAT

为什么需要端口映射？
容器使用的是私网IP（如 172.17.0.2），外部网络无法直接访问。端口映射通过 -p 宿主机端口:容器端口 将宿主机的某个端口转发到容器的端口，本质上是DNAT（目标地址转换）。
端口映射工作原理——分步解析 原理图 
  场景：你在浏览器访问 http://宿主机IP:8080

  第1步：浏览器发出请求
    目标：宿主机IP:8080

  第2步：宿主机收到请求（网卡 eth0）
    发现目标是端口8080，匹配到DNAT规则

  第3步：DNAT转换——改目标地址
    把目标从"宿主机:8080"改成"172.17.0.2:80"
    就像前台把"找公司前台"改成"找3楼301房间"

  第4步：数据包发送到 docker0 网桥
    网桥查表，发现172.17.0.2在哪，提前路由

  第5步：通过 veth 送达 Nginx 容器（端口80）

  第6步：Nginx处理请求，返回网页内容
命令与原理对照 对照表 
  docker run -p 8080:80 nginx

  -p            = port，暴露端口
  8080（左边）   = 宿主机的端口，外部访问这个端口
  80（右边）     = 容器内部的端口，Nginx监听在这个端口
  nginx         = 镜像名

  简单理解：把容器的80端口"映射"到宿主机的8080端口，
            访问宿主机8080就等于访问容器80

当你执行 docker run -p 8080:80 nginx 时，Docker在宿主机上配置了一条iptables DNAT规则：所有发往宿主机8080端口的流量，都被转发到容器的80端口。

动手验证：查看 iptables DNAT 规则 Bash 
# 启动一个带端口映射的容器
 docker run -d -p 8080:80 --name test-nginx nginx:alpine

# 查看 NAT 表中的 DNAT 规则（端口映射的底层实现）
 iptables -t nat -L -n -v

# 你会看到类似这样的规则：
# DNAT  tcp  --  anywhere  anywhere  tcp dpt:8080 to:172.17.0.2:80
# 这就是 Part 1 讲的 DNAT——把目标地址从宿主机 8080 改成容器 172.17.0.2:80

# 清理
 docker stop test-nginx && docker rm test-nginx
为什么要映射端口

容器的IP是私网IP（如172.17.0.2），外部网络根本无法直接送达。而宿主机通常有公网IP或被局域网访问，所以通过端口映射，外部只需要访问"宿主机:8080"，宿主机再帮忙转发到容器的80端口。

与Host模式的区别

端口映射（-p）：容器仍有独立IP，通过NAT转发，有少量性能损耗，但隔离性好，多个容器可以映射相同端口（因为是映射到不同宿主机端口）。

Host模式（--network host）：容器无独立IP，直接占用宿主机端口，性能最好，但端口容易冲突（两个容器不能同时用80）。

4.2

端口映射演示

以Nginx为例，演示多种端口映射写法。每种写法对应不同的访问方式和权限控制。

端口映射的多种写法 Bash 
# 基础映射：宿主机8080 -> 容器80
 docker run -d --name nginx-p1 -p 8080:80 nginx:alpine
# 访问：curl http://localhost:8080

# 映射多个端口
 docker run -d --name nginx-p2 -p 8080:80 -p 8443:443 nginx:alpine

# 指定协议（UDP）
 docker run -d --name dns-test -p 53:53/udp nginx:alpine

# 绑定到特定IP（仅允许本地访问，局域网其他机器无法访问）
 docker run -d --name nginx-local -p 127.0.0.1:8080:80 nginx:alpine

# 映射到所有网络接口（默认行为，0.0.0.0可省略）
 docker run -d --name nginx-all -p 0.0.0.0:8080:80 nginx:alpine

# -P（大写）：暴露Dockerfile中EXPOSE声明的所有端口到随机高位端口
 docker run -d --name nginx-random -P nginx:alpine
 docker port nginx-random
80/tcp -> 0.0.0.0:32768   # 自动分配的随机端口
查看端口映射 Bash 
# 查看单个容器的端口映射
 docker port nginx-p1
80/tcp -> 0.0.0.0:8080

# 查看所有运行容器的端口映射
 docker ps --format "table {{.Names}}\t{{.Ports}}"
NAMES       PORTS
nginx-p1    0.0.0.0:8080->80/tcp
nginx-p2    0.0.0.0:8080->80/tcp, 0.0.0.0:8443->443/tcp
安全建议

生产环境中，建议显式绑定到需要的IP地址（如 -p 127.0.0.1:8080:80 或 -p 192.168.1.10:80:80），避免不必要地将服务暴露到公网。

docker network命令与常见问题定位

5.1

网络管理命令速查

docker network 核心命令 Bash

 docker network ls
NETWORK ID     NAME      DRIVER    SCOPE
abc123...      bridge    bridge    local
def456...      host      host      local

# 将运行中的容器连接到另一个网络
 docker network connect my-net nginx-1

# 断开容器的某个网络

 docker network rm my-net

 docker network prune
一个容器连接多个网络

一个容器可以同时属于多个网络，每个网络分配独立的IP地址。例如：Nginx容器同时连接 frontend-network（对外提供Web服务）和 backend-network（连接数据库），实现网络层的安全隔离。

5.2

网络故障排查流程

1确认网络模式

容器使用的是 bridge / host / none / container 哪种模式？不同模式排查方向完全不同。

2检查IP配置

docker inspect <容器> 查看IP地址、网关、DNS配置是否正确。

3测试连通性

先 ping IP 测试三层连通，再 ping 域名 测试DNS解析。

4检查端口映射

docker port <容器> 确认 -p 参数是否正确，宿主机防火墙是否放行。

常见问题与解决 Bash 
# 问题1：容器无法访问外网
 docker exec <容器> ping -c 3 8.8.8.8   # 先ping公网IP
 docker exec <容器> ping -c 3 baidu.com # 再ping域名
# 如果IP通但域名不通 -> DNS问题，检查 /etc/resolv.conf
# 如果都不通 -> 检查宿主机Docker服务和防火墙

# 问题2：外部无法通过端口访问容器
 docker port <容器>              # 检查端口映射
 curl http://localhost:<port>    # 本机测试
 ss -tlnp | grep <port>          # 检查宿主机是否监听该端口
# 如果宿主机没监听 -> 检查 -p 参数或容器是否运行
# 如果本机通但外部不通 -> 检查防火墙/安全组

# 问题3：容器间无法通过名字通信
 docker network inspect <网络>   # 确认容器是否在同一个网络
# 如果不在同一网络 -> docker network connect
# 如果在默认bridge -> 切换到自定义bridge网络

  nginx:alpine

# 步骤3：验证容器间通信（DNS解析）
 docker exec nginx-web ping -c 3 nginx-api
 docker exec nginx-web wget -qO- http://nginx-api

# 步骤4：启动反向代理，映射8080端口
 docker run -d --name nginx-proxy \
  --network frontend-network \
  -p 8080:80 \
  nginx:alpine

# 步骤5：进入代理容器修改配置（简单演示）
 docker exec -it nginx-proxy sh -c \
  "echo 'server { listen 80; location / { proxy_pass http://nginx-web; } }' \
   > /etc/nginx/conf.d/default.conf && nginx -s reload"

# 步骤6：验证访问
 curl http://localhost:8080

# 扩展：验证网络隔离
# 启动一个不在 frontend-network 中的容器
 docker run -d --name nginx-isolated nginx:alpine
# 这个容器无法通过名字访问 nginx-web 和 nginx-api
本讲核心回顾

1. 网络命名空间 + veth + 网桥 + NAT 是Docker网络的四大基石

2. 默认bridge适合简单测试，自定义bridge支持DNS，适合生产部署

3. Host模式性能最高但无隔离，None模式完全断网，Container模式适合Sidecar

4. 端口映射本质是DNAT，让外部可以访问容器内的服务

课外扩展：体验原生的 Docker 网络

本讲义的 Part 1 已为 WSL2 和 Linux 原生环境分别提供了对应命令。WSL2 因 Hyper-V 虚拟网络架构的限制，部分实验（尤其是 Part 1.4 的 NAT 出海测试）无法完整复现。

如果你想体验 100% 完整的 Docker 网络，建议：

1. 在云服务器（如阿里云、腾讯云的 Ubuntu 系统）上部署纯净的 Linux 环境

2. 使用 VMware/VirtualBox 本地安装 Linux 虚拟机

在这些原生环境中，Part 1 的所有命令均可端到端完整复现，网络流转效果与讲义完全一致。

实验环境清理命令

下次上课重新演示前，请先执行以下命令清理环境，避免名称冲突。

Linux 网络命名空间清理 Bash 
# ===== Linux 命名空间与网桥清理 =====

# 1. 显式删除（推荐，最安全）
 sudo ip netns del ns-4 2>/dev/null
 sudo ip netns del ns-5 2>/dev/null
 sudo ip link del br-2 2>/dev/null

# 2. 一键暴力清理（处理命名空间）
 sudo ip netns list 2>/dev/null | awk '{print $1}' | xargs -I {} sudo ip netns del {} 2>/dev/null

# 3. 一键暴力清理（处理所有网桥，修复了冒号问题）
 ip link show type bridge | awk -F': ' '{print $2}' | xargs -I {} sudo ip link del {} 2>/dev/null

# 4. 清理防火墙残留（关键！否则 iptables 规则会堆积）
 sudo iptables -t nat -F POSTROUTING 2>/dev/null

# 验证清理完成
 sudo ip netns list
 ip link show type bridge
 ip addr show
 iptables -t nat -L POSTROUTING -n -v
Docker 资源清理 Bash 
# ===== Docker 资源清理 =====

# 强力清除所有容器（-f 强制停止并删除）

  nginx-web nginx-api \
