预备

本讲需要的预备知识

1指标是什么

指标（Metric）就是一个随时间变化的数字，比如"当前CPU使用率72%"。监控系统每隔几秒采集一次这些数字，连起来就能画出曲线图，反映系统的健康趋势。

2时序数据库是什么

时序数据库（TSDB）专门用来存储"带时间戳的数字"。普通数据库存的是"张三的年龄是25"，时序数据库存的是"10:00:01 CPU=72%，10:00:16 CPU=68%"，按时间排列，便于查询某段时间的变化。

3/metrics端点是什么

很多程序会暴露一个 /metrics 的HTTP网址，访问它就能拿到一大段当前的指标文本。Prometheus就是定时访问这个网址来采集数据。

本讲涉及的四个工具，先混个脸熟

- docker stats：Docker自带的命令，一行就能看到所有容器的实时资源占用，是最轻量的"监控"。

- Portainer：图形化的Docker管理面板，把命令行操作变成点鼠标。

- cAdvisor + Prometheus：cAdvisor负责"采集"容器指标，Prometheus负责"存储"这些指标并提供查询。

- Grafana：把Prometheus里的数据画成漂亮的仪表盘，还能在指标异常时告警。

记住这条数据流：容器 → cAdvisor采集 → Prometheus存储 → Grafana展示，本讲所有内容都围绕它展开。

3 分钟快速演示：最简单的监控

在搭建复杂的监控栈之前，先用一条命令体验"监控"到底是什么。Docker自带的 docker stats 就是一个迷你监控系统。

Step 1：启动几个容器制造负载 Bash 
# 启动一个Nginx和一个Redis，作为被监控的对象
$ docker run -d --name demo-nginx nginx:alpine
$ docker run -d --name demo-redis redis:alpine
Step 2：实时观察它们的资源占用 Bash 
# docker stats 会持续刷新，像任务管理器一样
$ docker stats

# 预期输出（每秒刷新）：
# CONTAINER ID   NAME         CPU %   MEM USAGE / LIMIT     MEM %    NET I/O
# a1b2c3...      demo-nginx   0.00%   3.5MiB / 1.9GiB       0.18%    1.2kB / 0B
# d4e5f6...      demo-redis   0.15%   8.1MiB / 1.9GiB       0.42%    900B / 0B
# 按 Ctrl+C 退出
等等！docker stats 够用吗？

docker stats 只能看此刻的数据，关掉终端就没了——它不保存历史。

如果想知道"昨天凌晨3点CPU为什么飙到100%"，docker stats 完全无能为力，因为它没有存储任何历史数据。

而且容器一多，几十行数字滚动刷新，根本看不过来，更别说在异常时自动告警了。

带着"如何保存历史、如何可视化、如何告警"这三个问题，我们进入本讲的核心内容。

Part 1

监控概述

理解容器监控的价值、指标与架构

1.1

为什么需要监控？

容器监控定义
容器监控是对容器运行状态的持续观察、记录和告警。通过周期性采集CPU、内存、网络、磁盘等指标并长期存储，可以及时发现容器故障、性能瓶颈、资源不足等问题，把"出了事再救火"的被动运维，变成"提前预警"的主动运维。

1故障预警

提前发现资源不足、进程异常等问题，在用户投诉之前就处理掉

2性能优化

了解资源使用情况，合理分配CPU和内存，避免浪费或过载

3容量规划

根据历史数据预测增长趋势，规划何时扩容、何时缩容

4故障排查

通过历史曲线还原故障现场，回答"昨晚3点到底发生了什么"

没有监控会发生什么？真实场景

场景1：半夜崩溃无人知

凌晨2点某个容器内存泄漏被系统杀死，服务中断。没有监控告警，直到早上8点用户投诉才发现，已经损失了6小时的业务。

场景2：找不到性能瓶颈

网站变慢，到底是数据库慢、缓存慢还是网络慢？没有监控只能逐个猜测、逐个重启，耗费大量时间。

场景3：盲目扩容浪费成本

担心资源不够，买了10台服务器，实际只用了2台的量。有了监控数据，才能精准评估真实需求。

1.2

关键监控指标

容器关键指标
容器监控主要关注四类指标：CPU使用率、内存使用率、网络流量、磁盘I/O。这四类指标合称"USE/RED"中的资源类指标，反映了容器的运行状态和性能表现，是判断容器是否健康的第一手依据。
指标类型 说明 过高可能原因 常见后果
CPU使用率 容器使用的CPU资源占比 死循环、计算密集型任务 响应变慢、请求堆积
内存使用率 容器使用的内存大小 内存泄漏、缓存配置不当 触发OOM，容器被杀死
网络流量 入站/出站带宽使用 异常流量、爬虫、攻击 带宽打满、服务不可达
磁盘I/O 读写速度和IOPS 日志过多、频繁写入 磁盘写满、读写阻塞
docker stats查看实时指标 Bash 
# 查看所有容器实时资源使用（持续刷新）
$ docker stats

# 预期输出：
# CONTAINER ID   NAME            CPU %   MEM USAGE / LIMIT     MEM %   NET I/O
# abc123...      nginx-web       0.32%   50.5MiB / 1.952GiB   2.53%   1.5MB / 2.3MB
# def456...      mysql-db        1.28%   320MiB / 1.952GiB    16.39%  5.2MB / 8.1MB

# 只查看特定容器
$ docker stats nginx-web mysql-db

# 只采集一次后退出（适合写脚本，不会持续刷新）
$ docker stats --no-stream

# 自定义输出格式，只看名字和内存
$ docker stats --no-stream --format "table {{.Name}}\t{{.MemUsage}}"
指标要看"率"还是"量"？

内存通常看"量"（用了多少MB，距离上限还有多少），因为内存有硬上限，超了就被杀。

CPU通常看"率"（占了百分之多少），因为CPU可以"挤一挤"，短时间100%不一定有问题，但持续100%就要警惕了。

这就是为什么监控既要看瞬时值，更要看一段时间的趋势曲线——而趋势曲线正是 docker stats 给不了的。

指标的两种典型形态：Gauge 与 Counter

理解这两种形态，对后面写PromQL查询至关重要：

- Gauge（瞬时值）：可上可下的数字，如"当前内存使用512MB""当前CPU 72%"。直接读数就有意义。

- Counter（累计值）：只增不减的数字，如"开机至今总共收到1亿个网络包"。单看没意义，必须用 rate() 算"每秒增加多少"才有意义。

容器的网络流量、磁盘读写、CPU时间，在Prometheus里大多是Counter，所以你会看到大量 rate(xxx_total[5m]) 的写法——把"累计值"变成"速率"。

1.3

监控架构与Pull/Push模式

现代监控系统的三层架构
一套完整的监控系统通常分为三层：采集层（采集器/Exporter，负责从被监控对象读取指标）、存储层（时序数据库，负责长期保存指标）、展示层（可视化与告警，负责画图和报警）。本讲的cAdvisor、Prometheus、Grafana正好对应这三层。
本讲监控栈的数据流 架构图 
  ┌──────────────┐   被监控对象
  │  Docker容器   │   nginx、mysql、redis……
  └──────┬───────┘
         │ 暴露指标
         v
  ┌──────────────┐   采集层（Exporter）
  │   cAdvisor    │   读取每个容器的CPU/内存/网络/磁盘
  │ /metrics 端点 │   把指标整理成 /metrics 文本
  └──────┬───────┘
         │ Prometheus 定时来"抓"（Pull）
         v
  ┌──────────────┐   存储层（时序数据库）
  │  Prometheus   │   每15秒抓一次，存进TSDB
  │     TSDB      │   支持用 PromQL 查询历史
  └──────┬───────┘
         │ Grafana 来"查"
         v
  ┌──────────────┐   展示层
  │   Grafana     │   画成仪表盘曲线图
  │  Dashboard    │   指标超阈值时触发告警
  └──────────────┘
         │
         v
   运维人员看大屏 / 收到告警通知
Pull vs Push 模式
Pull（拉取）：监控服务端主动定时去访问每个目标的 /metrics 端点抓取数据，Prometheus采用这种模式。Push（推送）：被监控方主动把数据发送给监控服务端，如InfluxDB+Telegraph的经典组合。
为什么Prometheus选择Pull模式？

1. 易发现故障：抓不到目标就知道它挂了，Push模式下"没收到数据"难以区分是宕机还是网络问题

2. 集中配置：要监控谁、多久抓一次，全在Prometheus一处配置，不用改每个被监控方

3. 天然限流：抓取频率由服务端控制，不会被大量客户端同时推送压垮

4. 便于调试：你可以用浏览器直接打开目标的 /metrics 网址，亲眼看到它暴露了什么指标

Part 2

Portainer可视化

图形化容器管理工具

2.1

Portainer概述

Portainer定义
Portainer是一个轻量级的Docker图形化管理工具，通过Web界面管理容器、镜像、网络、数据卷等资源。它把原本需要敲命令行的操作变成了点鼠标，特别适合中小规模Docker环境的日常管理，以及不熟悉Linux命令的团队成员。

前面几讲我们一直在用 docker run、docker ps、docker network 这些命令管理Docker。这些命令很强大，但也有门槛：参数多、容易打错、看不到全局。Portainer是Docker的图形化"控制面板"，把这些功能搬到了网页上——你可以在浏览器里点几下，就完成启动容器、查看日志、进入终端等操作。

1容器管理

创建、启动、停止、删除容器，查看日志，进入终端

2镜像管理

查看本地镜像、拉取新镜像、删除无用镜像

3网络管理

查看网络拓扑、创建自定义网络、查看连接的容器

4数据卷管理

查看挂载点、创建和删除数据卷

Portainer适合谁用？

适合：

- 不熟悉命令行的运维新手、前端/产品同学，需要快速查看容器状态

- 中小团队的单机或少量节点环境，需要一个统一的管理入口

- 临时排查问题时，快速翻看日志、进入容器终端

不太适合：

- 大规模集群（应使用Kubernetes Dashboard、Rancher等）

- 需要版本化、可复现的部署（应使用Docker Compose或CI/CD）

- 自动化场景（图形界面无法写进脚本，仍需命令行）

2.2

Portainer部署与访问

快速部署Portainer CE（社区版） Bash 
# 1. 创建Portainer数据卷（保存账号、配置等）
$ docker volume create portainer_data

# 2. 启动Portainer容器
$ docker run -d \
  --name portainer \
  --restart=always \
  -p 9443:9443 \
  -p 9000:9000 \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -v portainer_data:/data \
  portainer/portainer-ce:latest

# 参数说明：
# -p 9443:9443   HTTPS管理界面端口（新版默认，推荐）
# -p 9000:9000   HTTP管理界面端口（兼容旧版）
# -v /var/run/docker.sock   Docker socket（核心配置，Portainer靠它操作Docker）
# -v portainer_data         持久化数据，避免重启后账号丢失
# --restart=always          开机/异常退出自动重启
镜像名称说明

请使用 portainer/portainer-ce（Community Edition，社区版，免费）。旧教程里的 portainer/portainer 镜像早已废弃停止维护，不要再用。企业版镜像是 portainer/portainer-ee（需付费授权）。

访问Portainer Bash 
# 在浏览器打开（HTTPS，浏览器可能提示证书不受信任，点继续即可）
https://localhost:9443

# 或使用HTTP端口
http://localhost:9000

# 首次访问的初始化流程：
# 1. 设置管理员账号和密码（密码至少12位，请记牢）
# 2. 选择"Get Started"，管理本地Docker环境（Local）
# 3. 进入Dashboard，即可看到本机所有容器、镜像、网络、卷
⚠️ 安全提示：挂载docker.sock意味着什么

挂载 /var/run/docker.sock 后，Portainer拥有对宿主机Docker的完全控制权，等同于root权限。任何能登录Portainer的人，都能创建特权容器、读写宿主机文件。

因此：① 务必设置强密码；② 生产环境不要把9000/9443端口直接暴露到公网；③ 首次部署后若长时间未初始化，Portainer会出于安全自动锁定，需重启容器才能再次初始化。

注意事项

- 如果首次访问超时报"timed out for security purposes"，说明启动后超过几分钟才打开，重启容器 docker restart portainer 后立即访问即可

- WSL2/Docker Desktop环境下，直接用Windows浏览器访问 https://localhost:9443 通常可行

- 生产环境建议配置正式的HTTPS证书和反向代理

Part 3

Prometheus监控

采集、存储与查询时序指标

3.1

Prometheus概述

Prometheus定义
Prometheus是一个开源的监控和告警系统，最初由SoundCloud开发，现已成为CNCF（云原生计算基金会）继Kubernetes之后的第二个"毕业项目"，是云原生监控领域事实上的标准。它通过HTTP协议定期采集指标，存入自带的时序数据库，并提供强大的PromQL查询语言。

Prometheus的工作方式是：它拿着一份目标清单，每隔固定时间（默认15秒）就去访问每个目标的 /metrics 端点采集一次，把采集到的数据按时间顺序存进数据库。需要看数据时，用PromQL查询；指标超过阈值时，触发告警。

1多维度数据模型

指标由"指标名+标签集"定义，如 cpu{name="nginx"}，便于按维度切片

2PromQL查询语言

强大的查询和聚合能力，支持速率、求和、分位数等运算

3Pull拉取模式

服务端主动抓取，目标宕机一眼可见，配置集中

4内置告警

基于PromQL写告警规则，配合Alertmanager发通知

Prometheus自己不采集容器指标？

这是初学者最容易困惑的点：Prometheus本身并不知道怎么读取Docker容器的CPU、内存。它只会做一件事——访问 /metrics 网址抓文本。

所以需要一个专门的采集组件，把容器的资源数据转换成Prometheus能识别的 /metrics 格式。这个组件就是下一页要讲的 cAdvisor。同理，监控主机本身（CPU、磁盘）需要 Node Exporter，监控MySQL需要 mysqld-exporter……这些采集组件统称为 Exporter（导出器）。

指标名 + 标签：Prometheus的多维数据模型

一条指标由"指标名 + 一组标签"唯一确定：

- 指标名：表示测量的对象，如 container_memory_usage_bytes（容器内存使用量）

- 标签（label）：附加的维度属性，如 {name="nginx", image="nginx:alpine"}

有了标签，你就能精确查询："只看名字叫nginx的容器的内存"，或"按镜像分组统计内存总和"。这种"一个指标名+任意多个标签"的设计，让查询变得极其灵活。

3.2

cAdvisor监控容器

cAdvisor定义
cAdvisor（Container Advisor，容器顾问）是Google开发的容器指标采集工具，专门用于读取本机所有容器的资源使用信息。它原生支持Docker，启动后会自动发现所有容器，并把每个容器的CPU、内存、网络、磁盘指标，整理成Prometheus能抓取的 /metrics 格式。它就是Part 3.1所说的针对容器的Exporter。

cAdvisor需要读取宿主机的系统目录才能拿到容器数据，所以启动时要挂载一堆只读目录（:ro 表示read-only只读，保证cAdvisor不会改坏系统文件）。这些挂载是固定套路，照抄即可。

部署cAdvisor Bash 
# 启动cAdvisor（注意镜像地址已迁移到 gcr.io/cadvisor）
$ docker run -d \
  --name cadvisor \
  --restart=always \
  -p 8080:8080 \
  --volume=/:/rootfs:ro \
  --volume=/var/run:/var/run:ro \
  --volume=/sys:/sys:ro \
  --volume=/var/lib/docker/:/var/lib/docker:ro \
  --volume=/dev/disk/:/dev/disk:ro \
  --privileged \
  gcr.io/cadvisor/cadvisor:latest

# 参数说明：
# -p 8080:8080       Web界面与/metrics端点
# /rootfs            宿主机根文件系统（读取容器信息）
# /var/run           Docker运行时socket
# /sys               系统cgroup信息（资源统计的来源）
# /var/lib/docker    Docker数据目录
# --privileged       某些系统需要特权模式才能读取全部指标
镜像地址说明

老教程里的 google/cadvisor 镜像（Docker Hub）已经停止更新多年。官方现在维护的是 gcr.io/cadvisor/cadvisor。如果国内拉取 gcr.io 困难，可使用镜像加速器，或改用社区维护的兼容镜像。

prom/prometheus、grafana/grafana、cadvisor 这几个镜像都有数百MB，首次拉取在国内网络下通常要等 3–5 分钟，请耐心等待，不要以为是卡住了。

访问cAdvisor并查看原始指标 Bash 
# 1. Web界面：直接看到本机所有容器的实时图表
http://localhost:8080

# 2. 指标端点：Prometheus就是来抓这个网址（重点！）
http://localhost:8080/metrics

# 在终端抓一次看看长什么样
$ curl -s http://localhost:8080/metrics | grep container_memory_usage_bytes | head -3
# 输出类似（每行就是一个带标签的指标）：
# container_memory_usage_bytes{name="cadvisor",image="..."} 4.2e+07
# container_memory_usage_bytes{name="nginx",image="nginx:alpine"} 3.5e+06
# 看懂这行：指标名{标签} 数值 —— 这就是Prometheus的数据格式
动手理解 /metrics 端点

建议亲自 curl 一下 /metrics 端点。你会看到成百上千行 指标名{标签} 数值 的文本。Prometheus要做的，就是定时抓走这一大段文本，按时间存起来。理解了这一步，整个监控链路就通了。

实际每行的标签会比上面示例多很多——除了 name、image，还会带 id、container_label_com_docker_compose_* 等一长串，看起来比书里冗长是正常的。查询时用 {name!=""} 这类标签过滤照样有效，不影响使用。

cAdvisor暴露的常用指标

- container_cpu_usage_seconds_total：容器累计占用的CPU秒数（Counter，需配合rate）

- container_memory_usage_bytes：容器当前内存使用字节数（Gauge，直接读）

- container_network_receive_bytes_total：累计接收的网络字节数（Counter）

- container_network_transmit_bytes_total：累计发送的网络字节数（Counter）

- container_fs_usage_bytes：容器文件系统使用量（Gauge）

注意带 _total 后缀的几乎都是Counter（只增不减），查询时要套 rate()。

3.3

Node Exporter监控宿主机

Node Exporter定义
Node Exporter是Prometheus官方提供的主机指标采集器，负责采集宿主机本身的CPU、内存、磁盘、文件系统、网络等系统级指标。cAdvisor关注的是"每个容器"，Node Exporter关注的是"整台机器"，两者互补。

为什么有了cAdvisor还需要Node Exporter？因为容器跑在宿主机上，如果宿主机本身磁盘满了、内存耗尽，所有容器都会遭殃。只看容器指标，可能容器都"正常"，但整机已经濒临崩溃。所以生产环境通常两者都部署。

部署Node Exporter Bash 
# 启动Node Exporter，采集宿主机系统指标
$ docker run -d \
  --name node-exporter \
  --restart=always \
  -p 9100:9100 \
  --net=host \
  --pid=host \
  -v /:/host:ro,rslave \
  quay.io/prometheus/node-exporter:latest \
  --path.rootfs=/host

# 参数说明：
# -p 9100:9100      指标端点端口
# --net=host        使用宿主机网络，才能采到真实网络指标
# --pid=host        共享宿主机进程空间
# -v /:/host:ro     只读挂载宿主机根目录
# --path.rootfs     告诉它去/host读取宿主机文件系统
关于 -p 9100:9100 与 --net=host 同时出现

用了 --net=host 后容器直接共享宿主机网络，此时 -p 9100:9100 实际不生效——Docker会提示 Published ports are discarded when using host network mode。9100端口由宿主机网络直接暴露，浏览器照样能访问 localhost:9100/metrics。这里保留 -p 只作端口的文档标注，看到这条警告属于正常现象，不是报错。

验证Node Exporter Bash 
# 查看主机指标端点
http://localhost:9100/metrics

# 抓几个典型的主机指标看看
$ curl -s http://localhost:9100/metrics | grep -E "node_memory_MemAvailable|node_filesystem_avail" | head -3
# node_memory_MemAvailable_bytes 2.1e+09   ← 可用内存
# node_filesystem_avail_bytes{...} 5.0e+10  ← 磁盘可用空间
三个采集器的分工对比

- docker stats：临时看看，不存历史，不暴露/metrics

- cAdvisor：采集"每个容器"的指标，端口8080，指标前缀 container_

- Node Exporter：采集"整台宿主机"的指标，端口9100，指标前缀 node_

它们都把指标暴露成 /metrics，然后统一交给Prometheus抓取。本讲实验为了简化，主要用cAdvisor；理解了它，Node Exporter同理。

还有哪些常见的Exporter？

Exporter生态非常丰富，几乎每种主流软件都有对应的Exporter：

- mysqld-exporter：监控MySQL（连接数、慢查询、QPS）

- redis-exporter：监控Redis（命中率、内存、连接数）

- nginx-prometheus-exporter：监控Nginx（请求数、活跃连接）

- blackbox-exporter：从外部探测网站是否存活、响应时间

需要监控什么，就找对应的Exporter，配置Prometheus去抓它的 /metrics 即可。

3.4

部署Prometheus

prometheus.yml的作用
prometheus.yml 是Prometheus的核心配置文件，它告诉Prometheus"该去抓哪些目标、多久抓一次"。其中 scrape_configs 就是那份采集目标清单，每个 job 对应一类被监控目标。
prometheus.yml配置文件（逐行注释） YAML 
# 全局配置
global:
  scrape_interval: 15s        # 每15秒抓取一次所有目标
  evaluation_interval: 15s    # 每15秒评估一次告警规则

# 抓取目标清单（核心！）
scrape_configs:
  # job 1：监控Prometheus自己
  - job_name: 'prometheus'
    static_configs:
      - targets: ['localhost:9090']

  # job 2：抓取cAdvisor（容器指标）
  - job_name: 'cadvisor'
    static_configs:
      # 注意：用服务名cadvisor，因为它们在同一个Docker网络里
      - targets: ['cadvisor:8080']

  # job 3：抓取Node Exporter（主机指标）
  - job_name: 'node'
    static_configs:
      - targets: ['node-exporter:9100']
关键：targets为什么写服务名而不是localhost？

这正是第九讲"自定义网络DNS服务发现"的应用！Prometheus、cAdvisor、Node Exporter必须在同一个Docker自定义网络里，这样Prometheus才能用 cadvisor:8080（服务名）访问到cAdvisor。如果写 localhost:8080，Prometheus只会去找自己容器内的8080端口，根本抓不到cAdvisor。这是新手最常踩的坑。

启动Prometheus Bash 
# 创建配置目录并写入prometheus.yml
$ mkdir -p ~/monitoring && cd ~/monitoring
$ cat > prometheus.yml << 'EOF'
global:
  scrape_interval: 15s
scrape_configs:
  - job_name: 'prometheus'
    static_configs:
      - targets: ['localhost:9090']
  - job_name: 'cadvisor'
    static_configs:
      - targets: ['cadvisor:8080']
EOF

# 启动Prometheus（生产用Compose统一管理，见Part 5）
$ docker run -d \
  --name prometheus \
  --restart=always \
  -p 9090:9090 \
  -v ~/monitoring/prometheus.yml:/etc/prometheus/prometheus.yml \
  prom/prometheus:latest
验证Prometheus抓取状态 Bash 
# Web界面
http://localhost:9090

# 最重要的一步：检查抓取目标是否健康
# 点击顶部菜单 Status -> Targets
# 每个target应显示绿色的 "UP"
#   UP   = 抓取成功
#   DOWN = 抓不到（检查网络、服务名、端口）

# 也可以在 Graph 页面直接输入指标名测试，如：
#   container_memory_usage_bytes
# 点 Execute，能出数据就说明抓取成功
target刚启动时显示 UNKNOWN 是正常的

Prometheus刚启动后，Targets页里 cadvisor 可能先显示 UNKNOWN，大约15–30秒后才变成绿色 UP。这是因为 scrape_interval 是15s、首次抓取本身有延迟，并非故障。稍等一两个抓取周期再刷新即可，先别急着排查。

排查target显示DOWN的思路

如果 Status → Targets 里某个目标是红色DOWN：

1. 检查是否同网络：Prometheus和目标必须在同一Docker网络

2. 检查服务名拼写：targets里的名字要和容器/服务名完全一致

3. 检查端口：cAdvisor是8080，Node Exporter是9100，别写错

4. 手动测试连通性：用一次性curl容器接入同一网络验证 docker run --rm --network <监控网络名> curlimages/curl -s http://cadvisor:8080/metrics | head。注意 prom/prometheus 镜像内不含 wget/curl，所以不在Prometheus容器里测

3.5

PromQL查询入门

PromQL（Prometheus Query Language）
PromQL是Prometheus专用的查询语言，用来从海量时序数据中筛选、计算、聚合出你想要的结果。Grafana仪表盘上的每一条曲线，背后都是一句PromQL。掌握几个基本套路，就能应付大部分监控需求。
PromQL四个基本套路 PromQL 
# 套路1：直接查指标（Gauge，瞬时值）
# 查所有容器的当前内存使用
container_memory_usage_bytes

# 套路2：用标签过滤（大括号里写条件）
# 只看名字叫nginx的容器内存（name!="" 过滤掉无名的系统容器）
container_memory_usage_bytes{name="nginx"}

# 套路3：rate() 把Counter变速率（最常用！）
# 计算每个容器最近5分钟的CPU使用速率
rate(container_cpu_usage_seconds_total{name!=""}[5m])

# 套路4：sum()/avg() 聚合
# 按容器名分组，求每个容器的网络接收总速率
sum(rate(container_network_receive_bytes_total[5m])) by (name)
为什么CPU查询要用rate()？

container_cpu_usage_seconds_total 是个只增不减的Counter——它记录的是"容器从启动到现在，累计用了多少秒CPU时间"。

直接看这个数字毫无意义（永远在涨）。我们真正关心的是"最近这段时间，CPU用得快不快"。

rate(...[5m]) 的含义就是："看最近5分钟，这个累计值平均每秒增加多少"——增加得快，说明CPU正忙。这就把"累计值"变成了"使用率"。所有带 _total 的指标，几乎都要这样处理。

常用容器监控PromQL速查 PromQL 
# 容器CPU使用率（百分比，乘100）
rate(container_cpu_usage_seconds_total{name!=""}[5m]) * 100

# 容器内存使用量（MB）
container_memory_usage_bytes{name!=""} / 1024 / 1024

# 容器内存使用率（占limit的百分比）
container_memory_usage_bytes / container_spec_memory_limit_bytes * 100

# 容器网络接收速率（字节/秒）
rate(container_network_receive_bytes_total{name!=""}[5m])

# 容器网络发送速率（字节/秒）
rate(container_network_transmit_bytes_total{name!=""}[5m])

# 当前运行的容器数量（统计有名字的容器）
count(container_last_seen{name!=""})
先在Prometheus里试，再搬到Grafana

建议先在Prometheus的 Graph 页面调试PromQL：输入查询、点Execute、切到Graph标签页看曲线。调通了再把这句PromQL粘到Grafana的面板里。这样能快速定位"是查询写错了，还是Grafana配置错了"。

Part 4

Grafana可视化

仪表盘、查询与告警

4.1

Grafana概述

Grafana定义
Grafana是一个开源的度量分析与可视化平台。它本身不存储数据，而是从各种数据源（Prometheus、InfluxDB、Elasticsearch、MySQL等）读取数据，画成漂亮的仪表盘，并能在指标异常时告警。它是整套监控栈的可视化展示层。

前面我们已经能在Prometheus里用PromQL查出数据了，但Prometheus自带的图表很简陋，只能临时看看。Grafana则专注于"把数据画好看、组织成仪表盘、支持告警"。可以说：Prometheus负责"存得住、查得到"，Grafana负责"看得爽、报得准"。

1多数据源支持

Prometheus、InfluxDB、Elasticsearch、MySQL等几十种

2丰富的图表

折线图、仪表盘、热力图、状态灯、表格等

3告警功能

支持邮件、钉钉、Slack、Webhook等多种通知

4仪表盘市场

社区有上万个现成仪表盘，导入ID即可用

为什么不直接用Prometheus看，非要加Grafana？

1. 仪表盘持久化：Prometheus的图是临时查询，Grafana可以把一组图保存成仪表盘，下次直接打开

2. 多源聚合：一个Grafana面板可以同时展示Prometheus的指标、MySQL的业务数据、日志统计

3. 美观与交互：支持时间范围选择、变量下拉、钻取，演示和值班都更方便

4. 更强的告警：可视化配置告警规则，对接钉钉/企业微信等国内常用通知渠道

连接失败（数据源报错）怎么办？

点 Save & test 后如果报红：

1. 确认同网络：Grafana和Prometheus是否在同一Docker网络（用Compose部署默认同网络，最省心）

2. 确认URL：是 http://prometheus:9090 而非 localhost

3. 确认Prometheus活着：浏览器先打开 http://localhost:9090 确认它本身正常

#   1860   - Node Exporter Full（最全的主机监控）

# 导入后立刻就有一整面墙的精美图表，无需手动配置！
导入后图表空白怎么办？

通常是因为：① 选错了数据源（导入时要选我们配的Prometheus）；② 仪表盘用的指标名和你的cAdvisor版本对不上（不同版本指标名略有差异）。可以编辑某个空面板，把PromQL换成3.5节验证过能出数据的查询。

仪表盘组织的最佳实践

1. 按角色分盘：主机概览一个盘、容器明细一个盘、业务指标一个盘

2. 善用变量：用模板变量做一个"容器名"下拉框，切换即可看不同容器，不用重复建图

3. 统一时间范围：仪表盘右上角统一控制时间窗口，所有面板联动

4. 导出备份：把调好的仪表盘导出成JSON存进Git，换环境直接导入

4.4

告警配置

告警的意义
仪表盘再漂亮，也得有人盯着看才有用。告警（Alerting）让系统在指标超过阈值时主动通知你（邮件、钉钉、企业微信），实现"无人值守"——半夜出问题也能第一时间收到消息，而不是等用户投诉。

Grafana提供了可视化的告警配置（Grafana Alerting），适合新手；生产环境也常用Prometheus原生的告警规则配合Alertmanager。本讲以Grafana Alerting为例，因为它和仪表盘结合得最直观。

配置一条告警规则（操作步骤） 操作步骤 
# 目标：当某容器内存使用超过500MB时告警

# 1. 左侧 Alerting -> Alert rules -> New alert rule
# 2. 填写规则名，如 "容器内存过高"
# 3. 定义查询（PromQL）：
#    container_memory_usage_bytes{name!=""} / 1024 / 1024
# 4. 设置条件（Expression）：
#    WHEN last() OF query IS ABOVE 500
#    （当查询值高于500时触发）
# 5. 设置评估周期：每1分钟检查一次，持续5分钟超标才告警
#    （避免瞬时尖峰造成误报）
# 6. 选择通知渠道（Contact point）
# 7. 保存规则
配置通知渠道（以钉钉/Webhook为例） 操作步骤 
# 1. 左侧 Alerting -> Contact points -> Add contact point
# 2. 选择类型：
#    - Email：填写SMTP和收件邮箱
#    - Webhook：填写钉钉/企业微信机器人的Webhook地址
#    - 其他：Slack、Telegram、PagerDuty等
# 3. 点 "Test" 发送一条测试消息，确认能收到
# 4. 保存，回到告警规则里选用这个通知渠道
告警的三个关键参数

阈值（threshold）：触发告警的临界值，指标低于它视为正常，高于它则告警。

持续时间（for）：指标需连续超过阈值一段时间（如5分钟）才真正告警，用来过滤掉瞬时尖峰造成的误报。

通知渠道：告警触发后通过哪种方式通知（邮件、钉钉、企业微信、电话等），可按严重程度（warning / critical）分级到不同渠道。

设阈值的关键在于平衡：太敏感会频繁误报、让人麻木，太迟钝会漏报、出事才响。需要结合历史数据反复调整。

告警配置的常见误区

1. 没设持续时间：瞬时尖峰就告警，半夜被无意义的告警吵醒

2. 阈值拍脑袋：应基于历史曲线设置，而非凭感觉填个数

3. 告警风暴：一个根因引发几十条告警，应做告警聚合/抑制

4. 只告警不分级：所有告警一个级别，分不清哪个要立刻处理。应区分 warning / critical

5.1

常见问题排查指南

故障排查思路
监控栈出问题，按数据流顺序排查最有效："容器在跑吗 → cAdvisor能采到吗 → Prometheus抓到了吗 → Grafana读到了吗"。沿着数据流一段段检查，很快就能定位断点。
问题1：Prometheus里target显示DOWN

现象：Status→Targets里cadvisor是红色DOWN。

原因：最常见是没在同一网络，或服务名/端口写错。

解决：

1. 确认三个服务都在 monitor-net（用Compose默认满足）

2. 确认prometheus.yml里写的是 cadvisor:8080 而非localhost

3. 测试网络连通：docker run --rm --network <监控网络名> curlimages/curl -s http://cadvisor:8080/metrics | head（用一次性curl容器，因为 prom/prometheus 镜像不含 wget/curl）

问题2：Grafana数据源测试失败

现象：Save & test 报红，提示无法连接。

原因：URL填成了localhost，或Grafana与Prometheus不同网络。

解决：URL改为 http://prometheus:9090；确认同网络；先用浏览器确认 http://localhost:9090 本身可访问。

问题3：cAdvisor容器启动失败或采不到数据

现象：cadvisor容器反复重启，或/metrics数据残缺。

原因：用了废弃的 google/cadvisor 镜像，或缺少必要挂载/特权。

解决：

1. 改用 gcr.io/cadvisor/cadvisor:latest

2. 确认挂载了 /sys、/var/lib/docker 等只读目录

3. 部分系统需加 privileged: true

4. 查看日志定位：docker logs cadvisor

问题4：Grafana仪表盘有框但无数据

现象：导入的仪表盘面板都在，但全是 "No data"。

原因：导入时没选对数据源，或仪表盘的指标名与当前cAdvisor版本不一致。

解决：编辑面板，把PromQL换成在Prometheus里验证过能出数的查询（如 container_memory_usage_bytes{name!=""}）；导入时务必选中Prometheus数据源。

问题5：Portainer初始化超时被锁定

现象：首次打开Portainer提示 "timed out for security purposes"。

原因：容器启动后超过几分钟才首次访问，出于安全自动锁定。

解决：docker restart portainer 后立即（几分钟内）打开页面完成账号设置。

监控栈常用排查命令 Bash 
# 看三个服务状态
$ docker compose ps

# 看某个服务日志
$ docker compose logs -f prometheus

# 测试能否解析服务名并抓到cAdvisor（用一次性curl容器接入监控网络）
# 注意：prom/prometheus镜像内不含wget/curl，所以不在Prometheus容器里测
$ docker run --rm --network monitoring-stack_monitor-net \
    curlimages/curl -s http://cadvisor:8080/metrics | head

# 确认它们在同一网络、能解析服务名
$ docker network inspect monitoring-stack_monitor-net
总结

本讲核心要点回顾

核心知识点

1. 监控的价值在于故障预警、性能优化、容量规划、故障排查，关注CPU/内存/网络/磁盘四类指标

2. 监控数据流：容器 → cAdvisor采集 → Prometheus存储 → Grafana展示，这是本讲主线

3. Prometheus用Pull模式定时抓 /metrics，靠Exporter（cAdvisor/Node Exporter）采集并转换指标

4. PromQL查询：Gauge直接读，Counter套 rate()，再用 sum/avg ... by 聚合

5. Grafana不存数据，配数据源指向Prometheus，导入仪表盘+配置告警

6. Portainer提供图形化管理，但同样不存历史，与监控栈互补

贯穿全讲的一个关键点：服务名通信

本讲反复强调：Prometheus抓cAdvisor用 cadvisor:8080，Grafana连Prometheus用 prometheus:9090——都用服务名，不用localhost。这正是第九讲"自定义网络DNS服务发现"和第十讲"Compose同网络"的综合应用。理解了这一点，就理解了监控栈为什么必须放在同一个Docker网络里。

监控工具选型速查

- 只想临时看一眼 → docker stats

- 想要图形化管理容器 → Portainer

- 要长期存储+查询历史 → Prometheus + Exporter

- 要漂亮仪表盘+告警 → Grafana

- 大规模集群监控 → Prometheus Operator / Thanos / VictoriaMetrics（进阶）

课程收尾与进阶方向

  • 本讲是Docker系列的监控收官，结合前十讲已可独立完成"构建—编排—联网—监控"全流程
  • 进阶方向：Prometheus告警规则+Alertmanager、日志监控（Loki）、链路追踪（Tempo）
  • 集群方向：从Docker Compose迈向Kubernetes，监控栈演进为Prometheus Operator
清理

实验环境清理

为什么要清理？
监控栈中的Prometheus和Grafana会持续占用内存与磁盘（Prometheus还会不断写入时序数据）。完成实验后及时清理，可以释放资源、避免端口冲突（9090/3000/8080/9000等），保持环境整洁。
本讲实验端口分配表

- Portainer：https://localhost:9443 / http://localhost:9000

- cAdvisor：http://localhost:8080

- Node Exporter：http://localhost:9100

- Prometheus：http://localhost:9090

- Grafana：http://localhost:3000

提示：监控栈实验项目目录为 monitoring-stack。

清理Compose监控栈 Bash 
# 进入项目目录
$ cd ~/monitoring-stack

# 停止并删除容器和网络（保留数据卷）
$ docker compose down

# 连同数据卷一起删除（Grafana仪表盘、Prometheus历史数据都会清空）
$ docker compose down -v
清理单独启动的监控容器 Bash 
# 删除用docker run单独启动的监控相关容器
$ docker rm -f portainer cadvisor node-exporter prometheus grafana \
  demo-nginx demo-redis load-test 2>/dev/null

# 删除监控相关数据卷
$ docker volume rm portainer_data grafana_data 2>/dev/null

# 清理未使用的网络
$ docker network prune -f

# 验证清理结果
$ docker ps -a
$ docker volume ls
⚠️ 重要提示

docker compose down -v 会删除Grafana里你辛苦配置的仪表盘和Prometheus积累的历史数据。如果想保留，先用Grafana的导出功能把仪表盘存成JSON，再清理。

生产环境切勿随意执行 docker system prune -a --volumes，它会删除所有未使用资源，可能误删监控历史数据。
