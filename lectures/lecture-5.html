Part 1

容器基础概念

理解容器的本质和特点

Part 1.1

什么是Docker容器？

容器定义
Docker容器是镜像的运行实例，是一个轻量级、独立、可执行的软件包，包含运行应用所需的所有内容：代码、运行时、系统工具、系统库和配置。

容器与镜像的关系类似于对象与类的关系。镜像是静态的模板，容器是镜像的动态实例。容器在镜像的基础上添加了一个可写层，所有运行时的修改都发生在这个可写层中。

1轻量级

共享宿主机内核，启动时间秒级，资源占用少

2隔离性

通过Namespace和Cgroups实现进程、网络、文件系统等隔离

3可移植性

在任何Docker环境中行为一致，实现"一次构建，到处运行"

4可控制

可限制CPU、内存、IO等资源使用，确保系统稳定

容器 vs 虚拟机

容器：轻量级，启动快，资源占用少，共享内核

虚拟机：重量级，启动慢，资源占用多，完整操作系统

容器操作命令

掌握容器的创建、启动、停止等操作

Part 2.1

创建与启动容器

docker create - 创建容器（不启动） Bash 
# 语法：docker create [OPTIONS] IMAGE [COMMAND] [ARG...]
# 作用：创建一个新容器但不启动它，返回容器ID
#       适合需要先配置再启动的场景

# 基础用法 - 创建Ubuntu容器
$ docker create --name my-ubuntu ubuntu:22.04

# 预期输出：
# 容器ID（如：1a2b3c4d5e6f...）

# 参数详解：
# --name "容器名"      为容器指定名称，便于后续管理
#                      不指定则自动生成随机名称
# --env, -e "KEY=VAL"  设置环境变量，可多次使用
#                      示例：-e TZ=Asia/Shanghai
# --volume, -v "宿主路径:容器路径"  挂载数据卷
#                      示例：-v /data/app:/app
# --port, -p "宿主端口:容器端口"   端口映射
#                      示例：-p 8080:80
# --network "网络名"   将容器连接到指定网络
#                      示例：--network my-network
# --restart "策略"     容器退出后的重启策略
#                      no：不重启（默认）
#                      always：总是重启
#                      on-failure：失败时重启
#                      unless-stopped：除非手动停止否则重启

# 示例：创建带完整配置的Ubuntu容器
$ docker create \
  --name ubuntu-app \
  -e TZ=Asia/Shanghai \
  -e LANG=C.UTF-8 \
  -v /data/app:/app \
  --restart unless-stopped \
  ubuntu:22.04

# 预期输出：
# 容器ID

# 创建后需要手动启动
$ docker start ubuntu-app
docker run - 创建并启动容器（最常用） Bash 
# 语法：docker run [OPTIONS] IMAGE [COMMAND] [ARG...]
# 作用：创建容器并立即启动，相当于 create + start

# 基础用法 - 启动Ubuntu容器（交互式）
$ docker run -it --name my-ubuntu ubuntu:22.04 bash

# 预期输出：
# 进入Ubuntu容器的bash终端
root@1a2b3c4d5e6f:/#

# 参数详解：
# --detach, -d         后台运行模式，容器在后台运行并返回容器ID
#                      不加此参数则前台运行，占用当前终端
# --interactive, -i    保持STDIN打开，即使没有附加
#                      通常与-t配合使用（-it），用于交互式操作
# --tty, -t            分配一个伪终端（pseudo-TTY）
#                      与-i配合使用，提供完整的终端体验
# --rm                 容器退出后自动删除
#                      适合临时测试，不会留下停止的容器
# --name "名称"        指定容器名称，便于后续管理
# --env, -e "KEY=VAL"  设置环境变量
# --volume, -v         挂载数据卷
# --port, -p           端口映射（宿主端口:容器端口）
# --network "网络名"   指定网络
# --restart "策略"     重启策略（no/always/on-failure/unless-stopped）
# --cpus "数量"        限制CPU核心数
# --memory, -m "大小"  限制内存使用

# 示例：交互式运行Ubuntu容器（临时测试）
$ docker run -it --rm ubuntu:22.04 bash

# 预期输出：
# 进入Ubuntu容器的bash终端
root@1a2b3c4d5e6f:/# cat /etc/os-release
PRETTY_NAME="Ubuntu 22.04.3 LTS"
...
root@1a2b3c4d5e6f:/# exit
# 退出后容器自动删除

# 示例：后台运行Ubuntu容器
$ docker run -d --name ubuntu-server \
  -e TZ=Asia/Shanghai \
  --restart unless-stopped \
  ubuntu:22.04 sleep infinity

# 预期输出：
# 容器ID

# 示例：带资源限制运行
$ docker run -d --name ubuntu-limited \
  --cpus 0.5 \
  --memory 512m \
  --restart unless-stopped \
  ubuntu:22.04 sleep infinity
运行模式

前台运行：容器在当前终端运行，按Ctrl+C停止

后台运行：使用-d参数，容器在后台运行

交互式运行：使用-it参数，进入容器终端

查看容器

docker ps - 列出容器 Bash 
# 语法：docker ps [OPTIONS]
# 作用：列出容器信息，默认只显示运行中的容器

# 基础用法 - 列出运行中的容器
$ docker ps

# 预期输出：
CONTAINER ID   IMAGE          COMMAND           CREATED         STATUS         PORTS     NAMES
1a2b3c4d5e6f   ubuntu:22.04   "sleep infinity"  5 minutes ago   Up 5 minutes             my-ubuntu

# 参数详解：
# --all, -a            显示所有容器，包括已停止的
#                      默认只显示运行中的容器
# --quiet, -q          只显示容器ID，适合脚本处理
#                      常用于批量操作：docker rm $(docker ps -aq)
# --filter, -f "条件"  根据条件过滤容器
#                      示例：-f status=running（运行中）
#                            -f name=ubuntu（名称包含ubuntu）
#                            -f ancestor=ubuntu:22.04（基于ubuntu镜像）
# --format "模板"      使用Go模板格式化输出
#                      示例：--format "table {{.Names}}\t{{.Status}}"
# --last, -n "数量"    显示最近创建的n个容器（包括停止的）
# --no-trunc           不截断输出，显示完整信息
# -s, --size           显示容器大小

# 示例：显示所有容器
$ docker ps -a

# 预期输出：
CONTAINER ID   IMAGE          COMMAND           CREATED          STATUS                      PORTS     NAMES
1a2b3c4d5e6f   ubuntu:22.04   "sleep infinity"  5 minutes ago    Up 5 minutes                          my-ubuntu
2b3c4d5e6f7g   ubuntu:22.04   "bash"            10 minutes ago   Exited (0) 8 minutes ago              my-ubuntu-2

# 示例：只显示容器ID（常用于批量操作）
$ docker ps -q

# 预期输出：
1a2b3c4d5e6f

# 示例：显示最近创建的3个容器
$ docker ps -n 3

# 预期输出：
# 最近3个容器的信息

# 示例：过滤特定状态的容器
$ docker ps -f status=exited

# 示例：自定义格式输出
$ docker ps --format "table {{.Names}}\t{{.Status}}\t{{.Image}}"

Part 2.3

停止与删除容器

docker stop - 优雅停止容器 Bash 
# 语法：docker stop [OPTIONS] CONTAINER [CONTAINER...]
# 作用：发送SIGTERM信号，等待容器优雅停止
#       容器内的进程有时间清理资源、保存数据

# 基础用法 - 停止Ubuntu容器
$ docker stop my-ubuntu

# 预期输出：
my-ubuntu

# 参数详解：
# --time, -t "秒数"    停止前的等待时间，默认10秒
#                      Docker先发送SIGTERM，等待指定时间
#                      如果超时仍未停止，则发送SIGKILL强制停止
#                      建议给重要应用更长的停止时间

# 示例：停止多个容器
$ docker stop ubuntu-app1 ubuntu-app2

# 预期输出：
ubuntu-app1
ubuntu-app2

# 示例：给容器更长的停止时间（30秒）
$ docker stop -t 30 ubuntu-database

# 预期输出：
ubuntu-database

# 停止流程说明：
# 1. Docker发送SIGTERM信号给容器主进程
# 2. 主进程收到信号，开始清理工作（如保存数据、关闭连接）
# 3. 如果在-t时间内停止，容器正常退出
# 4. 如果超时，Docker发送SIGKILL强制终止
docker start - 启动已停止的容器 Bash 
# 语法：docker start [OPTIONS] CONTAINER [CONTAINER...]
# 作用：启动已停止的容器，恢复之前的运行状态

# 基础用法 - 启动容器
$ docker start my-ubuntu

# 预期输出：
my-ubuntu

# 参数详解：
# --attach, -a         附加到容器的标准输入/输出/错误
#                      启动后立即看到容器输出
# --interactive, -i    附加到容器的STDIN
#                      与-a配合使用，可以与容器交互
# --checkpoint "名称"  从检查点恢复（实验性功能）
# --checkpoint-dir     检查点目录（实验性功能）

# 示例：启动并附加到容器（查看输出）
$ docker start -a my-ubuntu

# 预期输出：
# 显示容器启动日志

# 示例：启动并进入交互模式
$ docker start -a -i my-ubuntu

# 预期输出：
# 进入容器终端
docker rm - 删除容器 Bash 
# 语法：docker rm [OPTIONS] CONTAINER [CONTAINER...]
# 作用：删除已停止的容器，释放资源

# 基础用法 - 删除容器
$ docker rm my-ubuntu

# 预期输出：
my-ubuntu

# 参数详解：
# --force, -f          强制删除，即使容器正在运行
#                      会先停止容器再删除，慎用
# --volumes, -v        删除容器关联的匿名卷
#                      注意：不会删除命名卷
# --link, -l           删除容器间的链接（已弃用）

# 示例：强制删除正在运行的容器
$ docker rm -f my-ubuntu

# 预期输出：
my-ubuntu

# 示例：删除多个容器
$ docker rm ubuntu-app1 ubuntu-app2

# 预期输出：
ubuntu-app1
ubuntu-app2

# 示例：删除所有停止的容器
$ docker rm $(docker ps -a -q)

# 预期输出：
# 所有停止容器的ID
注意事项

- 删除容器前需先停止容器（除非使用-f参数）

- 使用-f参数强制删除可能导致数据丢失

- 删除容器不会自动删除关联的数据卷，需使用-v参数

Part 2.4

容器交互与执行命令

docker exec - 在容器中执行命令（推荐） Bash 
# 语法：docker exec [OPTIONS] CONTAINER COMMAND [ARG...]
# 作用：在运行中的容器内启动新进程执行命令
#       不影响容器主进程，是进入容器的推荐方式

# 基础用法 - 在Ubuntu容器中执行命令
$ docker exec my-ubuntu ls -la /home

# 预期输出：
# 目录内容列表

# 参数详解：
# --interactive, -i    保持STDIN打开，即使没有附加
#                      用于需要输入的命令
# --tty, -t            分配伪终端（pseudo-TTY）
#                      与-i配合使用（-it），提供完整终端体验
# --user, -u "用户"    指定执行命令的用户（用户名或UID）
#                      示例：-u root 或 -u 0
# --workdir, -w "路径" 指定命令的工作目录
#                      示例：-w /app
# --env, -e "KEY=VAL"  设置环境变量
# --privileged         给命令扩展权限
# --detach, -d         后台运行命令

# 示例：进入Ubuntu容器终端（最常用）
$ docker exec -it my-ubuntu bash

# 预期输出：
root@1a2b3c4d5e6f:/#

# 示例：以root用户身份执行命令
$ docker exec -u root my-ubuntu apt-get update

# 预期输出：
# 更新包列表

# 示例：在指定工作目录执行命令
$ docker exec -w /tmp my-ubuntu touch test.txt

# 预期输出：
# 在/tmp目录创建test.txt文件

# 示例：带环境变量执行命令
$ docker exec -e DEBUG=true my-ubuntu env

# 示例：后台执行命令
$ docker exec -d my-ubuntu /scripts/backup.sh
docker attach 与 docker exec 的区别

docker attach：附加到容器的主进程（PID 1），直接连接到容器的标准输入/输出/错误流

docker exec：在容器内启动一个新的进程，不影响主进程运行

attach 适合查看容器主进程的实时输出，exec 适合在容器内执行命令和调试

docker attach - 附加到容器主进程 Bash 
# 语法：docker attach [OPTIONS] CONTAINER
# 作用：连接到容器主进程的标准输入/输出/错误流，实时查看容器日志

# 附加到运行中的容器（查看实时输出）
$ docker attach my-ubuntu

# 预期输出：
# 直接看到容器主进程的输出内容

# 参数详解：
# --no-stdin=true      不附加标准输入，只查看输出，无法输入命令
#                      适用场景：只想查看日志，不想意外输入影响容器
# --sig-proxy=true     代理信号到容器（默认开启）
#                      开启时：Ctrl+C 会发送 SIGINT 到容器主进程，可能导致容器停止
#                      关闭时：Ctrl+C 只断开连接，容器继续运行

# 示例：只查看输出，不接受输入
$ docker attach --no-stdin my-ubuntu

# 示例：关闭信号代理，Ctrl+C不会停止容器
$ docker attach --sig-proxy=false my-ubuntu

# 重要提示：
# 1. 使用 Ctrl+C 可能会停止容器（当 --sig-proxy=true 时）
# 2. 安全退出（不停止容器）：按 Ctrl+P 然后按 Ctrl+Q
# 3. attach 连接的是容器启动时的主进程，不是新开shell
注意事项

- attach 连接主进程：退出可能影响容器运行状态

- 推荐使用 exec：更安全，不会影响容器主进程

- 安全退出方式：Ctrl+P Ctrl+Q 分离而不停止容器

Part 3

容器资源管理

控制容器的资源使用

Part 3.1

CPU资源限制 --cpus

--cpus 限制CPU核心数 Bash 
# 语法：docker run --cpus <value> IMAGE
# 作用：限制容器可以使用的CPU核心数

# 示例：限制Ubuntu容器使用1个CPU核心
$ docker run -d --name ubuntu-cpu1 --cpus 1 ubuntu:22.04 sleep infinity

# 验证CPU限制
$ docker inspect --format '{{.HostConfig.NanoCpus}}' ubuntu-cpu1
# 输出：1000000000（1个CPU核心，单位纳秒）
说明

--cpus：限制容器可以使用的CPU核心数，可以是小数（如0.5表示半个核心）

Part 3.1

CPU资源限制 --cpu-shares

--cpu-shares 设置CPU权重 Bash 
# 语法：docker run --cpu-shares <value> IMAGE
# 作用：设置CPU权重，当CPU资源竞争时生效（相对值）

# 示例：设置CPU权重（相对值）
$ docker run -d --name ubuntu-high --cpu-shares 1024 ubuntu:22.04 sleep infinity
$ docker run -d --name ubuntu-low --cpu-shares 512 ubuntu:22.04 sleep infinity
# 当CPU资源竞争时，ubuntu-high获得2倍于ubuntu-low的CPU时间
说明

--cpu-shares：设置CPU权重，当CPU资源竞争时生效。默认值为1024，值越大权重越高

Part 3.1

CPU资源限制 --cpuset-cpus

--cpuset-cpus 指定CPU核心 Bash 
# 语法：docker run --cpuset-cpus <value> IMAGE
# 作用：指定容器可以使用的CPU核心（逗号分隔或范围）

# 示例：指定容器使用特定CPU核心（0和1）
$ docker run -d --name ubuntu-pinned --cpuset-cpus 0,1 ubuntu:22.04 sleep infinity

# 示例：使用范围表示法（等同于0,1）
$ docker run -d --name ubuntu-range --cpuset-cpus 0-1 ubuntu:22.04 sleep infinity
说明

--cpuset-cpus：指定容器可以使用的CPU核心。可以是逗号分隔的列表（如"0,1,2"）或范围（如"0-3"）

Part 3.2

内存资源限制 --memory

--memory 硬内存限制 Bash 
# 语法：docker run --memory <value> IMAGE
# 作用：限制容器使用的内存，不能超过此值

# 示例：限制Ubuntu容器使用512MB内存
$ docker run -d --name ubuntu-mem --memory 512m ubuntu:22.04 sleep infinity

# 内存单位：b, k, m, g（字节、KB、MB、GB）

# 验证内存限制
$ docker inspect --format '{{.HostConfig.Memory}}' ubuntu-mem
# 输出：536870912（512MB的字节数）

# 在容器内查看内存限制
$ docker exec ubuntu-mem cat /sys/fs/cgroup/memory/memory.limit_in_bytes
536870912
说明

--memory：硬限制，容器使用的内存不能超过此值。如果容器尝试使用更多内存，会被OOM killer终止

Part 3.2

内存资源限制 --memory-swap

--memory-swap 交换空间限制 Bash 
# 语法：docker run --memory <mem> --memory-swap <swap> IMAGE
# 作用：memory-swap = memory + swap，限制总内存使用

# 示例：限制内存512MB，交换空间1GB
$ docker run -d --name ubuntu-swap \
  --memory 512m --memory-swap 1g ubuntu:22.04 sleep infinity
# swap = 1g - 512m = 512m

# 注意：--memory-swap必须大于--memory
说明

--memory-swap：内存+交换空间的总限制。当物理内存不足时，使用交换空间

Part 3.2

--memory-reservation 软限制 Bash 
# 语法：docker run --memory <mem> --memory-reservation <res> IMAGE
# 作用：设置软限制，内存紧张时触发，不强制执行

# 示例：硬限制512MB，软限制256MB
$ docker run -d --name ubuntu-soft \
  --memory 512m --memory-reservation 256m ubuntu:22.04 sleep infinity

--memory-reservation：软限制，当系统内存不足时会提前警告，不会强制终止容器

内存资源限制 --oom-kill-disable

--oom-kill-disable 禁用OOM killer Bash

$ docker run -d --name ubuntu-no-oom \

谨慎使用：禁用OOM killer后，如果容器耗尽内存，可能会影响宿主机上其他进程

巩固容器生命周期管理命令

实验

容器操作命令综合练习

实验 容器生命周期管理综合实践

实验目的

通过综合练习，熟练掌握docker create、run、start、stop、rm、exec等核心命令的使用

实验步骤

1 使用docker create创建一个Nginx容器（不启动）
2 使用docker start启动该容器
3 使用docker exec进入容器查看文件
4 使用docker stop优雅停止容器
5 使用docker rm删除容器
6 使用docker run一步完成创建并启动
实验操作指南 Bash 
# 步骤1：使用docker create创建容器（不启动）
$ docker create --name my-nginx -p 8080:80 nginx

# 预期输出：
# 容器ID（如：1a2b3c4d5e6f）

# 步骤2：查看已创建但未启动的容器
$ docker ps -a --filter status=created

# 步骤3：启动容器
$ docker start my-nginx

# 预期输出：
my-nginx

# 步骤4：查看运行中的容器
$ docker ps

# 步骤5：进入容器查看文件
$ docker exec -it my-nginx bash
root@1a2b3c4d5e6f:/# ls /usr/share/nginx/html/
root@1a2b3c4d5e6f:/# cat /etc/nginx/nginx.conf
root@1a2b3c4d5e6f:/# exit

# 步骤6：查看容器日志
$ docker logs my-nginx

# 步骤7：优雅停止容器
$ docker stop my-nginx

# 预期输出：
my-nginx

# 步骤8：删除容器
$ docker rm my-nginx

# 预期输出：
my-nginx

# 步骤9：使用docker run一步完成（常用方式）
$ docker run -d --name web-server -p 8080:80 --restart unless-stopped nginx

# 预期输出：
# 容器ID
命令对比总结

- docker create + docker start：分步操作，适合需要先配置再启动的场景

- docker run：一步完成创建和启动，最常用

- docker stop：优雅停止，给容器时间清理资源

- docker rm：删除已停止的容器，-f可强制删除运行中的容器

- docker exec：在运行中的容器内执行命令，推荐用于进入容器

Part 4

容器网络管理

配置容器的网络连接

Part 4.1

Docker网络模式概述

Docker网络模式
Docker提供了多种网络模式来满足不同的应用场景需求。理解这些网络模式对于正确配置容器通信至关重要。

Docker支持以下5种网络模式，每种模式适用于不同的场景：

网络模式 说明 适用场景
bridge 默认模式，容器连接到Docker虚拟网桥 独立应用容器，需要网络隔离
host 移除网络隔离，直接使用宿主机网络 对网络性能要求高的场景
none 禁用容器网络功能，完全隔离 不需要网络的应用，如计算任务
container 与其他容器共享网络命名空间 容器间紧密耦合，如Sidecar模式
自定义网络 用户创建的bridge或overlay网络 多容器应用，需要DNS解析
查看Docker网络

1. 查看所有网络：docker network ls

2. 查看网络详情：docker network inspect bridge

3. 查看容器网络：docker inspect --format '{{.NetworkSettings.Networks}}' <container>

Part 4.2

Bridge网络模式

Bridge网络
Bridge是Docker的默认网络模式。当容器启动时，如果未指定网络模式，Docker会创建一个虚拟网桥（docker0），容器通过该网桥与外部网络通信。
Bridge网络原理 概念说明 
                    ┌─────────────────────────────────────┐
                    │           宿主机                      │
                    │                                      │
                    │    ┌────────────────────────────┐   │
                    │    │     docker0 网桥           │   │
                    │    │     172.17.0.1           │   │
                    │    └──────────┬───────────────┘   │
                    │               │                    │
              ┌────┴────┐    ┌────┴────┐    ┌────┴────┐
              │ Container│    │ Container│    │ Container│
              │ 172.17.0.2│    │ 172.17.0.3│    │ 172.17.0.4│
              └──────────┘    └──────────┘    └──────────┘
                    │               │                    │
                    └───────────────┴────────────────────┘
                                       │
                                 ┌─────┴─────┐
                                 │ NAT转换   │
                                 └─────┬─────┘
                                       │
                                 ┌─────┴─────┐
                                 │ 物理网卡   │
                                 └───────────┘
Bridge网络示例（Ubuntu容器） Bash 
# 查看默认bridge网络信息
$ docker network inspect bridge

# 预期输出（部分）：
{
    "Name": "bridge",
    "Id": "abc123...",
    "Driver": "bridge",
    "Subnet": "172.17.0.0/16",
    "Gateway": "172.17.0.1"
}

# 启动Ubuntu容器（默认使用bridge网络）
$ docker run -d --name ubuntu-bridge ubuntu:22.04 sleep infinity

# 查看容器IP地址
$ docker inspect --format '{{.NetworkSettings.Networks.bridge.IPAddress}}' ubuntu-bridge
172.17.0.2

# 从容器内测试网络连接
$ docker exec ubuntu-bridge ping -c 3 baidu.com

# 容器间通信（通过容器名，需要使用自定义网络）
$ docker run -d --name ubuntu-1 ubuntu:22.04 sleep infinity
$ docker exec ubuntu-1 ping -c 3 ubuntu-1
# ping: bad address 'ubuntu-1'  （默认bridge不支持DNS解析）
Bridge网络特点

- 每个容器有独立的IP地址

- 容器可以访问外部网络（通过NAT）

- 外部访问容器需要端口映射

- 同一bridge网络的容器可以通过IP通信，但不能通过容器名通信

Part 4.3

Host网络模式

Host网络
Host模式移除容器与宿主机之间的网络隔离，容器直接使用宿主机的网络命名空间。容器不会获得独立的IP地址，而是共享宿主机的IP地址和端口。
Host网络示例 Bash 
# 使用host网络模式启动Ubuntu容器
$ docker run -d --name ubuntu-host --network host ubuntu:22.04 sleep infinity

# 查看容器网络信息
$ docker inspect --format '{{.NetworkSettings.Networks}}' ubuntu-host
# 输出：{} （无独立网络，使用宿主机网络）

# 在容器内查看网络
$ docker exec ubuntu-host hostname -I
192.168.1.100  # 与宿主机IP相同

# 端口监听测试
$ docker exec ubuntu-host netstat -tuln | grep :80
# 容器内可直接绑定宿主机的80端口
Host网络 vs Bridge网络对比 Bash 
# Bridge网络（需要端口映射）
$ docker run -d --name nginx-bridge -p 8080:80 nginx
# 访问：http://localhost:8080

# Host网络（直接使用宿主机端口）
$ docker run -d --name nginx-host --network host nginx
# 访问：http://localhost:80  （无需映射）

# 性能对比
# Host网络：性能最好，延迟最低，无NAT开销
# Bridge网络：有NAT转换，性能略低
注意事项

- 端口冲突：如果宿主机已有服务占用端口，容器无法启动

- 网络隔离：容器与宿主机共享网络，无法隔离

- Windows/Mac：Host网络在Docker Desktop中行为不同

对比项 Bridge Host
IP地址 独立IP（172.17.0.x） 与宿主机共享
端口映射 需要（-p 参数） 不需要
网络性能 有NAT开销 最优
隔离性 网络隔离 无隔离
Part 4.4

None与Container网络模式

None网络
None模式完全禁用容器的网络功能。容器拥有独立的网络命名空间，但不配置任何网络接口（除loopback外）。
None网络示例 Bash 
# 启动无网络Ubuntu容器
$ docker run -d --name ubuntu-none --network none ubuntu:22.04 sleep infinity

# 查看容器网络
$ docker inspect --format '{{.NetworkSettings.Networks}}' ubuntu-none
# 输出：null 或 {}

# 在容器内查看网络接口
$ docker exec ubuntu-none ip addr
1: lo: <LOOPBACK,UP,LOWER_UP> mtu 65536 qdisc noqueue state UNKNOWN
    inet 127.0.0.1/8 scope host lo
       valid_lft forever preferred_lft forever
# 注意：没有eth0，只有loopback
Container网络
Container模式与其他容器共享网络命名空间。新容器不会创建自己的网络接口，而是与指定容器共享同一网络栈。
Container网络示例 Bash 
# 启动第一个容器（作为网络基础）
$ docker run -d --name ubuntu-base ubuntu:22.04 sleep infinity

# 查看基础容器的IP
$ docker inspect --format '{{.NetworkSettings.Networks.bridge.IPAddress}}' ubuntu-base
172.17.0.2

# 启动新容器，共享基础容器的网络
$ docker run -d --name ubuntu-share --network container:ubuntu-base ubuntu:22.04 sleep infinity

# 验证：两个容器IP相同
$ docker inspect --format '{{.NetworkSettings.Networks.bridge.IPAddress}}' ubuntu-share
172.17.0.2

# 在新容器内测试
$ docker exec ubuntu-share ping -c 3 ubuntu-base
# 可以通信（因为共享网络命名空间）

# 端口测试
$ docker exec ubuntu-base nc -l 8888 &
$ docker exec ubuntu-share nc localhost 8888
# 可以直接通过localhost通信
适用场景

None网络：批处理任务、离线计算、安全隔离场景

Container网络：Sidecar模式、多容器共享配置、紧密耦合的应用

Part 4.5

端口映射详解

端口映射
端口映射（Port Mapping）是将宿主机的端口与容器的端口进行关联，使外部流量可以通过宿主机端口访问容器内服务。
端口映射语法与示例 Bash 
# 语法：-p <host-port>:<container-port>[:<protocol>]

# 基本端口映射
$ docker run -d --name nginx-web -p 8080:80 nginx
# 访问：http://localhost:8080

# 映射多个端口
$ docker run -d --name nginx-multi -p 8080:80 -p 8443:443 nginx

# 指定协议（tcp/udp）
$ docker run -d --name dns-server -p 53:53/udp redis:alpine

# 绑定到特定IP（仅本地访问）
$ docker run -d --name nginx-local -p 127.0.0.1:8080:80 nginx
# 只能通过 localhost:8080 访问

# 绑定到所有IP（默认）
$ docker run -d --name nginx-all -p 0.0.0.0:8080:80 nginx
# 可以通过任何网络接口访问

# 随机端口映射（仅指定容器端口）
$ docker run -d --name nginx-random -p 80 nginx
# Docker自动分配一个宿主机端口

# 查看端口映射
$ docker port nginx-web
80/tcp -> 0.0.0.0:8080

# 查看所有映射端口
$ docker ps --format "table {{.Names}}\t{{.Ports}}"
NAMES        PORTS
nginx-web    0.0.0.0:8080->80/tcp
端口映射原理 概念说明 
# 端口映射工作流程：
# 1. 宿主机监听 8080 端口
# 2. 收到请求后，NAT转换目的地址
# 3. 转发到容器 IP:80 端口
# 4. 容器处理请求，返回响应
# 5. NAT转换源地址，返回客户端

# 示例流程：
# 客户端 -> http://192.168.1.100:8080
#      -> 宿主机eth0:8080
#      -> NAT转换（目的IP改为172.17.0.2:80）
#      -> 容器eth0:80
#      -> 处理请求
#      -> 返回响应
#      -> NAT转换（源IP改为192.168.1.100）
#      -> 客户端
注意事项

- 端口冲突：确保宿主机端口未被占用

- 防火墙：检查防火墙是否允许访问映射端口

- 容器重启：端口映射在容器重启后保持不变

- 动态端口：生产环境建议使用固定端口

端口映射故障排查

问题1：端口被占用

错误信息：port is already allocated

# Windows查看端口占用
$ netstat -ano | findstr :8080

# Linux/Mac查看端口占用
$ lsof -i :8080

# 解决方案1：更换端口
$ docker run -d -p 8081:80 --name web nginx

# 解决方案2：停止占用端口的进程或容器

问题2：容器启动成功但访问失败

错误信息：ERR_CONNECTION_REFUSED

# 1. 确认容器正在运行
$ docker ps

# 2. 检查端口映射
$ docker port web
# 预期输出：80/tcp -> 0.0.0.0:8080

# 3. 查看容器日志
$ docker logs web

# 4. 检查防火墙（Windows）
# 关闭防火墙或添加端口例外

问题3：混淆端口映射方向

常见错误：-p 80:8080（写反了）

# 正确理解：-p 宿主机端口:容器端口
# 外部访问 -> 宿主机:8080 -> 容器:80
$ docker run -d -p 8080:80 nginx

# 记忆方法：外部 -> 内部，从左到右
Part 4.6

自定义网络创建与管理

自定义网络
自定义网络允许用户创建自定义的bridge网络，支持容器间通过容器名进行DNS解析，无需手动配置链接。
docker network create - 创建网络 Bash 
# 查看所有网络
$ docker network ls

# 预期输出：
NETWORK ID     NAME        DRIVER      SCOPE
abc123...      bridge      bridge      local
def456...      host        host        local
ghi789...      none        null        local

# 创建自定义bridge网络
$ docker network create my-network

# 创建时指定子网和网关
$ docker network create \
  --driver bridge \
  --subnet 192.168.100.0/24 \
  --gateway 192.168.100.1 \
  my-subnet
docker network inspect/rm/prune - 管理网络 Bash 
# 查看网络详情
$ docker network inspect my-network

# 删除网络
$ docker network rm my-network

# 清理未使用的网络
$ docker network prune
自定义网络优势

- DNS解析：容器可以通过名称相互访问

- 网络隔离：不同项目使用不同网络

- 灵活配置：可自定义子网、网关、IP范围

Part 4.7

容器网络故障排查 - 常见问题

问题1：容器无法访问外网 Bash 
# 原因：DNS配置错误或网络不通
# 解决：
$ docker exec <container> ping 8.8.8.8
# 如果IP能通但域名不通，检查DNS：
$ docker exec <container> cat /etc/resolv.conf
问题2：无法通过端口访问容器 Bash 
# 原因：端口未映射、防火墙阻止、服务未启动
# 解决：
$ docker port <container>
$ curl http://localhost:<port>
$ docker logs <container>
问题3：容器间无法通信 Bash 
# 原因：不在同一网络、未使用自定义网络
# 解决：
$ docker network inspect <network>

docker logs 查看日志

docker logs 基础用法 Bash 
# 语法：docker logs [OPTIONS] CONTAINER

$ docker logs my-nginx

# 常用参数说明：

# --until             显示指定时间之前的日志
--follow 和 --tail 参数 Bash

Part 6.1

docker logs 时间过滤

--timestamps 和 --since 参数 Bash 
# 示例：显示带时间戳的日志
$ docker logs -t my-nginx

# 示例：显示过去1小时的日志
$ docker logs --since 1h my-nginx
--since 和 --until 时间段 Bash 
# 示例：显示特定时间段的日志
$ docker logs --since "2024-01-15T10:00:00" \
  --until "2024-01-15T11:00:00" my-nginx
时间格式说明

--since：支持相对时间（如1h、30m）和绝对时间

--until：与--since配合使用，指定结束时间

Part 6.2

docker stats 容器监控

docker stats 基础用法 Bash 
# 语法：docker stats [OPTIONS] [CONTAINER...]

# 基础用法 - 查看所有容器的资源使用情况
$ docker stats

# 预期输出：
CONTAINER ID   NAME      CPU %     MEM USAGE / LIMIT     MEM %
1a2b3c4d5e6f   my-nginx  0.03%     2.562MiB / 1.945GiB   0.13%
docker stats 参数说明 Bash 
# 常用参数说明：
# --all, -a           显示所有容器（包括停止的）
# --no-stream         只显示当前状态，不持续更新
# --no-trunc          不截断输出

# 示例：查看特定容器的资源使用情况
$ docker stats my-nginx

# 示例：只显示当前状态
$ docker stats --no-stream
监控指标说明

- CPU %：CPU使用率

- MEM USAGE / LIMIT：内存使用量/限制

- MEM %：内存使用率

- NET I/O：网络输入/输出

Part 6.2

docker stats 格式化输出

--format 格式化输出 Bash 
# 示例：格式化输出
$ docker stats --format \
  "table {{.Name}}\t{{.CPUPerc}}\t{{.MemUsage}}"

# 预期输出：
NAME       CPU %     MEM USAGE / LIMIT
my-nginx   0.03%     2.562MiB / 1.945GiB
常用格式化字段

- {{.Name}}：容器名称

- {{.CPUPerc}}：CPU使用率

- {{.MemUsage}}：内存使用情况

- {{.MemPerc}}：内存使用率

- {{.NetIO}}：网络IO

- {{.BlockIO}}：块设备IO

- {{.Pids}}：进程数

保障容器的安全运行

Part 7.1

容器安全最佳实践

1使用官方镜像

从Docker Hub官方仓库拉取镜像，避免使用未知来源的镜像

2最小化镜像

使用Alpine等轻量级基础镜像，减少攻击面

3定期更新镜像

及时更新镜像以修复安全漏洞

4使用非root用户

在Dockerfile中创建并使用普通用户，避免以root身份运行容器

5限制容器权限

使用--cap-drop参数移除不必要的Linux能力

6网络隔离

使用自定义网络，限制容器间的网络访问

7资源限制

设置CPU、内存等资源限制，防止资源耗尽攻击

8安全扫描

使用工具扫描镜像中的安全漏洞

安全配置示例 Bash 
# 示例：以非root用户运行容器
$ docker run --user 1000:1000 nginx

# 示例：限制容器能力
$ docker run --cap-drop ALL --cap-add NET_BIND_SERVICE nginx

# 示例：只读文件系统
$ docker run --read-only nginx

# 示例：禁用特权模式
$ docker run --privileged=false nginx

# 示例：设置seccomp配置
$ docker run --security-opt seccomp=profile.json nginx

总结

Docker容器管理小结

* 容器概念：镜像的运行实例，轻量级、隔离、可移植
* 生命周期：Created → Running → Paused → Stopped → Deleted
* 核心命令：run、start、stop、rm、exec、ps
* 资源管理：CPU、内存、IO限制
* 网络配置：bridge、host、none、自定义网络
* 数据管理：数据卷、绑定挂载、tmpfs
* 监控日志：docker stats、docker logs
* 安全最佳实践：最小化镜像、非root用户、资源限制
核心要点回顾

1. 容器是Docker的核心运行单元，具有轻量级、隔离性和可移植性

2. 掌握容器的生命周期管理是Docker使用的基础

3. 合理配置资源限制和网络设置可以提高容器的安全性和性能

4. 数据持久化是生产环境中必须考虑的重要因素

5. 监控和日志管理有助于及时发现和解决容器运行中的问题

6. 遵循安全最佳实践可以减少容器环境的安全风险
