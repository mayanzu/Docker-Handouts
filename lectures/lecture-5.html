Part 1

容器基础概念

理解容器的本质和特点

Part 1.1

什么是Docker容器？

容器定义
Docker容器是镜像的运行实例，是一个轻量级、独立、可执行的软件包，包含运行应用所需的所有内容：代码、运行时、系统工具、系统库和配置。

容器与镜像的关系类似于对象与类的关系。镜像是静态的模板，容器是镜像的动态实例。容器在镜像的基础上添加了一个可写层，所有运行时的修改都发生在这个可写层中。

1轻量级

共享宿主机内核，启动时间秒级，资源占用少

2隔离性

通过Namespace和Cgroups实现进程、网络、文件系统等隔离

3可移植性

在任何Docker环境中行为一致，实现"一次构建，到处运行"

4可控制

可限制CPU、内存、IO等资源使用，确保系统稳定

容器 vs 虚拟机

容器：轻量级，启动快，资源占用少，共享内核

虚拟机：重量级，启动慢，资源占用多，完整操作系统

容器操作命令

掌握容器的创建、启动、停止等操作

Part 2.1

创建与启动容器

docker create - 创建容器（不启动） Bash 
# 语法：docker create [OPTIONS] IMAGE [COMMAND] [ARG...]
# 作用：创建一个新容器但不启动它，返回容器ID
#       适合需要先配置再启动的场景

# 基础用法 - 创建Ubuntu容器
$ docker create --name my-ubuntu ubuntu:22.04

# 预期输出：
# 容器ID（如：1a2b3c4d5e6f...）

# 参数详解：
# --name "容器名"      为容器指定名称，便于后续管理
#                      不指定则自动生成随机名称
# --env, -e "KEY=VAL"  设置环境变量，可多次使用
#                      示例：-e TZ=Asia/Shanghai
# --volume, -v "宿主路径:容器路径"  挂载数据卷
#                      示例：-v /data/app:/app
# --port, -p "宿主端口:容器端口"   端口映射
#                      示例：-p 8080:80
# --network "网络名"   将容器连接到指定网络
#                      示例：--network my-network
# --restart "策略"     容器退出后的重启策略
#                      no：不重启（默认）
#                      always：总是重启
#                      on-failure：失败时重启
#                      unless-stopped：除非手动停止否则重启

# 示例：创建带完整配置的Ubuntu容器
$ docker create \
  --name ubuntu-app \
  -e TZ=Asia/Shanghai \
  -e LANG=C.UTF-8 \
  -v /data/app:/app \
  --restart unless-stopped \
  ubuntu:22.04

# 预期输出：
# 容器ID

# 创建后需要手动启动
$ docker start ubuntu-app
docker run - 创建并启动容器（最常用） Bash 
# 语法：docker run [OPTIONS] IMAGE [COMMAND] [ARG...]
# 作用：创建容器并立即启动，相当于 create + start

# 基础用法 - 启动Ubuntu容器（交互式）
$ docker run -it --name my-ubuntu ubuntu:22.04 bash

# 预期输出：
# 进入Ubuntu容器的bash终端
root@1a2b3c4d5e6f:/#

# 参数详解：
# --detach, -d         后台运行模式，容器在后台运行并返回容器ID
#                      不加此参数则前台运行，占用当前终端
# --interactive, -i    保持STDIN打开，即使没有附加
#                      通常与-t配合使用（-it），用于交互式操作
# --tty, -t            分配一个伪终端（pseudo-TTY）
#                      与-i配合使用，提供完整的终端体验
# --rm                 容器退出后自动删除
#                      适合临时测试，不会留下停止的容器
# --name "名称"        指定容器名称，便于后续管理
# --env, -e "KEY=VAL"  设置环境变量
# --volume, -v         挂载数据卷
# --port, -p           端口映射（宿主端口:容器端口）
# --network "网络名"   指定网络
# --restart "策略"     重启策略（no/always/on-failure/unless-stopped）
# --cpus "数量"        限制CPU核心数
# --memory, -m "大小"  限制内存使用

# 示例：交互式运行Ubuntu容器（临时测试）
$ docker run -it --rm ubuntu:22.04 bash

# 预期输出：
# 进入Ubuntu容器的bash终端
root@1a2b3c4d5e6f:/# cat /etc/os-release
PRETTY_NAME="Ubuntu 22.04.3 LTS"
...
root@1a2b3c4d5e6f:/# exit
# 退出后容器自动删除

# 示例：后台运行Ubuntu容器
$ docker run -d --name ubuntu-server \
  -e TZ=Asia/Shanghai \
  --restart unless-stopped \
  ubuntu:22.04 sleep infinity

# 预期输出：
# 容器ID

# 示例：带资源限制运行
$ docker run -d --name ubuntu-limited \
  --cpus 0.5 \
  --memory 512m \
  --restart unless-stopped \
  ubuntu:22.04 sleep infinity
运行模式

前台运行：容器在当前终端运行，按Ctrl+C停止

后台运行：使用-d参数，容器在后台运行

交互式运行：使用-it参数，进入容器终端

查看容器

docker ps - 列出容器 Bash 
# 语法：docker ps [OPTIONS]
# 作用：列出容器信息，默认只显示运行中的容器

# 基础用法 - 列出运行中的容器
$ docker ps

# 预期输出：
CONTAINER ID   IMAGE          COMMAND           CREATED         STATUS         PORTS     NAMES
1a2b3c4d5e6f   ubuntu:22.04   "sleep infinity"  5 minutes ago   Up 5 minutes             my-ubuntu

# 参数详解：
# --all, -a            显示所有容器，包括已停止的
#                      默认只显示运行中的容器
# --quiet, -q          只显示容器ID，适合脚本处理
#                      常用于批量操作：docker rm $(docker ps -aq)
# --filter, -f "条件"  根据条件过滤容器
#                      示例：-f status=running（运行中）
#                            -f name=ubuntu（名称包含ubuntu）
#                            -f ancestor=ubuntu:22.04（基于ubuntu镜像）
# --format "模板"      使用Go模板格式化输出
#                      示例：--format "table {{.Names}}\t{{.Status}}"
# --last, -n "数量"    显示最近创建的n个容器（包括停止的）
# --no-trunc           不截断输出，显示完整信息
# -s, --size           显示容器大小

# 示例：显示所有容器
$ docker ps -a

# 预期输出：
CONTAINER ID   IMAGE          COMMAND           CREATED          STATUS                      PORTS     NAMES
1a2b3c4d5e6f   ubuntu:22.04   "sleep infinity"  5 minutes ago    Up 5 minutes                          my-ubuntu

# 示例：只显示容器ID（常用于批量操作）
$ docker ps -q

# 预期输出：
1a2b3c4d5e6f

# 示例：显示最近创建的3个容器
$ docker ps -n 3

# 预期输出：
# 最近3个容器的信息

# 示例：过滤特定状态的容器
$ docker ps -f status=exited

# 示例：自定义格式输出
$ docker ps --format "table {{.Names}}\t{{.Status}}\t{{.Image}}"

Part 2.3

停止与删除容器

docker stop - 优雅停止容器 Bash 
# 语法：docker stop [OPTIONS] CONTAINER [CONTAINER...]
# 作用：发送SIGTERM信号，等待容器优雅停止
#       容器内的进程有时间清理资源、保存数据

# 基础用法 - 停止Ubuntu容器
$ docker stop my-ubuntu

# 预期输出：
my-ubuntu

# 参数详解：
# --time, -t "秒数"    停止前的等待时间，默认10秒
#                      Docker先发送SIGTERM，等待指定时间
#                      如果超时仍未停止，则发送SIGKILL强制停止
#                      建议给重要应用更长的停止时间

# 示例：停止多个容器
$ docker stop ubuntu-app1 ubuntu-app2

# 预期输出：
ubuntu-app1
ubuntu-app2

# 示例：给容器更长的停止时间（30秒）
$ docker stop -t 30 ubuntu-database

# 预期输出：
ubuntu-database

# 停止流程说明：
# 1. Docker发送SIGTERM信号给容器主进程
# 2. 主进程收到信号，开始清理工作（如保存数据、关闭连接）
# 3. 如果在-t时间内停止，容器正常退出
# 4. 如果超时，Docker发送SIGKILL强制终止
docker start - 启动已停止的容器 Bash 
# 语法：docker start [OPTIONS] CONTAINER [CONTAINER...]
# 作用：启动已停止的容器，恢复之前的运行状态

# 基础用法 - 启动容器
$ docker start my-ubuntu

# 预期输出：
my-ubuntu

# 参数详解：
# --attach, -a         附加到容器的标准输入/输出/错误
#                      启动后立即看到容器输出
# --interactive, -i    附加到容器的STDIN
#                      与-a配合使用，可以与容器交互
# --checkpoint "名称"  从检查点恢复（实验性功能）
# --checkpoint-dir     检查点目录（实验性功能）

# 示例：启动并附加到容器（查看输出）
$ docker start -a my-ubuntu

# 预期输出：
# 显示容器启动日志

# 示例：启动并进入交互模式
$ docker start -a -i my-ubuntu

# 预期输出：
# 进入容器终端
docker rm - 删除容器 Bash 
# 语法：docker rm [OPTIONS] CONTAINER [CONTAINER...]
# 作用：删除已停止的容器，释放资源

# 基础用法 - 删除容器
$ docker rm my-ubuntu

# 预期输出：
my-ubuntu

# 参数详解：
# --force, -f          强制删除，即使容器正在运行
#                      会先停止容器再删除，慎用
# --volumes, -v        删除容器关联的匿名卷
#                      注意：不会删除命名卷
# --link, -l           删除容器间的链接（已弃用）

# 示例：强制删除正在运行的容器
$ docker rm -f my-ubuntu

# 预期输出：
my-ubuntu

# 示例：删除多个容器
$ docker rm ubuntu-app1 ubuntu-app2

# 预期输出：
ubuntu-app1
ubuntu-app2

# 示例：删除所有停止的容器
$ docker rm $(docker ps -a -q)

# 预期输出：
# 所有停止容器的ID
注意事项

- 删除容器前需先停止容器（除非使用-f参数）

- 使用-f参数强制删除可能导致数据丢失

- 删除容器不会自动删除关联的数据卷，需使用-v参数

Part 2.4

容器交互与执行命令

docker exec - 在容器中执行命令（推荐） Bash 
# 语法：docker exec [OPTIONS] CONTAINER COMMAND [ARG...]
# 作用：在运行中的容器内启动新进程执行命令
#       不影响容器主进程，是进入容器的推荐方式

# 基础用法 - 在Ubuntu容器中执行命令
$ docker exec my-ubuntu ls -la /home

# 预期输出：
# 目录内容列表

# 参数详解：
# --interactive, -i    保持STDIN打开，即使没有附加
#                      用于需要输入的命令
# --tty, -t            分配伪终端（pseudo-TTY）
#                      与-i配合使用（-it），提供完整终端体验
# --user, -u "用户"    指定执行命令的用户（用户名或UID）
#                      示例：-u root 或 -u 0
# --workdir, -w "路径" 指定命令的工作目录
#                      示例：-w /app
# --env, -e "KEY=VAL"  设置环境变量
# --privileged         给命令扩展权限
# --detach, -d         后台运行命令

# 示例：进入Ubuntu容器终端（最常用）
$ docker exec -it my-ubuntu bash

# 预期输出：
root@1a2b3c4d5e6f:/#

# 示例：以root用户身份执行命令
$ docker exec -u root my-ubuntu apt-get update

# 预期输出：
# 更新包列表

# 示例：在指定工作目录执行命令
$ docker exec -w /tmp my-ubuntu touch test.txt

# 预期输出：
# 在/tmp目录创建test.txt文件

# 示例：带环境变量执行命令
$ docker exec -e DEBUG=true my-ubuntu env

# 示例：后台执行命令
$ docker exec -d my-ubuntu /scripts/backup.sh
docker attach 与 docker exec 的区别

docker attach：附加到容器的主进程（PID 1），直接连接到容器的标准输入/输出/错误流

docker exec：在容器内启动一个新的进程，不影响主进程运行

attach 适合查看容器主进程的实时输出，exec 适合在容器内执行命令和调试

docker attach - 附加到容器主进程 Bash 
# 语法：docker attach [OPTIONS] CONTAINER
# 作用：连接到容器主进程的标准输入/输出/错误流，实时查看容器日志

# 附加到运行中的容器（查看实时输出）
$ docker attach my-ubuntu

# 预期输出：
# 直接看到容器主进程的输出内容

# 参数详解：
# --no-stdin=true      不附加标准输入，只查看输出，无法输入命令
#                      适用场景：只想查看日志，不想意外输入影响容器
# --sig-proxy=true     代理信号到容器（默认开启）
#                      开启时：Ctrl+C 会发送 SIGINT 到容器主进程，可能导致容器停止
#                      关闭时：Ctrl+C 只断开连接，容器继续运行

# 示例：只查看输出，不接受输入
$ docker attach --no-stdin my-ubuntu

# 示例：关闭信号代理，Ctrl+C不会停止容器
$ docker attach --sig-proxy=false my-ubuntu

# 重要提示：
# 1. 使用 Ctrl+C 可能会停止容器（当 --sig-proxy=true 时）
# 2. 安全退出（不停止容器）：按 Ctrl+P 然后按 Ctrl+Q
# 3. attach 连接的是容器启动时的主进程，不是新开shell
注意事项

- attach 连接主进程：退出可能影响容器运行状态

- 推荐使用 exec：更安全，不会影响容器主进程

- 安全退出方式：Ctrl+P Ctrl+Q 分离而不停止容器

Part 3

容器资源管理

控制容器的资源使用

Part 3.1

CPU资源限制

CPU限制参数 Bash 
# 语法：docker run --cpus <value> --cpu-shares <value> --cpuset-cpus <value> IMAGE

# 示例：限制Ubuntu容器使用1个CPU核心
$ docker run -d --name ubuntu-cpu1 --cpus 1 ubuntu:22.04 sleep infinity

# 示例：设置CPU权重（相对值）
$ docker run -d --name ubuntu-high --cpu-shares 1024 ubuntu:22.04 sleep infinity
$ docker run -d --name ubuntu-low --cpu-shares 512 ubuntu:22.04 sleep infinity
# 当CPU资源竞争时，ubuntu-high获得2倍于ubuntu-low的CPU时间

# 示例：指定容器使用特定CPU核心（0和1）
$ docker run -d --name ubuntu-pinned --cpuset-cpus 0,1 ubuntu:22.04 sleep infinity

# 示例：限制CPU使用百分比
$ docker run -d --name ubuntu-half --cpus 0.5 ubuntu:22.04 sleep infinity  # 使用50%的CPU

# 示例：设置CPU周期和配额
$ docker run -d --name ubuntu-quota --cpu-period 100000 --cpu-quota 50000 ubuntu:22.04 sleep infinity
# cpu-period：CPU周期（微秒）
# cpu-quota：CPU配额（微秒）
# 50000/100000 = 50% CPU使用率

# 验证CPU限制
$ docker inspect --format '{{.HostConfig.NanoCpus}}' ubuntu-cpu1
# 输出：1000000000（1个CPU核心，单位纳秒）
CPU限制说明

- --cpus：限制容器可以使用的CPU核心数

- --cpu-shares：设置CPU权重，当CPU资源竞争时生效

- --cpuset-cpus：指定容器可以使用的CPU核心

- --cpu-period/--cpu-quota：更精细的CPU使用限制

Part 3.2

内存资源限制

内存限制参数 Bash 
# 语法：docker run --memory <value> --memory-swap <value> IMAGE

# 示例：限制Ubuntu容器使用512MB内存
$ docker run -d --name ubuntu-mem --memory 512m ubuntu:22.04 sleep infinity

# 示例：限制内存和交换空间
$ docker run -d --name ubuntu-swap --memory 512m --memory-swap 1g ubuntu:22.04 sleep infinity
# memory-swap = memory + swap，所以swap = 1g - 512m = 512m

# 示例：设置内存软限制（警告但不强制）
$ docker run -d --name ubuntu-soft --memory 512m --memory-reservation 256m ubuntu:22.04 sleep infinity

# 示例：禁用OOM killer（谨慎使用）
$ docker run -d --name ubuntu-no-oom --oom-kill-disable --memory 512m ubuntu:22.04 sleep infinity

# 内存单位：b, k, m, g（字节、KB、MB、GB）

# 验证内存限制
$ docker inspect --format '{{.HostConfig.Memory}}' ubuntu-mem
# 输出：536870912（512MB的字节数）

# 在容器内查看内存限制
$ docker exec ubuntu-mem cat /sys/fs/cgroup/memory/memory.limit_in_bytes
536870912
内存限制说明

- --memory：硬限制，容器使用的内存不能超过此值

- --memory-swap：内存+交换空间的总限制

- --memory-reservation：软限制，内存紧张时会触发

- --oom-kill-disable：禁用OOM killer（谨慎使用）

Part 3.3

IO限制参数 Bash 
# 语法：docker run --blkio-weight <value> --device-read-bps <value> --device-write-bps <value> IMAGE

# 示例：设置块设备IO权重（10-1000）
$ docker run -d --name ubuntu-io --blkio-weight 500 ubuntu:22.04 sleep infinity

# 示例：限制设备读取速度（每秒1MB）

Part 2.5

容器与镜像交互

容器与镜像之间的转换操作

Part 2.5.1

docker export与docker import - 容器归档

docker export用于将容器的当前文件系统导出为tar归档文件，docker import用于从tar文件创建一个新的镜像。这与docker save/load不同，export操作的是容器，保存的是容器运行时的快照。

docker export - 导出容器 Bash 
# 语法：docker export [OPTIONS] CONTAINER

# 导出容器文件系统到文件
$ docker export my-container -o container-filesystem.tar

# 使用标准输出重定向
$ docker export my-container > container.tar

# 参数说明：
# -o, --output    指定输出文件名
# --pause          导出时暂停容器（默认true）

# 示例：导出容器
$ docker export my-container -o my-container-export.tar

# 预期输出：
# my-container-export.tar 文件已创建
docker import - 导入镜像 Bash 
# 语法：docker import [OPTIONS] URL|- [REPOSITORY[:TAG]]

# 从tar文件创建镜像
$ docker import container-filesystem.tar my-image:v1.0

# 使用标准输入
$ docker import - my-image:v1.0 < container-filesystem.tar

# 从URL导入
$ docker import http://example.com/rootfs.tar x.tar.gz rootfs:v1.0

# 参数说明：
# -c, --change    应用Dockerfile指令到创建的镜像
# -m, --message   设置导入的说明信息

# 示例：从导出文件创建镜像
$ docker import my-container-export.tar my-custom-image:v1

# 预期输出：
# sha256:def456... Importing from tar
sha256:def456... 72.8MB
docker export/import - 实战场景 Bash 
# 场景1：从容器创建新镜像（包含修改）
# 先在容器中做修改
$ docker run -it --name temp-container ubuntu:22.04 bash
root@xxx:/# apt update && apt install -y vim curl git
root@xxx:/# exit

# 导出修改后的容器
$ docker export temp-container -o my-modified.tar

# 从导出文件创建新镜像
$ docker import my-modified.tar my-dev-env:v1.0

# 验证新镜像
$ docker run -it my-dev-env:v1.0 bash

# 清理临时容器
$ docker rm temp-container

# 场景2：最小化镜像创建
$ docker export base-container -o base.tar
$ docker import base.tar minimal-image:latest

# 场景3：离线环境部署
# 在有网络环境导出
$ docker export app-container -o app-for-offline.tar
# 传输到离线服务器
$ scp app-for-offline.tar offline-server:/tmp/
# 在离线服务器导入
$ docker load -i /tmp/app-for-offline.tar
重要区别：export/import vs save/load

- docker export：导出容器的当前状态（不包括镜像历史和元数据）

- docker import：创建一个新的空白镜像（无CMD/ENTRYPOINT等配置）

- docker save：保存完整镜像（包括历史和所有元数据）

- docker load：恢复完整镜像

- export导出的文件通常更小，适合基础系统镜像

Part 2.5.2

docker commit - 从容器创建镜像

docker commit命令用于从运行中的容器创建一个新的镜像。这个功能适合快速保存容器当前状态的快照，但不推荐用于生产环境的镜像构建（生产环境应使用Dockerfile）。

docker commit - 从容器创建镜像 Bash 
# 语法：docker commit [OPTIONS] CONTAINER [REPOSITORY[:TAG]]

# 基本用法 - 创建镜像
$ docker commit my-container my-image:v1.0

# 预期输出：
sha256:abc123...  0B

# 参数详解：
# -a, --author "作者名"    添加作者信息
# -m, --message "说明"     添加提交说明
# -c, --change CMD [...]   应用Dockerfile指令到镜像
# -p, --pause             提交时暂停容器（默认true）

# 示例：添加作者和说明
$ docker commit -a "Developer Name" -m "Added development tools" \
    my-container my-image:v1.0

# 示例：提交时暂停容器（默认行为）
$ docker commit --pause=true my-container my-image:v1.0

# 示例：不暂停容器提交
$ docker commit -p=false my-container my-image:v1.0
docker commit - 实战示例 Bash 
# 场景1：保存容器修改为新镜像
$ docker run -it --name my-dev-container ubuntu:22.04 bash
root@xxx:/# apt update
root@xxx:/# apt install -y vim curl git
root@xxx:/# exit

$ docker commit my-dev-container my-dev-env:v1.0
sha256:abc123...

$ docker run -it my-dev-env:v1.0 bash
# 已包含vim、curl、git

# 场景2：保存配置好的应用容器
$ docker run -d --name configured-app ubuntu:22.04 sleep infinity
$ docker exec configured-app bash -c "echo 'configured' > /app/config.txt"
$ docker commit configured-app my-configured-app:v1

# 场景3：使用--change应用Dockerfile指令
$ docker commit --change='CMD ["sleep", "infinity"]' \
    configured-app my-ubuntu-configured:v1

# 场景4：查看提交后的镜像历史
$ docker history my-image:v1.0

IMAGE          CREATED        CREATED BY                                      SIZE
abc123...      1 minute ago   sleep infinity                                 0B
08d22c0f15e2   3 weeks ago    /bin/sh -c #(nop) CMD ["/bin/bash"]           77.8MB
注意事项

- docker commit只保存文件系统的当前状态，不包括镜像历史

- 不推荐用于生产环境，应使用Dockerfile进行镜像构建

- 提交前确保容器内没有敏感信息（密码、密钥等）

- 使用-m参数添加清晰的提交说明

- 查看提交后的镜像：docker history IMAGE

Part 2.5.3

容器打包成镜像的省空间技巧

在使用docker export/import或docker commit将容器打包成镜像时，有一些优化技巧可以帮助减少镜像体积，节省存储空间和加快传输速度。

省空间优化核心原则

1. 清理缓存和临时文件：删除apt/yum缓存、日志文件、临时文件

2. 使用精简基础镜像：alpine比ubuntu更小

3. 压缩传输：使用gzip等压缩工具

4. 多阶段构建：生产环境推荐使用Dockerfile

技巧1：打包前清理容器 Bash 
# 清理包管理器缓存（Ubuntu/Debian）
$ docker exec my-container apt clean
$ docker exec my-container rm -rf /var/lib/apt/lists/*

# 清理临时文件
$ docker exec my-container rm -rf /tmp/*
$ docker exec my-container rm -rf /var/tmp/*

# 清理日志文件
$ docker exec my-container sh -c "find /var/log -type f -exec truncate -s 0 {} \;"

# 清理其他不需要的文件
$ docker exec my-container rm -rf /root/.cache/*
$ docker exec my-container rm -rf /var/cache/*
技巧2：清理用户进程和日志 Bash 
# 停止容器前清空日志
$ docker exec my-container sh -c "echo '' > /var/log/access.log"
$ docker exec my-container sh -c "echo '' > /var/log/error.log"

# 查看容器大小
$ docker exec my-container du -sh /

# 查看占用空间最大的目录
$ docker exec my-container du -sh /var/*

# 常见可以清理的目录
# /var/log/*        - 日志文件
# /var/cache/*      - 缓存文件
# /tmp/*            - 临时文件
# /root/.cache/*    - 用户缓存
技巧3：使用压缩减少传输大小 Bash 
# 未压缩的导出
$ docker export my-container -o my-container.tar
# 大小示例：500MB

# 使用gzip压缩（压缩率约70%）
$ docker export my-container | gzip > my-container.tar.gz
# 大小示例：150MB

# 使用xz压缩（压缩率更高，但速度慢）
$ docker export my-container | xz > my-container.tar.xz
# 大小示例：100MB

# 恢复时解压导入
$ gunzip -c my-container.tar.gz | docker import - my-image:v1
$ unxz -c my-container.tar.xz | docker import - my-image:v1
技巧4：优化commit操作 Bash 
# 在commit前暂停容器确保数据一致
$ docker pause my-container
$ docker commit my-container my-image:v1
$ docker unpause my-container

# 使用--change添加最小化启动命令
$ docker commit \
    --change='CMD ["tail", "-f", "/dev/null"]' \
    my-container my-image:v1

# 查看生成的镜像大小
$ docker images | grep my-image

# 对比优化前后的镜像大小
$ docker history my-image:v1
最佳实践建议

1. 开发测试：可以使用docker commit快速保存状态

2. 生产环境：必须使用Dockerfile构建镜像

3. 离线部署：使用export+压缩方式传输

4. 镜像优化：优先使用多阶段构建

重要提示

- commit和export会丢失镜像的构建历史

- 导出的镜像无法回滚到之前的状态

- 对于重要应用，强烈建议使用Dockerfile进行镜像构建

- Dockerfile可以版本控制，方便团队协作
