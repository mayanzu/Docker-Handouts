Part 1

Dockerfile基础

理解Dockerfile的作用与基本语法

Part 1.1

什么是Dockerfile？

镜像定义
Docker镜像是一个只读模板，包含了运行应用程序所需的所有内容：代码、运行时环境、系统工具、系统库和配置。镜像是容器的基础，容器是镜像的运行实例。

可以把镜像理解为类（Class），容器理解为对象（Object）。一个镜像可以创建多个容器实例，每个容器都是独立的运行环境。

镜像的组成

1. 基础镜像层：如ubuntu、alpine等操作系统基础

2. 依赖层：应用程序运行所需的库和工具

3. 应用层：应用程序代码和配置文件

4. 元数据：镜像的配置信息（环境变量、端口、启动命令等）

Part 1.2

镜像分层机制

Docker镜像采用联合文件系统（UnionFS）技术，由多个只读层（Layer）叠加而成。每一层代表Dockerfile中的一条指令。

镜像分层示例 概念说明 
# 以Ubuntu镜像为例，查看其分层结构
$ docker history ubuntu:latest

IMAGE          CREATED        CREATED BY                                      SIZE
605c77e624dd   2 weeks ago    /bin/sh -c #(nop)  CMD ["bash"]                0B
      2 weeks ago    /bin/sh -c #(nop)  ADD file:... in /            80.4MB
分层优势

- 共享层：多个镜像可以共享相同的基础层，节省存储空间

- 快速构建：只重建变化的层，未变化层直接使用缓存

- 版本控制：每层有唯一ID，便于追踪和管理

Part 1.3

写时复制（CoW）

写时复制机制
写时复制（Copy-on-Write）是Docker镜像的核心技术。当容器启动时，Docker会在镜像的最上层添加一个可写层（Container Layer）。所有对文件的修改都发生在这个可写层，而不会改变底层的只读镜像层。

1读取文件

容器首次读取文件时，如果文件在只读层，则从只读层读取

2写入文件

容器修改文件时，先将文件从只读层复制到可写层，然后修改

3删除文件

容器删除文件时，在可写层创建删除标记，不影响只读层

优势

多个容器可以共享同一个镜像的可写层，实现资源高效利用

Part 2

Dockerfile指令详解

掌握每个指令的用途和用法

Part 2.1

FROM指令 - 指定基础镜像

FROM指令
FROM指令用于指定基础镜像，是Dockerfile的第一条指令（除ARG外）。基础镜像是构建的起点，所有后续操作都在此基础上进行。
FROM指令语法 Dockerfile 
# 语法：FROM [--platform=<platform>] <image>[:<tag>] [AS <name>]

# 基础用法
FROM ubuntu:22.04

# 使用特定平台
FROM --platform=linux/amd64 ubuntu:22.04

# 多阶段构建命名
FROM golang:1.19 AS builder
FROM alpine:latest AS production
基础镜像选择建议

1. alpine：最小化，约5MB，适合简单应用

2. slim：语言运行时精简版，平衡大小和功能

3. 完整版：如ubuntu:22.04，功能最全但体积大

Part 2.2

RUN指令 - 执行命令

RUN指令
RUN指令用于在镜像构建过程中执行命令，并将结果提交到新的镜像层。每个RUN指令都会创建一个新的镜像层。
简单的Dockerfile示例 Dockerfile 
# 基础镜像
FROM ubuntu:22.04

# 维护者信息
LABEL maintainer="your-email@example.com"

# 设置环境变量
ENV DEBIAN_FRONTEND=noninteractive

# 更新包列表并安装nginx
RUN apt-get update && \
    apt-get install -y nginx && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*

# 复制本地文件到镜像
COPY index.html /var/www/html/

# 暴露端口
EXPOSE 80

# 启动命令
CMD ["nginx", "-g", "daemon off;"]
Dockerfile特点

1. 自动化：一次编写，多次构建，确保环境一致性

2. 可维护：文本文件，版本控制友好

3. 可复用：基于基础镜像，层层叠加

4. 透明：构建过程清晰可见

Part 3.2

docker build命令

docker build - 构建镜像 Bash 
# 语法：docker build [OPTIONS] PATH | URL | -

# 基础用法 - 在当前目录构建（Dockerfile需在目录中）
$ docker build -t myapp:1.0 .

# 预期输出：
[+] Building 15.3s (8/8) FINISHED
 => [internal] load build definition from Dockerfile
 => [internal] load .dockerignore
 => [1/4] FROM ubuntu:22.04@sha256:度
 => [2/4] RUN apt-get update && apt-get install -y nginx
 => [3/4] COPY index.html /var/www/html/
 => [4/4] EXPOSE 80
 => naming to docker.io/library/myapp:1.0
常用参数说明 Bash 
# -t, --tag              指定镜像名称和标签（name:tag）
# --file, -f            指定Dockerfile路径（默认是当前目录的Dockerfile）
# --no-cache            构建时不使用缓存
# --build-arg           设置构建参数（ARG指令的值）

# 示例：指定Dockerfile路径
$ docker build -t myapp:1.0 -f /path/to/Dockerfile /path/to/context

# 示例：不使用缓存构建
$ docker build --no-cache -t myapp:1.0 .

# 示例：传递构建参数
$ docker build --build-arg VERSION=1.0 -t myapp:1.0 .

镜像构建实践

实际构建应用镜像

Part 4.1

FROM - 指定基础镜像

FROM指令 Dockerfile 
# 语法：FROM [--platform <platform>] <image>[:<tag>] [AS <name>]

# 作用：指定构建镜像的基础镜像，必须是第一条指令（除ARG外）

# 示例：使用官方Ubuntu
FROM ubuntu:22.04

# 示例：使用官方Python（带有平台指定）
FROM --platform=linux/amd64 python:3.11

# 示例：多阶段构建中为阶段命名
FROM golang:1.20 AS builder
# ... 构建过程 ...
FROM alpine:latest
COPY --from=builder /bin/myapp /bin/myapp
基础镜像选择建议

- 优先使用官方镜像，如ubuntu、alpine、python等

- 生产环境建议使用固定版本标签，避免latest导致的不确定性

- 考虑使用Alpine等轻量镜像减小镜像体积

Part 4.2

RUN - 执行命令

RUN指令 Dockerfile 
# 语法1：RUN <command>  （shell形式，默认使用 /bin/sh -c）
# 语法2：RUN ["executable", "param1", "param2"]  （exec形式，推荐）

# shell形式
RUN echo "Hello Docker" > /tmp/test.txt
RUN apt-get update && apt-get install -y nginx

# exec形式（不会调用shell，避免shell解析问题）
RUN ["apt-get", "update"]
RUN ["apt-get", "install", "-y", "nginx"]

# 优化RUN指令：合并多个命令减少层数
RUN apt-get update && \
    apt-get install -y nginx curl && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*
RUN指令最佳实践 Dockerfile 
# 技巧1：使用换行符提高可读性
RUN apt-get update && \
    apt-get install -y \
        nginx \
        curl \
        vim

# 技巧2：清理缓存减少镜像体积
RUN apt-get update && \
    apt-get install -y nginx && \
    rm -rf /var/lib/apt/lists/*

# 技巧3：按正确顺序放置指令，充分利用缓存
# 先放置不常变化的指令
FROM ubuntu:22.04
RUN apt-get update
COPY requirements.txt /tmp/
RUN pip install -r /tmp/requirements.txt
COPY . /app
# 修改代码时，前面的apt-get update会被缓存
Part 4.3

COPY与ADD - 复制文件

COPY指令 Dockerfile 
# 语法：COPY [--chown=<user>:<group>] <src>... <dest>

# 复制本地文件到镜像
COPY index.html /var/www/html/
COPY . /app  # 复制整个构建上下文到镜像

# 带权限复制
COPY --chown=www-data:www-data index.html /var/www/html/

# 示例：复制应用代码
COPY package*.json /app/
COPY src /app/src
ADD指令 Dockerfile 
# 语法：ADD [--chown=<user>:<group>] <src>... <dest>

# ADD功能类似COPY，但多了以下特性：
# 1. 可以从URL复制文件
# 2. 可以解压tar文件到镜像

# 从URL复制（不推荐，建议使用RUN wget或curl）
ADD http://example.com/file.tar.gz /tmp/

# 解压tar文件（自动识别压缩格式）
ADD app.tar.gz /app/

# 从另一个构建阶段复制（多阶段构建）
FROM golang:1.20 AS builder
ADD . /src
RUN go build -o app
FROM alpine:latest
COPY --from=builder /src/app /usr/local/bin/
选择建议

- 基本文件复制使用COPY（更透明、可预测）

- 只有需要URL复制或自动解压时才使用ADD

Part 4.4

WORKDIR与ENV - 工作目录与环境变量

WORKDIR指令 Dockerfile 
# 语法：WORKDIR /path/to/workdir

# 设置工作目录（类似cd）
WORKDIR /app
COPY . /app
RUN npm install

# 可以切换已存在的工作目录
WORKDIR /app
WORKDIR /app/src
RUN make build

# 如果目录不存在会自动创建
WORKDIR /app/subdir

# 注意：WORKDIR会创建目录，如果创建失败会报错
ENV指令 Dockerfile 
# 语法：ENV <key>=<value> ...

# 设置环境变量
ENV APP_HOME=/app
ENV NODE_ENV=production

# 设置多个环境变量
ENV DEBIAN_FRONTEND=noninteractive \
    PYTHONUNBUFFERED=1 \
    NGINX_PORT=80

# 使用环境变量
RUN echo $APP_HOME
COPY . $APP_HOME/

# docker run时覆盖环境变量
# docker run -e NODE_ENV=development myapp
Part 4.5

EXPOSE与CMD - 端口与启动命令

EXPOSE指令 Dockerfile 
# 语法：EXPOSE <port> [<port>/<protocol>...]

# 声明容器监听的端口（仅文档作用，不实际绑定）
EXPOSE 80
EXPOSE 443
EXPOSE 8080

# 指定协议
EXPOSE 80/tcp
EXPOSE 53/udp

# 注意：EXPOSE只是文档，运行时需要用-p或-P绑定端口
CMD指令 Dockerfile 
# 语法：CMD ["executable", "param1", "param2"]  （exec形式，推荐）
# 语法2：CMD command param1 param2  （shell形式）
# 语法3：CMD ["param1", "param2"]  （作为ENTRYPOINT的参数）

# exec形式（直接执行，可被docker run参数覆盖）
CMD ["nginx", "-g", "daemon off;"]

# shell形式（会被包装为 /bin/sh -c "command"）
CMD echo "Hello"

# 作为ENTRYPOINT的参数（常用于设置默认参数）
CMD ["--version"]
ENTRYPOINT ["myapp"]

# Dockerfile中只能有一个CMD，多个时只有最后一个生效
ENTRYPOINT指令 Dockerfile 
# 语法：ENTRYPOINT ["executable", "param1", "param2"]  （exec形式）
# 语法2：ENTRYPOINT command param1 param2  （shell形式）

# 与CMD配合：CMD提供默认参数，ENTRYPOINT提供可执行文件
ENTRYPOINT ["python", "app.py"]
CMD ["--help"]

# docker run效果：
# python app.py --help

# 常见应用场景：配置启动命令
ENTRYPOINT ["docker-entrypoint.sh"]
CMD ["postgres"]
Part 4.6

其他重要指令

ARG指令 - 构建参数 Dockerfile 
# 语法：ARG <name>[=<default value>]

# 定义构建参数（类似变量）
ARG VERSION=latest
ARG APP_ENV=production

# 使用参数
RUN echo $VERSION
RUN pip install myapp==$VERSION

# docker build时传递参数
# $ docker build --build-arg VERSION=1.0 -t myapp:1.0 .
LABEL指令 - 元数据 Dockerfile 
# 语法：LABEL <key>=<value> ...

# 添加镜像元数据
LABEL maintainer="email@example.com"
LABEL version="1.0"
LABEL description="My application"

# 建议使用label组织元数据
LABEL org.opencontainers.image.title="My App"
LABEL org.opencontainers.image.version="1.0"
LABEL org.opencontainers.image.authors="author"
VOLUME指令 - 数据卷 Dockerfile 
# 语法：VOLUME ["path1", "path2", ...]

# 定义挂载点（容器运行时可覆盖）
VOLUME ["/data", "/logs"]
VOLUME /var/www/html

# 注意：
# - 无法在Dockerfile中指定主机挂载点（运行时指定）
# - 定义的卷会创建在宿主机 /var/lib/docker/volumes/ 下

Part 5.3

多阶段构建

多阶段构建允许在单个Dockerfile中使用多个FROM指令，每个FROM开始一个新的构建阶段。这可以大大减小最终镜像的体积。

多阶段构建示例 Dockerfile 
# 阶段1：构建应用
FROM golang:1.20 AS builder
WORKDIR /src
COPY . .
RUN go build -o myapp

# 阶段2：运行应用（使用轻量基础镜像）
FROM alpine:latest
WORKDIR /app
COPY --from=builder /src/myapp /app/
CMD ["/app/myapp"]

# 最终镜像只包含运行所需的最小文件，不包含构建工具
构建并运行 Bash 
# 构建多阶段镜像
$ docker build -t my-python-app:multi -f Dockerfile.multi .

# 对比镜像大小
$ docker build -t my-python-app:single -f Dockerfile.single .
多阶段构建优势

- 减小镜像体积：只复制运行所需文件到最终镜像

- 提高安全性：最终镜像不包含编译工具和源码

- 分离构建和运行环境：构建阶段可以使用完整的开发工具

Part 6.1.1

docker login - 登录Docker Hub

docker login命令用于登录Docker Hub。登录后才能推送镜像到Docker Hub或从私有仓库拉取镜像。

docker login - 登录仓库 Bash 
# 登录Docker Hub
$ docker login

# 预期输出：
Username: your-username
Password:
Login Succeeded

# 登录信息会保存在 ~/.docker/config.json
参数说明

- 无参数：登录Docker Hub（默认）

- -u, --username：指定用户名

- -p, --password：指定密码（不建议在命令行使用）

Part 6.1.2

docker tag 与 docker push - 推送镜像

推送镜像到Docker Hub需要先打标签，然后推送。

docker tag - 给镜像打标签 Bash 
# 标记本地镜像
# 格式：docker tag 源镜像 目标仓库/用户名/镜像名:标签
$ docker tag myapp:latest your-username/myapp:latest

# 如果镜像没有tag，默认使用latest
docker push - 推送镜像到仓库 Bash 
# 推送镜像到Docker Hub
$ docker push your-username/myapp:latest

# 预期输出：
# The push refers to repository [docker.io/your-username/myapp]
# latest: digest: sha256:abc123... size: 2048
推送流程

1. docker login - 登录Docker Hub

2. docker tag - 给镜像打上仓库标签

3. docker push - 推送到仓库

Part 6.1.3

docker pull 与 docker logout

docker pull用于从仓库拉取镜像。docker logout用于登出仓库。

docker pull - 拉取镜像 Bash 
# 拉取公开镜像（无需登录）
$ docker pull your-username/myapp:latest

# 拉取官方镜像
$ docker pull ubuntu:22.04

# 公开镜像无需登录即可拉取
docker logout - 登出仓库 Bash 
# 登出Docker Hub
$ docker logout

# 登出后，凭证会从 ~/.docker/config.json 中移除
Docker Hub操作完整流程

1. docker login - 登录仓库

2. docker tag - 打标签

3. docker push - 推送镜像

4. docker pull - 拉取镜像

5. docker logout - 登出仓库

Part 6.4

私有镜像仓库

除了公共仓库，企业通常使用私有镜像仓库存储内部镜像。常见选择包括Docker Registry、Harbor、阿里云镜像仓库等。

使用私有仓库 Bash 
# 标记镜像为私有仓库地址
$ docker tag myapp:latest registry.example.com/myapp:latest

# 推送到私有仓库
$ docker push registry.example.com/myapp:latest

# 从私有仓库拉取
$ docker pull registry.example.com/myapp:latest
总结

Docker镜像管理小结

* 镜像概念：只读模板，包含运行应用所需的所有内容
* 分层存储：UnionFS技术，每层只读，节省空间
* Dockerfile：构建镜像的脚本，指令顺序影响构建效率和缓存利用
* 核心指令：FROM、RUN、COPY、CMD、EXPOSE、ENV、WORKDIR
* 多阶段构建：减小镜像体积，提高安全性
* 镜像仓库：分享和分发镜像的标准方式
核心要点回顾

1. 镜像分层是Docker的核心机制，每层对应Dockerfile的一条指令

2. Dockerfile是构建镜像的脚本，指令顺序影响构建效率和缓存利用

3. 掌握多阶段构建可以显著减小镜像体积

4. 镜像仓库是分享和分发镜像的标准方式
