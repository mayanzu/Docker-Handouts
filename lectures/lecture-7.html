Part 1

Dockerfile基础

理解Dockerfile的作用与基本概念

1.1

什么是Dockerfile？

Dockerfile定义
Dockerfile是一个文本文件，包含了构建Docker镜像的所有指令。通过读取Dockerfile中的指令，docker build命令可以自动构建出所需的镜像。

可以把Dockerfile理解为镜像的构建脚本。Dockerfile定义了如何创建镜像，而镜像则是容器的模板。

打个比方：如果把制作蛋糕比作Docker镜像构建，那么基础镜像就是面粉，Dockerfile就是配方食谱，而docker build就是按照食谱制作蛋糕的过程。

Dockerfile示例 Dockerfile 
# 基础镜像
FROM ubuntu:22.04

# 维护者信息
LABEL maintainer="your-email@example.com"

# 设置环境变量
ENV DEBIAN_FRONTEND=noninteractive

# 安装nginx
RUN apt-get update && \
    apt-get install -y nginx && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*

# 复制网站文件
COPY index.html /var/www/html/

# 暴露端口
EXPOSE 80

# 启动命令
CMD ["nginx", "-g", "daemon off;"]
Dockerfile工作原理

1. Docker读取Dockerfile中的指令顺序执行

2. 每条指令执行后生成一个新的镜像层

3. 后续指令基于前面指令的结果进行操作

4. 任何步骤失败都可以从缓存点恢复构建

1.2

镜像分层机制

联合文件系统（UnionFS）
Docker镜像采用联合文件系统技术，由多个只读层（Layer）叠加而成。每一层代表Dockerfile中的一条指令。这种分层结构是Docker高效存储和快速构建的基础。

想象一下，镜像就像是一叠透明的塑料片，每一片上都记录着不同的修改。最底层的塑料片是基础镜像，往上每一片都是对该镜像的一次修改。最终我们看到的是所有塑料片叠加在一起的效果。

镜像分层示例 概念说明 
# 以Ubuntu镜像为例，查看其分层结构
$ docker history ubuntu:latest

IMAGE          CREATED        CREATED BY                                      SIZE
605c77e624dd   2 weeks ago    /bin/sh -c #(nop)  CMD ["bash"]                0B
<missing>      2 weeks ago    /bin/sh -c #(nop)  ADD file:... in /            80.4MB
分层优势

- 共享层：多个镜像可以共享相同的基础层，节省存储空间

- 快速构建：只重建变化的层，未变化层直接使用缓存

- 版本控制：每层有唯一ID，便于追踪和管理

- 增量更新：只需传输变化的层，而非整个镜像

1.3

写时复制（CoW）

写时复制机制
写时复制（Copy-on-Write，简称CoW）是一种高效的共享和复制技术。当容器需要修改某个文件时，Docker不会直接修改镜像层的文件，而是将文件复制到容器的可写层，然后在可写层进行修改。

这个机制就像图书馆的图书借阅：你可以先阅读公共区域的图书（只读层），但如果你想要在书上做笔记（写入），图书馆会先帮你复制一本（复制到可写层），你在这本复制本上做笔记，不会影响公共区域的原书。

1读取文件

容器首次读取文件时，如果文件在只读层，则从只读层直接读取，无需复制

2写入文件

容器修改文件时，先将文件从只读层复制到可写层，然后在可写层修改原文件

3删除文件

容器删除文件时，在可写层创建删除标记（whiteout文件），不影响只读层

CoW机制的优势

多个容器可以共享同一个镜像的可写层，实现资源高效利用。这意味着即使有100个容器运行同一个镜像，实际上内存和存储中只保留了一份基础数据。

Part 2

Dockerfile指令详解

掌握每个指令的用途和用法

2.1

FROM - 指定基础镜像

FROM指令原理
FROM指令是Dockerfile的起点，用于指定构建镜像所依赖的基础镜像。所有后续指令都在这个基础镜像之上执行。FROM是唯一必须放在最前面的指令（ARG指令可以在FROM之前）。

基础镜像的选择对最终镜像的大小、功能和安全性都有重要影响。就像建筑房屋的地基，地基选择不当，后面的工作都会受影响。

基础镜像类型

- 操作系统镜像：如ubuntu、debian、alpine，只包含操作系统

- 语言运行时镜像：如python、node、golang，已包含语言环境

- 应用镜像：如nginx、redis、mysql，已配置好应用

FROM指令语法 Dockerfile 
# 语法：FROM [--platform=<platform>] <image>[:<tag>] [AS <name>]

# 基础用法
FROM ubuntu:22.04

# 使用特定平台（跨平台构建时）
FROM --platform=linux/amd64 ubuntu:22.04

# 多阶段构建命名（后续会详细介绍）
FROM golang:1.19 AS builder
最佳实践

- 优先使用官方镜像，如ubuntu、alpine、python等

- 生产环境建议使用固定版本标签，避免latest导致的不确定性

- 考虑使用Alpine等轻量镜像减小镜像体积

2.2

RUN - 执行命令

RUN指令原理
RUN指令用于在镜像构建过程中执行命令，并将执行结果提交为新的镜像层。这是Dockerfile中最强大的指令，可以安装软件、配置环境、创建文件等。

RUN指令执行时会启动一个临时容器，执行命令后，将结果提交为一个新的只读层。这就是为什么每次RUN都会增加镜像体积。

两种执行形式

- Shell形式：RUN command，使用/bin/sh -c执行，优点是方便，缺点是有shell解析开销

- Exec形式：RUN ["executable", "param1", "param2"]，直接调用程序，避免shell解析问题

RUN指令语法 Dockerfile 
# Shell形式（默认使用 /bin/sh -c 执行）
RUN echo "Hello Docker" > /tmp/test.txt
RUN apt-get update && apt-get install -y nginx

# Exec形式（直接执行，不会调用shell）
RUN ["apt-get", "update"]
RUN ["apt-get", "install", "-y", "nginx"]

# 优化RUN指令：合并多个命令减少层数
RUN apt-get update && \
    apt-get install -y nginx curl && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*
RUN指令最佳实践

- 使用换行符提高可读性

- 合并RUN指令减少镜像层数，每层都占用空间

- 清理缓存：apt-get clean 和 rm -rf /var/lib/apt/lists/*

- 按正确顺序放置指令，充分利用缓存

2.3

COPY与ADD - 复制文件

COPY指令原理
COPY指令将文件或目录从构建上下文复制到镜像中。每次执行COPY都会创建一个新的镜像层。如果目的路径不存在，Docker会自动创建。

COPY的工作方式类似于cp命令，但复制的源路径是相对于构建上下文的（就是docker build命令指定的PATH）。这意味着COPY只能复制构建上下文内的文件。

COPY指令 Dockerfile 
# 语法：COPY [--chown=<user>:<group>] <src>... <dest>

# 复制单个文件到镜像
COPY index.html /var/www/html/

# 复制整个构建上下文到镜像
COPY . /app

# 带权限复制（指定文件所有者）
COPY --chown=www-data:www-data index.html /var/www/html/

# 复制多个文件到目录
COPY package*.json /app/
ADD指令原理
ADD是COPY的增强版，除了基本的复制功能外，还支持从URL复制和自动解压tar文件。但正因为这些特性，ADD的行为不如COPY直观可预测。
ADD指令 Dockerfile 
# ADD功能类似COPY，但多了以下特性：

# 从URL复制文件（不推荐，建议使用RUN wget或curl）
ADD http://example.com/file.tar.gz /tmp/

# 自动解压tar文件（但不会自动删除源文件）
ADD app.tar.gz /app/

# 从另一个构建阶段复制（多阶段构建特有）
FROM golang:1.20 AS builder
ADD . /src
RUN go build -o app
FROM alpine:latest
COPY --from=builder /src/app /usr/local/bin/
选择建议

- 基本文件复制使用COPY（更透明、可预测）

- 只有需要URL复制或自动解压时才使用ADD

- 从其他构建阶段复制使用COPY --from

2.4

WORKDIR与ENV - 工作目录与环境变量

WORKDIR指令原理
WORKDIR设置工作目录，类似于Linux的cd命令。它会影响后续所有指令的工作路径。如果指定的目录不存在，WORKDIR会自动创建。

WORKDIR不像RUN那样创建新的镜像层，它只是一个路径设置。可以多次使用WORKDIR切换目录，也可以在一条指令中使用绝对路径。

WORKDIR指令 Dockerfile 
# 语法：WORKDIR /path/to/workdir

# 设置工作目录
WORKDIR /app
COPY . /app
RUN npm install

# 切换到子目录
WORKDIR /app
WORKDIR /app/src
RUN make build

# 如果目录不存在会自动创建
WORKDIR /app/subdir
ENV指令原理
ENV用于设置环境变量。这些变量会在镜像构建过程中生效，也会在容器运行时存在。环境变量是容器化应用配置的重要方式。

环境变量的作用范围很广：可以在RUN指令中使用，可以在后续的ENV指令中引用，可以在容器运行时被覆盖，也会被应用程序读取。

ENV指令 Dockerfile 
# 语法：ENV <key>=<value> ...

# 设置单个环境变量
ENV APP_HOME=/app
ENV NODE_ENV=production

# 设置多个环境变量（推荐写法）
ENV DEBIAN_FRONTEND=noninteractive \
    PYTHONUNBUFFERED=1 \
    NGINX_PORT=80

# 使用环境变量
RUN echo $APP_HOME
COPY . $APP_HOME/

# 运行时覆盖环境变量
# docker run -e NODE_ENV=development myapp
2.5

EXPOSE与CMD - 端口与启动命令

EXPOSE指令原理
EXPOSE是文档性指令，用于声明容器将监听的网络端口。这不会实际在容器启动时打开端口，而是告诉用户该镜像期望暴露的端口。

EXPOSE就像商品说明书上的规格信息，告诉使用者"这个容器会监听80端口"。但真正打开端口的是docker run的-p参数。

EXPOSE指令 Dockerfile 
# 语法：EXPOSE <port> [<port>/<protocol>...]

# 声明容器监听端口（仅文档作用）
EXPOSE 80
EXPOSE 443
EXPOSE 8080

# 指定协议
EXPOSE 53/udp

# 注意：EXPOSE不会实际绑定端口，运行时需要用-p或-P
CMD指令原理
CMD指定容器启动时执行的命令。如果docker run指定了命令，CMD的内容会被覆盖。Dockerfile中只能有一个CMD，多个CMD时只有最后一个生效。

CMD就像是容器的"默认行为"。当你运行一个nginx镜像而没有指定任何命令时，它会执行CMD中定义的启动命令。

CMD与ENTRYPOINT指令 Dockerfile 
# CMD的三种形式：

# 1. Exec形式（推荐）：直接执行命令
CMD ["nginx", "-g", "daemon off;"]

# 2. Shell形式：会被包装为 /bin/sh -c "command"
CMD echo "Hello"

# 3. 作为ENTRYPOINT的参数
CMD ["--help"]
ENTRYPOINT ["myapp"]

# ENTRYPOINT：定义容器的主命令
ENTRYPOINT ["python", "app.py"]
CMD ["--help"]
# 运行效果：python app.py --help
CMD与ENTRYPOINT区别

CMD：可以被docker run参数覆盖，常用于设置默认值

ENTRYPOINT：不可被覆盖，定义容器的固定行为

两者配合：ENTRYPOINT定义主命令，CMD提供默认参数

2.6

ARG、LABEL与VOLUME - 其他重要指令

ARG指令原理
ARG用于定义构建参数，它只在镜像构建过程中生效，不会传递到容器运行时。与ENV不同，ARG的值在镜像构建完成后就不存在了。

ARG就像构建时的"变量"，可以在docker build命令中传入不同的值，实现镜像的差异化构建。例如用同一个Dockerfile构建不同版本的镜像。

ARG指令 Dockerfile 
# 语法：ARG <name>[=<default value>]

# 定义构建参数（可设置默认值）
ARG VERSION=latest
ARG APP_ENV=production

# 使用参数（在RUN指令中引用）
RUN echo $VERSION
RUN pip install myapp==$VERSION

# docker build时传递参数
# $ docker build --build-arg VERSION=1.0 -t myapp:1.0 .
LABEL指令原理
LABEL用于添加元数据到镜像中，不会影响镜像的功能。元数据以键值对的形式存储，可以包含版本、作者、描述等信息。
LABEL与VOLUME指令 Dockerfile 
# LABEL：添加镜像元数据
LABEL maintainer="email@example.com"
LABEL version="1.0"
LABEL description="My application"

# 建议使用label组织元数据（符合OCI标准）
LABEL org.opencontainers.image.title="My App"
LABEL org.opencontainers.image.version="1.0"

# VOLUME：定义数据卷挂载点
# 创建容器时，Docker会在宿主机创建匿名卷
# 容器运行时可以用 -v 覆盖
VOLUME ["/data", "/logs"]
VOLUME /var/www/html
VOLUME的重要性

VOLUME声明的目录会被持久化存储，即使容器被删除，数据也不会丢失。这是Docker数据管理的核心机制。

Part 3

Dockerfile实践与优化

从问题Dockerfile到优化实践

3.1

未优化的Dockerfile示例

下面是一个常见的未优化Dockerfile，存在多个导致镜像体积过大的问题。

未优化的Dockerfile Dockerfile 
# 基础镜像（完整版Ubuntu，体积较大）
FROM ubuntu:22.04

# 安装vim，每个RUN一层
RUN apt-get update
RUN apt-get install -y vim

WORKDIR /app

CMD ["vim", "--version"]
存在的问题

1. 每个RUN一层：产生多层镜像，体积大

2. 未清理缓存：apt-get缓存留在镜像中

3. 未使用--no-install-recommends：安装了不必要的依赖

构建并查看体积 Bash 
# 构建未优化镜像
$ docker build -t myapp:unoptimized .

# 查看镜像体积（会比较大）
$ docker images | grep myapp
myapp    unoptimized    450MB
3.2

docker build - 构建镜像

docker build原理
docker build命令读取Dockerfile，按照指令顺序执行每条命令，最终生成一个新的镜像。构建过程中，Docker会为每条指令创建容器、执行命令、提交更改。

docker build的工作流程：读取Dockerfile → 创建临时容器 → 执行指令 → 提交新镜像层 → 删除临时容器 → 继续下一条指令。

docker build命令 Bash 
# 语法：docker build [OPTIONS] PATH | URL | -

# 基础用法 - 在当前目录构建
$ docker build -t myapp:1.0 .

# 预期输出：
[+] Building 15.3s (8/8) FINISHED
 => [internal] load build definition from Dockerfile
 => [internal] load .dockerignore
 => [1/4] FROM ubuntu:22.04
 => [2/4] RUN apt-get update && apt-get install -y vim
 => naming to docker.io/library/myapp:1.0
常用参数说明 Bash 
# -t, --tag              指定镜像名称和标签（name:tag）
# --file, -f            指定Dockerfile路径
# --no-cache            构建时不使用缓存
# --build-arg           设置构建参数

# 指定Dockerfile路径和构建上下文
$ docker build -t myapp:1.0 -f /path/to/Dockerfile /path/to/context

# 不使用缓存构建
$ docker build --no-cache -t myapp:1.0 .

# 传递构建参数
$ docker build --build-arg VERSION=1.0 -t myapp:1.0 .
构建上下文说明

- PATH是构建上下文，docker build会在该目录下查找Dockerfile和所需文件

- .表示当前目录作为构建上下文

- COPY指令只能复制构建上下文内的文件，无法访问外部文件

3.3

镜像优化技巧

镜像优化原理
每个RUN指令都会创建一个新的镜像层。通过合并RUN指令、清理缓存等技巧，可以显著减小镜像体积。

apt-get安装软件时，会下载软件包到/var/lib/apt/lists/目录。如果不清理，这些缓存会被保存到镜像层中。

四大优化技巧

1. 合并RUN指令：减少镜像层数，每层都占用空间

2. 清理缓存：apt-get clean 和 rm -rf /var/lib/apt/lists/*

3. 删除临时文件：/tmp/* 和 /var/tmp/*

4. 使用--no-install-recommends：只安装必需依赖

优化技巧详解 Dockerfile 
# 技巧1：合并RUN指令
# 合并前（多Layer）
RUN apt-get update
RUN apt-get install -y vim

# 合并后（单Layer）
RUN apt-get update && \
    apt-get install -y vim

# 技巧2：清理缓存
RUN apt-get update && \
    apt-get install -y vim && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*

# 技巧3：使用 --no-install-recommends
RUN apt-get update && \
    apt-get install -y --no-install-recommends vim && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*

# 技巧4：使用多行符提高可读性
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
        vim && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/* /tmp/* /var/tmp/*
3.4

优化后的Dockerfile

下面是针对之前未优化Dockerfile的改进版本，应用了所有优化技巧。

优化后的Dockerfile Dockerfile 
# 使用轻量基础镜像
FROM ubuntu:22.04

# 设置环境变量（避免交互式提示）
ENV DEBIAN_FRONTEND=noninteractive

# 合并RUN指令 + 清理缓存 + 使用--no-install-recommends
RUN apt-get update && \
    apt-get install -y --no-install-recommends vim && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/* /tmp/* /var/tmp/*

WORKDIR /app

CMD ["vim", "--version"]

多阶段构建

多阶段构建原理
多阶段构建允许在单个Dockerfile中使用多个FROM指令，每个FROM开始一个新的构建阶段。最终镜像只包含最后一个FROM阶段的内容。

多阶段构建的核心思想是"分离构建环境和运行环境"。第一阶段安装所有构建工具、编译代码，第二阶段只复制运行所需的最小文件。

多阶段构建示例 Dockerfile 
# 阶段1：构建应用（使用完整工具链）
FROM golang:1.20 AS builder
WORKDIR /src
COPY . .
RUN go build -o myapp

# 阶段2：运行应用（使用轻量基础镜像）
FROM alpine:latest
WORKDIR /app
COPY --from=builder /src/myapp /app/
CMD ["/app/myapp"]

# 最终镜像只包含运行所需的最小文件
多阶段构建优势

- 减小镜像体积：只复制运行所需文件到最终镜像

- 提高安全性：最终镜像不包含编译工具和源码

- 分离构建和运行环境：构建阶段可以使用完整的开发工具
